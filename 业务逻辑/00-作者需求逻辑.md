> 本文从**长篇连载作者**的真实需求出发，定义 InkFlow 要解决什么问题、产品应如何表现。

# 上篇 · 作者要什么、我们解决什么

## 1. 我们服务谁

InkFlow 面向**长篇连载作者**——同一本书写几十到数百章，跨周、跨月持续更新。

不是偶尔用 AI 写一段的人，而是：

- 隔几天再打开，需要快速找回「写到哪、前面发生了什么」
- 世界观和人物关系会不断生长，越写越难靠脑子记
- 改一个设定，会担心后面几十章对不上

**我们要解决的不是「帮你想剧情」，而是：**

> 在篇幅长到人脑记不住之后，仍能保持前后一致、改得动、写得顺。

---

## 2. 作者面临的四个问题

写长了才会有的痛，可以归纳成四句话：

| 问题 | 作者心里的声音 | 本质 |
|------|----------------|------|
| **记忆过载** | 「我写的东西太多了，自己也记不清了」 | 信息规模超出人脑；续写前必须知道「前面发生了什么」 |
| **一致性焦虑** | 「会不会前后矛盾？伏笔忘了收？」 | 怕读者发现 bug，不是怕写得慢 |
| **改动扩散** | 「改一处，后面很多地方都要跟着变」 | 想局部改，但不知道影响面；更怕系统乱改已写正文 |
| **心流 vs 维护** | 「我想写，不想每章都搞一堆后台任务」 | 维护必要，但不能抢写作状态 |

四个问题背后，作者真正要的是：

> **正文随时能写能改；系统记得住百万字；局部改动可控；未来规划听我的。**

---

## 3. 产品承诺

InkFlow 对作者的三条承诺（与实现无关，是验收的上位标准）：

| 承诺 | 含义 |
|------|------|
| **写得顺** | 续写、改稿时不被维护、规划、同步打断 |
| **记得住** | 系统能基于**已写内容**回答、续写、检查——不靠作者翻几百章回忆 |
| **改得动** | 局部修改可控；大范围影响有提醒；**已写正文不被系统擅自改写** |

### 铁律

**已写章节正文是最高真相。**

章摘要、设定档案、未来细纲都是正文的**索引与备忘录**。备忘录可以晚一点、成批对一次；正文不能丢，也不能被系统悄悄覆盖。

---

## 4. 作者心里的两类东西

Implementation 之前，作者不会想「章摘要」「sync_characters」。心里大致只有两类：

| 类型 | 作者怎么说 | 性质 |
|------|------------|------|
| **已经写进书里的** | 「前面发生了什么」「这章写了啥」 | **事实**——定了，不能悄悄变 |
| **还没写、或只是打算的** | 「后面打算怎么写」 | **意图**——可以改、可以废 |

为了服务 AI 和作者自己，「已经发生的」在系统里再拆成三层——但**作者感知上仍是「书里的事实」**：

```
已经发生的（事实）
├── 正文          … 完整故事，唯一真相
├── 按章压缩      … 快速回忆「每章发生了什么」
└── 设定档案      … 角色、**地点**、关系、世界观、伏笔清单

还没发生的（意图）
└── 未来细纲      … 打算怎么推进，可随时推翻
```

**「已发生」和「打算」必须分开维护。**  
否则写一章就自动改未来规划，作者会失去控制感。

**「按章压缩」和「设定档案」性质相同**——都是把正文收进系统记忆，只是粒度不同。因此产品上**合并为同一个「对一对」动作**，不打断心流。这是心流优先下的设计选择，不是作者天然只关心其中一项。

---

## 5. 我们刻意不做什么

划清边界，避免产品发散：

| 不是核心 | 原因 |
|----------|------|
| 替作者决定剧情 | 作者是创作者，AI 是助手 |
| 每章自动改未来细纲 | 规划属于作者意图 |
| 万能聊天机器人 | 作者要写作环境，不是闲聊 |
| 自动改已写正文 | 违反「正文是真相」 |
| 零维护 | 长篇不可能零维护；**维护时机由作者掌控** |

# 下篇 · InkFlow 如何满足这些需求

## 6. 四种日常意图

作者打开软件，高频动作归纳为四类：

| 意图 | 解决哪个问题 | 作者例子 | 产品入口 |
|------|--------------|----------|----------|
| **写** | 心流 | 「续写」「写下一章」 | Chat · 写 |
| **问** | 记忆过载、一致性 | 「第几章埋的线？」 | Chat · 问 |
| **改** | 改动扩散 | 「把主角改名」 | Chat · 改，或侧边栏直接改 |
| **对** | 记忆过载、一致性 | 「把最近写的收进系统记忆」 | **Editor 顶栏按钮**（主路径） |

前三个是**创作行为**，作者愿意在 Chat 里说。

「对一对」回答的是：**「我这几天写了不少，帮系统记住书里的新事实，别让我下次续写、提问时对不上。」**

它不是第五个聊天助手，而是像**保存 / 导出**一样的按钮级动作——作者知道要做，但不想在写作过程中被打断。

---

## 7. 作者的一天

### 打开软件

- 看侧边栏：写到第几章；**已「对过」的章**有摘要可浏览
- 顶栏 **[同步设定]** 始终可见；有待同步时显示角标
- 可选：当日首次轻提示「有 N 章尚未同步」（与按钮同一动作）
- **不会**自动改未来细纲

角标含义：**有正文尚未沉淀进系统记忆**（缺章摘要，或设定档案落后于最新已写章）。

### 写作（心流优先）

- 续写 / 重写 → **只保存正文**
- 侧边栏改正文 → 保存后立刻更新；正文有改动时**旧摘要作废**，等下次「对一对」再补
- **不会**：每章弹 sync、自动改设定、自动改细纲

连写多章后摘要、档案暂时旧一点——**正常、符合预期**。

### 对一对（作者主动，常见天级）

点 **[同步设定]** → 确认 → 批量把近期正文收进系统记忆：

1. 补全 / 更新**章摘要**
2. 同步**角色、地点、关系、伏笔、写作设定**

不改未来细纲。Chat 里说「同步设定」→ 引导点按钮（同一流程，非主路径）。

> 按钮文案是「同步设定」，作者应理解为：**「把最近写的，收进系统记忆（摘要 + 档案）」**。

### 改主意 / 调规划

| 作者想做什么 | 怎么做 |
|--------------|--------|
| 改名、加规则、调基调 | Chat **改**，或侧边栏直接编辑 |
| 更新「后面怎么写」 | 单独说「同步细纲」——与「同步设定」**分开** |
| 从零重做未来规划 | 「重新生成未来细纲」 |

局部修改后，系统**提醒**可能受影响的范围（如关系、未来细纲），以提醒为主，不强制全自动改。

---

## 8. 两个必须分开的动作

| | **同步设定**（顶栏按钮） | **同步细纲**（单独发起） |
|--|--------------------------|---------------------------|
| **作者问题** | 「最近写的，系统记住了吗？」 | 「后面几章打算怎么写？」 |
| **改什么** | 章摘要 + 角色 + **地点** + 关系 + 伏笔 + 世界观 | 未来细纲 |
| **性质** | 从正文提取**已发生的事实** | **创作规划**（可推翻） |
| **入口** | Editor 顶栏 | Chat：「同步细纲」「更新未来大纲」 |
| **典型时机** | 连写一批后、每天一次、开新卷前对账 | 改走向、卡文规划、开新卷 |

---

## 9. 产品边界（改功能前先对照）

| 发生什么 | 作者期望 | 系统不应 |
|----------|----------|----------|
| 写完 / 保存一章 | 正文立即可见 | 当场生成摘要、弹 sync、改细纲 |
| 手动改了正文 | 旧摘要作废，等新同步 | 保留与正文不符的摘要 |
| 连写多章 | 正文齐；记忆可暂时旧 | 每章改档案或摘要 |
| 删一章 | 该章正文与摘要一并消失 | 留孤儿摘要 |
| 点 **同步设定** | 章摘要 + **地点** + 档案 | 改未来细纲 |
| 说 **同步细纲** | 才动未来规划 | 与设定按钮绑在一起 |

### 三层维护节奏

| 层级 | 作者理解 | 何时发生 | 自动？ |
|------|----------|----------|--------|
| **正文** | 写进书里了 | 写 / 存 | ✅ |
| **记忆归档** | 正文 → 摘要 + 档案 | 作者点 **同步设定** | ❌ 需确认 |
| **规划** | 调整后面怎么写 | 作者说「同步细纲」等 | ❌ 显式发起 |

**删章例外**：删除时 outline 条目（含摘要）同步移除，不调用 LLM——数据一致性，不是「维护 batch」。

---

## 10. 作者感知：三助手 + 一个按钮

聊天里三个助手，对应写 / 改 / 问：

| 感受 | 例子 |
|------|------|
| **写** | 续写、开新书、生成设定 |
| **改** | 改名、加规则、同步细纲 |
| **问** | 伏笔在哪？节奏如何？ |

不在聊天里的：**[同步设定]**、侧边栏直接编辑、保存章节。

没有「第五个维护助手」。维护是 Editor 上的按钮，不是 Chat 里的 Agent。

---

## 11. 典型场景

**连写三章**  
写 → 正文落盘 → 当晚或次日点 **同步设定** → 摘要与档案一次补齐。

**改了一章旧稿**  
保存 → 旧摘要作废 → 下次 **同步设定** 时重生成该章摘要并更新档案。

**删一章**  
正文与摘要条目一并删除；未来细纲若仍引用该章，作者自行 **同步细纲** 调整。

**改角色名**  
Chat 改 → 系统提醒检查关系与细纲 → 档案与正文对齐仍靠 **同步设定** batch。

**开新卷前对账**  
先 **同步设定** 对「已发生的事实」→ 再 **同步细纲** 调「后面的打算」——两步分开，顺序自定。

---

## 12. 验收清单

从作者需求出发，功能是否达标看以下条目：

1. **写得顺**：写章 / 保存仅动正文；不写章后自动 sync、不改细纲。  
2. **记得住**：**同步设定** 后，问、续写能用到最新摘要与档案。  
3. **改得动**：局部改可控；级联以提醒为主；正文不被系统擅自改。  
4. **摘要与档案同 batch**：章摘要只在 **同步设定** 中生成/更新。  
5. **删章一致**：删章时摘要条目同步移除。  
6. **规划独立**：未来细纲与 **同步设定** 按钮分离。  
7. **三助手 + 一按钮**：Chat 无第五个维护 Agent；「同步设定」引导点按钮。  
8. **角标诚实**：缺摘要或档案落后时提示待同步。
