## 1. 这篇管什么

| | |
|--|--|
| **作者想干什么** | 「第几章埋的线？」「节奏怎么样？」「有没有前后矛盾？」「之前聊过什么？」 |
| **入口** | Chat → Reader Agent |
| **铁律** | **只读**。不改正文、不改档案、不改细纲 |

**答得准的前提：** 作者点过 **[同步设定]**，章摘要和档案是最新的。还没同步时，应降级读最近正文，并告诉作者「设定可能还没同步」。

**不属于「问」：** Chat 说「扫描伏笔 / 同步设定」→ 应引导点顶栏按钮，见 [01-对](01-对-功能设计.md)。

---

## 2. 功能一览

| 作者大概想问 | 做什么 | 工具 | 主要读什么 |
|--------------|--------|------|------------|
| 某字段 / 某章写了啥 | 读内容 | `read_novel_content` | 字段 md、摘要或正文 |
| 有没有设定矛盾 | 一致性分析 | Reader 自行推理 | read_novel_content 读档案 + 章节正文 |
| 节奏快还是慢 | 节奏分析 | Reader 自行推理 | read_novel_content 读最近 N 章正文 |
| 伏笔埋了没、收了没 | 伏笔健康度 | `foreshadowing_status` | foreshadowing 字段（正则解析，零 LLM） |
| 跨章搜索关键词 | 全文检索 | `read_novel_content`（search） | 章节正文 + 字段内容 |
| 之前聊过什么 | 搜对话记忆 | `search_memory` | MEMORY / chat |

> **设计决策：** `check_consistency` / `analyze_pacing` 已移除。Reader 本身就是 LLM，读完内容后自行分析矛盾、节奏等本就是其本职工作，不需要再包一层 LLM 调用。

## 3. 完整请求链路

```
作者发消息：「第几章埋的伏笔？」

  ▼
intent.resolve_fast_route()          [# 确定性路由]
  │
  ├── 匹配「扫描伏笔」→ FastGuideSyncButton → 引导点顶栏按钮
  └── 其余「问」类请求 → 返回 None → 交给 LLM
        │
        ▼
LeadAgent._plan_or_handoff()         [# Harness 单次 LLM 调用]
  │
  ├── Fast Route 命中 → 按上述处理
  └── LLM 调用（tool_schemas: handoff_to_reader / _creator / _editor）
        │
        ├── LLM 输出 tool_call handoff_to_reader → handle_handoff()
        │     └── execute_subagent("reader", task, state, w)
        │
        ├── LLM 输出纯文本（闲聊）→ 直接回复
        └── LLM 输出 JSON Plan（复合任务）→ Plan-Execute 循环

  ▼
Reader Subagent                      [# 独立上下文执行]
  │
  ├── system prompt: reader.md (审阅者角色)
  ├── allowed_tools: read_novel_content / foreshadowing_status /
  │                  search_memory / task_complete
  ├── max_tool_rounds: 4
  │
  └── ReAct 循环
        │
        ├── 轮1: LLM 判断需 read_novel_content → 读取文章/档案
        ├── 轮2: 如需伏笔汇总 → foreshadowing_status
        ├── 轮3: LLM 综合信息 + 自行分析矛盾/节奏/弧光 → 组装回答
        └── 轮4: task_complete(message="最终回复")

  ▼
结果返回 Lead Agent → 流式推送给用户
```

## 4. 工具详解

### 4.1 read_novel_content — 核心读取

**文件**：`novel_agent/agent/tools/read.py`

**content_type 支持：**

| content_type | 实际读取 | 说明 |
|---|---|---|
| `settings` | `settings_md_content` | 写作设定（世界观/力量体系/卷级规划） |
| `characters` | `characters_md_content` | 角色档案 |
| `locations` | `locations_md_content` | 地点档案 |
| `relationships` | `relationships_md_content` | 关系图谱 |
| `foreshadowing` | `foreshadowing_md_content` | 伏笔清单 |
| `outline_future` | `outline_future` | 未来大纲 |
| `chapter_summaries` | `assemble_historical_outline()` | 已完成章摘要（outline_structure） |
| `chapter` | `load_chapter()` | 指定章正文（需 chapter_num） |
| `recent_chapters` | 最近 N 章摘要 + 正文片段 | 默认 3 章，每章正文截断 800 字 |
| `search` | 关键词跨章 + 跨字段搜索 | 搜索正文和所有字段内容 |

**关键参数：**
- `query`：关键词过滤，只返回包含关键词的段落（前后各 1 行），节省 token
- `chapter_num`：指定章节号（content_type=chapter 时必填）
- `count`：recent_chapters 时的章数（默认 3），search 时的结果数（默认 5）

**内容截断策略：**
- 字段内容 + query 过滤 → 返回匹配段落
- 章节正文 → 取头 2/3 + 尾 1/3，省略中间（`_truncate_content`，默认 3000 字）
- 最近章正文 → 截断 800 字
- query 搜索 → 每个结果最多 5 行上下文

**批量读取**：content_type 支持列表，多个类型结果用 `---` 分隔。

### 4.2 foreshadowing_status — 伏笔健康度

**文件**：`novel_agent/agent/tools/analysis.py`

**逻辑**：解析 `foreshadowing_md_content` 中的伏笔条目（按状态图标正则匹配），按状态分组统计。**零 LLM 调用**，纯正则解析。

**状态解析**：
- 🔵 规划中 → `planning`
- 🟡 活跃中 → `active`
- 🟢 已回收 → `resolved`
- 🔴 已废弃 → `abandoned`
- ⚪ 已偏移 → `deviated`

**filter_status 参数**：
- `all`：全部伏笔
- `active`：仅有活跃中的
- `unresolved`：规划中 + 活跃中

**输出**：按状态分组的伏笔列表 + 总计数量。

此工具只做**统计汇总**，深度分析（如"哪些伏笔该回收了"）由 Reader 在 task_complete 中自行推理。

### 4.3 search_memory — 对话记忆搜索

**文件**：`novel_agent/agent/tools/memory.py`（handler） + `read.py`（注册在 memory toolset）

**逻辑**：调用 `search_memory()` 搜索 MEMORY.md + chat 记录。

**参数**：
- `query`：搜索关键词
- `top_k`：返回结果数，默认 5
- `source_filter`：`memory` / `daily` / `field` / `chat`，不传则搜全部

**输出**：`[来源标签] 内容片段`，每条截断 300 字。

## 5. Reader 提示词（reader.md）

**文件**：`novel_agent/agent/templates/subagent/reader.md`

**角色定位**：专业小说审阅者，服务长篇作者，帮记住跨章细节、追踪角色弧光、发现矛盾。

**审阅哲学：**
- 基于证据：所有回答基于实际读取内容；截断则再次读取
- 长篇视角：不看单章看跨章演变
- 关联回答：补充关联角色、章节出处、演变轨迹、潜在问题
- 主动发现：发现矛盾/节奏问题/未回收伏笔/弧光断裂时主动指出
- 只读边界：不修改文件；发现问题只报告

**工具使用：**
- 读内容：`read_novel_content`（+ 必要时 `search_memory` 查历史上下文）
- 伏笔汇总：`foreshadowing_status`（结构化统计）
- 矛盾/节奏/弧光/信息密度：Reader 读完内容后在 task_complete 中自行分析，不依赖独立工具

**task_complete message 规范**：所有回复统一结构——`📖 [标题]` + 正文 + `📂 已读取：[来源]  [来源]`。

**长篇专项分析维度（宽泛问题时主动覆盖）：**
1. 跨章一致性
2. 角色弧光追踪
3. 伏笔健康度（埋太多/超期未回/回收质量）
4. 节奏曲线（高潮密集 vs 平淡铺垫）
5. 信息密度（塞太多 vs 纯过渡）

## 6. 路由规则

### 6.1 Fast Route（确定性匹配，跳过 LLM）

**文件**：`novel_agent/agent/multi_agent/intent.py`

| 用户说 | 路由 | 说明 |
|--------|------|------|
| 扫描伏笔 / 检查伏笔 / 伏笔扫描 | `FastGuideSyncButton` | 引导点顶栏按钮，不走 Reader |
| 同步设定 / 补摘要 / 同步角色 / 同步关系 / 同步伏笔 | `FastGuideSyncButton` | 同上 |
| 其余所有"问"类请求 | 返回 None | 走 LLM Harness → handoff_to_reader |

### 6.2 LLM Harness

Fast Route 未命中时，Lead Agent 进行单次 LLM 调用（`drain_stream`），tool_schemas 包含 `handoff_to_reader` / `handoff_to_creator` / `handoff_to_editor`。LLM 根据 lead_harness.md 中的路由规则自行输出 tool_call。

### 6.3 Leader Harness 路由规则

**文件**：`novel_agent/agent/templates/subagent/lead-router.md`

三问框架：
1. 文件存在吗？不存在 → Creator
2. 改动范围多大？整体重写 → Creator，局部增删改 → Editor
3. 用户说辞关键词匹配

"问"类请求自然落入 handoff_to_reader（没有写入语义、不触发 Creator/Editor）。
