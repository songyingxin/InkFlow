# 历史版本备份设计

> **上位：** [01-写-功能设计.md](01-写-功能设计.md) §5「待改进」  
> **交叉引用：** [00-作者需求逻辑.md](00-作者需求逻辑.md) §3「正文是最高真相」  
> **参考：** 飞书文档的版本管理策略（实时保存 + 版本快照 + 历史版本恢复），快照粒度按写作场景调至 30 分钟

---

## 1. 设计理念

### 1.1 参考飞书文档

飞书文档的版本策略可以拆成三层：

| 层 | 飞书做法 | InkFlow 对等实现 |
|----|---------|-----------------|
| **实时保存** | 每次编辑立即同步到服务器，无需手动保存 | 停止输入 **5 秒**后防抖保存到本地文件；连续编辑最长 **30 秒**强制保存一次 |
| **版本快照** | 每 **5 分钟**自动生成一个版本快照 | 同一文件每 **30 分钟**最多生成一个快照（`_SNAPSHOT_INTERVAL_SECONDS = 1800`） |
| **版本恢复** | 历史记录列表 → 预览 → 一键恢复至任意版本 | 版本列表 → 预览（全文） → 恢复（自动备份当前版本） |

差异：飞书是在线服务，实时写网络请求无本地磁盘负担，5 分钟粒度适合多人协作的 blame 追踪。InkFlow 是本地桌面软件，单人写作场景下 5 分钟过快（翻版本列表难以定位），调整为 30 分钟——刚好匹配大多数作者一次连续写作的审视节奏。

### 1.2 设计目标

长篇连载作者的核心诉求：

- **写过后悔** → 重写一章后想找回上一版
- **误删找回** → 删了一段重要的，隔几天才发现
- **版本对比** → 对比 AI 生成的版本和自己改后的版本
- **无心智负担** → 不想手动版本号、不想 `Ctrl+S`

**一句话：** 像飞书文档一样，系统默默帮你记下每一份中间稿，作者只负责写作。

## 2. 全链路：从敲字到恢复

```
作者敲字
  │
  ▼
markDirty() → isDirty = true
  │
  ▼
watch(isDirty) 触发
  │
  ├── 启动 30s 强制保存计时器（MAX_DIRTY_MS）
  └── 启动 5s 防抖计时器（DEBOUNCE_MS）
        │
        │  作者继续敲字 → 防抖重置
        │  作者停止 5s  → 触发保存
        │  或 30s 到期  → 强制保存
        ▼
doSaveCurrent(isAuto=true)
  │
  ▼
POST /api/chapters/update/{idx}
  │
  ▼
chapter_service.update_chapter()
  │
  ▼
NovelMemory.save_chapter() → _save_text_file()
  │
  ├── 内容有变化？
  │     ├── 是 → _do_backup()
  │     │         ├── 同文件距上次快照 < 1800s？ → 跳过
  │     │         └── 否则 → 旧内容写入 backups/YYYY-MM-DD/HHMMSS_原始文件名
  │     └── 否 → 跳过
  │
  └── 新内容写入 chapters/NNN.md

注：手动加章（add_chapter）、AI 续写完成（continue_writing）同样经过
save_chapter → _do_backup 路径，只是 HTTP 入口不同。

... 若干天后 ...

作者点章节旁的 🕐 按钮
  │
  ▼
GET /api/chapters/{idx}/backups  →  版本列表（含前 200 字预览 + hash）
  │
  ▼
作者点某个版本的 [预览]
  │
  ▼
GET /api/chapters/{idx}/backups/preview  →  全文（最多 5000 字）
  │
  ▼
作者点 [恢复]
  │
  ▼
POST /api/chapters/{idx}/backups/restore
  │
  ▼
restore_chapter_backup()
  ├── save_chapter() 将备份内容写为工作文件（触发 _do_backup，当前版本进历史）
  └── 清空该章的 content_summary → 同步设定角标亮起
```

## 3. 自动保存机制

### 3.1 触发逻辑

| 场景 | 行为 |
|------|------|
| 用户输入内容 | `isDirty` 变为 `true`，启动 5s 防抖 |
| 用户连续输入 | 每次输入重置防抖计时器 |
| 用户停止 5 秒 | 自动保存 |
| 连续编辑超过 30 秒 | 强制保存一次（防丢） |
| Chat 生成完成 | `startGeneratedSaveTimer()` 立即保存 |
| 用户手动点保存 | `doSaveCurrent()` 立即保存 |
| 用户切换侧边栏目标 | 如有未保存 → 弹出「未保存」确认框 |

### 3.2 实现

- [useAutoSave.ts](file:///d:/Novel-LangGraph/InkFlow/novel_agent/client/src/composables/useAutoSave.ts)
- `DEBOUNCE_MS = 5000`（停止输入到保存的延迟）
- `MAX_DIRTY_MS = 30000`（最长不保存时间，强制保存保底）
- 使用 `watch(isDirty)` 替代 `setInterval` 轮询

> **保存 ≠ 快照：** 保存只是把正文写入工作文件，快照才进 `backups/`。频繁保存不会导致备份爆炸——因为快照有 30 分钟节流（见 §4）。

### 3.3 为什么用防抖而非实时

| 方案 | 优点 | 缺点 |
|------|------|------|
| 实时保存（飞书） | 零延迟，不丢数据 | 每字一次 HTTP 请求 + 磁盘写入 + 备份检查，开销极大 |
| **5s 防抖（当前）** | 省 I/O、合并多次修改 | 最多丢最后 5s 的内容 |
| 旧方案 60s 轮询 | 简单 | 最多丢 59s 内容，且保存和编辑不同步 |

## 4. 版本快照机制

### 4.1 快照触发

每次 `_save_text_file` 发现**内容有变化**时，触发 `_do_backup`。`_do_backup` 内部做了节流：

| 规则 | 值 | 代码位置 |
|------|----|---------|
| 同一文件最小快照间隔 | `1800` 秒（30 分钟） | [novel.py#L33](file:///d:/Novel-LangGraph/InkFlow/novel_agent/agent/memory/novel/novel.py#L33) |
| 节流追踪 | `NovelMemory._backup_timestamps` 字典 | [novel.py#L71-L74](file:///d:/Novel-LangGraph/InkFlow/novel_agent/agent/memory/novel/novel.py#L71-L74) |

**效果：** 如果在 30 分钟内频繁保存（防抖自动保存 + 手动保存），只有第一次会生成快照，之后的都跳过。写 4 小时约产生 8 个快照，版本列表清爽可控。

### 4.2 快照内容

- 备份 **上一次**的完整文件内容（不是 diff）
- 保存在 `backups/YYYY-MM-DD/HHMMSS_原始文件名`
- 快照的 SHA-256 前 16 位写入列表 API 返回的 `hash` 字段，前端用来标记「与当前相同」

### 4.3 备份覆盖范围

| 文件类型 | 触发写入的操作 | 备份 |
|---------|--------------|------|
| `chapters/NNN.md` | 续写、重写、编辑器保存、手动加章、Chat 生成 | ✅ |
| `settings.md` | `generate_settings`、`update_field`、`daily_sync` | ✅ |
| `characters.md` | `generate_characters`、`update_field`、`daily_sync` | ✅ |
| `locations.md` | `generate_locations`、`update_field`、`daily_sync` | ✅ |
| `relationships.md` | `generate_relationships`、`update_field`、`daily_sync` | ✅ |
| `foreshadowing.md` | `generate_foreshadowing`、`update_field`、`daily_sync` | ✅ |
| `outline_future.md` | `generate_outline`、`update_outline` | ✅ |
| `outline_structure.json` | 章节增删改 | ✅ |
| `meta.json` | 元信息更新 | ✅ |

**所有文件走同一套 `_save_text_file → _do_backup` 流水线。**

## 5. 存储布局

```
workspace/牧神记/
├── chapters/
│   ├── 001.md              ← 工作文件（当前版本）
│   ├── 002.md
│   └── ...
├── settings.md
├── characters.md
├── outline_future.md
├── ...
├── meta.json
├── outline_structure.json
└── backups/
    ├── 2026-06-25/
    │   ├── 143015_001.md   ← 第 1 章的 14:30:15 快照
    │   ├── 150020_001.md   ← 第 1 章的 15:00:20 快照（≥30min 后）
    │   ├── 143015_settings.md
    │   └── ...
    ├── 2026-06-24/
    │   └── ...
    └── ...                  ← 超过 10 天的目录次日自动删除
```

命名规则：`HHMMSS_原始文件名`（如 `143015_001.md`），其中 `HHMMSS` 取快照生成时刻。

## 6. 保留与清理

| 策略 | 值 | 代码 |
|------|----|------|
| 最大保留天数 | `10` 天（`_MAX_BACKUP_DAYS`） | [novel.py#L29](file:///d:/Novel-LangGraph/InkFlow/novel_agent/agent/memory/novel/novel.py#L29) |
| 清理触发 | 每次生成新快照时，按天检查一次 | `_maybe_cleanup_old_backups` |
| 清理粒度 | 删除整个 `YYYY-MM-DD` 目录 | `_cleanup_old_backups` |
| 删章的影响 | 只删 `chapters/NNN.md`，`backups/` 中的对应快照独立存活到 10 天 | — |

**为什么是 10 天而非永久：**

| 永久保留 | 10 天滚动 |
|---------|----------|
| 长篇写几百章，备份量可达数 GB | 可控存储 |
| 作者通常隔几天才发现问题 | 10 天足够覆盖写完→改→后悔的周期 |
| 无限历史对写作场景是过度设计 | 够用就好 |

## 7. 恢复机制

### 7.1 API

#### 列出历史版本

```http
GET /api/chapters/{idx}/backups
```

响应（[chapters.py#L105-L110](file:///d:/Novel-LangGraph/InkFlow/novel_agent/api/routes/chapters.py#L105-L110)）：

```json
{
  "chapter_idx": 5,
  "current_hash": "a1b2c3d4e5f6g7h8",
  "backups": [
    {
      "timestamp": "2026-06-25T14:30:15",
      "date": "2026-06-25",
      "time": "14:30:15",
      "size": 3240,
      "preview": "前200字...",
      "hash": "e5f6g7h8i9j0k1l2"
    }
  ]
}
```

- 按时间倒序
- `current_hash` 为当前版本哈希，前端标记 `hash === current_hash` 的条目为「与当前相同」
- `preview` 为正文前 200 字

#### 预览备份

```http
GET /api/chapters/{idx}/backups/preview?timestamp=2026-06-25T14:30:15
```

返回全文，超过 5000 字截断并标记 `is_full: false`。

#### 恢复

```http
POST /api/chapters/{idx}/backups/restore
Content-Type: application/json

{ "timestamp": "2026-06-25T14:30:15" }
```

### 7.2 恢复流程细节

[novel.py#L559-L581](file:///d:/Novel-LangGraph/InkFlow/novel_agent/agent/memory/novel/novel.py#L559-L581) `restore_chapter_backup`：

1. 读取备份文件内容
2. `save_chapter()` 写入 — 此时 `_do_backup` 自动触发，**当前版本成为新快照**（不丢数据）
3. 清空 `content_summary` — 正文变，摘要作废
4. 更新 `content_hash`、`word_count`、`is_written`
5. 保存 `outline_structure.json`
6. 对话日志记录：`[页面操作] 恢复了第N章到 ... 的版本`
7. 返回完整 `state`，前端刷新侧边栏和编辑器

**恢复是可逆的：** 恢复后旧版本在下一次列表中可见，可以再恢复回去。

## 9. 与其他功能的交互

### 9.1 重写（`regenerate_chapter`）

`handle_regenerate_chapter` 调用 `_finalize_chapter_write` → `save_chapter` → `_do_backup`。

**重写前自动备份当前版本。** 作者在版本历史中能找到重写前的版本。

### 9.2 编辑器保存（`update_chapter`）

每次保存触发 `save_chapter` → 30 分钟节流内的第一次保存生成快照。形成合理粒度的版本链。

### 9.3 删章

`delete_chapter` 只删除 `chapters/NNN.md` 和 `outline_structure` 中的条目。`backups/` 中的快照完全不受影响，独立存活到 10 天过期。作者可在版本历史中查看已删章节的往期版本。

### 9.4 同步设定（`daily_sync`）

恢复操作会清空章节的 `content_summary`，导致 `_has_pending_sync` 返回 `true`，顶栏角标亮起。作者点「同步设定」后重生成摘要。

### 9.5 AI 续写（`continue_writing`）

续写生成完成后 `_finalize_chapter_write` → `save_chapter` → 若之前有正文，旧版本进快照。AI 生成的初版和作者手动修改的版本都在历史中可见。

## 10. 边缘情况

| 场景 | 行为 |
|------|------|
| 章节从未被写过，点版本历史 | 弹窗显示「暂无历史版本」 |
| 章节被删后再查版本历史 | 标题显示「已删除」红色标记；备份仍可预览和恢复 |
| 恢复到被删章节的历史版本 | 章节在 outline_structure 中恢复，正文写入 work 目录 |
| 章节没有任何快照 | 弹窗显示「暂无历史版本」 |
| 恢复到一个不存在的 timestamp | HTTP 404 |
| 点恢复不确认 | 弹出二次确认弹窗；取消则不动 |
| 连续 30 分钟内多次保存 | 只在第一次保存时生成快照，其余跳过 |
| 恢复时 `backups/` 目录不存在 | `list_chapter_backups` 返回空列表 |
| 同时打开两个编辑器修改同一章 | 各自保存时分别生成快照；后保存的版本覆盖前者（与飞书一致） |
| 服务器重启 | `_backup_timestamps` 字典清空，不影响快照（只是节流重置，不会丢数据） |
| 预览超过 5000 字的备份 | 截断并标记 `is_full: false` |
| 恢复后被恢复的章节摘要过期 | 自动清空摘要 → 同步设定角标提醒 |

## 11. 自测

| 操作 | 预期 |
|------|------|
| 打开编辑器，修改内容，等待 5 秒 | 自动保存触发 |
| 连续修改超过 30 秒不停止 | 30 秒时强制保存一次 |
| 30 分钟内保存同一章多次 | `GET /api/chapters/N/backups` 只返回 1 个快照 |
| 间隔 30 分钟以上保存同一章两次 | 返回 2 个快照 |
| 点预览 | 弹窗底部显示备份全文 |
| 点恢复 | 编辑器刷新为历史版本内容；版本列表刷新，旧当前版本出现在列表中 |
| 恢复后检查摘要 | 该章 `content_summary` 为空 |
| 删章后查版本历史 | 快照仍可列出（10 天有效） |
| 10 天前的备份目录 | 已被自动清理 |
| 重写第 N 章后查版本历史 | 重写前的版本在列表中 |

## 13. 不做的

| 不做 | 原因 |
|------|------|
| 字段文件的独立版本历史 UI | 6 个字段文件已有备份，但 UI 入口与章节不同，优先级更低，后续按需补 |
| 备份永久保留 | 10 天覆盖大多数「后悔」场景，永久保留会带来存储膨胀 |
| 跨章 diff | 只做同章不同版本的对比 |
| 分支（branch）管理 | 不是写作工具的核心需求 |
| 手动创建快照 | 快照全自动，不暴露手动入口（与飞书一致） |
