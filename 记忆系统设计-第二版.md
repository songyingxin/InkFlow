## 1. 记忆系统总体架构

InkFlow 的记忆系统由两个职责明确的子系统组成：

```
┌──────────────────────────────────────────────────────────────┐
│                    InkFlow 记忆系统                            │
├──────────────────────────────┬───────────────────────────────┤
│     对话记忆系统              │       小说记忆系统              │
│  （管理 Agent 与用户的交互）    │  （管理小说创作的结构化知识）     │
├──────────────────────────────┼───────────────────────────────┤
│ · 对话记录 (chat.db)         │ · 字段文件 6 个                │
│ · 每日日志 (memory/YYYY-MM-DD)│   settings / characters /     │
│ · 长期记忆 (MEMORY.md)       │   relationships / foreshadowing│
│ · 分层蒸馏管线                │   outline_historical /         │
│ · 上下文构建与缓存            │   outline_future               │
│ · 检索引擎 (FTS5+jieba)      │ · 章节全文 (chapters/NNN.md)  │
│ · 归因追踪 (MemoryProvenance) │ · 章节索引 (outline_structure) │
│ · 消息压缩 (LLM 摘要)        │ · 元信息 (meta.json)           │
│                              │ · 同步缓存 (sync_state.json)   │
├──────────────────────────────┼───────────────────────────────┤
│         共享基础设施                                           │
│  · 备份机制 (backups/)                                       │
│  · 检索引擎 (FTS5+jieba，对话记忆和小说记忆共用)                │
└──────────────────────────────────────────────────────────────┘
```

**两个子系统的关系**：

- 对话记忆系统是**通用 Agent 基础能力**，负责"记住用户说了什么"
- 小说记忆系统是**领域专用知识管理**，负责"记住小说写了什么"
- 对话记忆的蒸馏产物（MEMORY.md 中的创作决策）会**影响**小说记忆的字段内容
- 小说记忆的字段内容会**注入**对话记忆的上下文构建中

**文件体系**：

```
workspace/书名/
├── MEMORY.md              ← 长期记忆（对话记忆：提炼后的持久事实）
├── settings.md            ← 设定（小说记忆）
├── characters.md          ← 角色档案（小说记忆）
├── relationships.md       ← 关系图谱（小说记忆）
├── foreshadowing.md       ← 伏笔清单（小说记忆）
├── outline_historical.md  ← 历史大纲（小说记忆）
├── outline_future.md      ← 未来大纲（小说记忆）
├── outline_structure.json ← 章节索引（小说记忆）
├── meta.json              ← 元信息（小说记忆）
├── sync_state.json        ← 同步缓存（小说记忆：chapter_hashes + field_sync_hashes）
├── chat.db                ← 对话记录（对话记忆）
├── memory/
│   └── YY-MM-DD.md        ← 每日日志（对话记忆）
├── chapters/
│   └── NNN.md             ← 章节全文（小说记忆）
└── backups/               ← 共享基础设施
```

## 2. 对话记忆系统

对话记忆系统负责管理 Agent 与用户的交互历史，确保 Agent 在多轮对话中保持上下文连贯。

### 2.1 分层蒸馏管线

```
chat.db 原始对话
    │  每 20 轮 → _supplement_daily_log（补充日志遗漏）
    │  每 10 轮 → _generate_daily_log（结构化日志生成）
    │            → _generate_retain_from_log（W/O/S 结构化事实提取）
    ▼
memory/YY-MM-DD.md 每日日志
    │  跨天触发 → _distill_daily_to_memory（提取持久事实）
    ▼
MEMORY.md 长期记忆
    │  pending ≥ 5 条 → rewrite_memory_md（LLM 整合重写）
    │  Agent 手动 → memory_append / memory_rewrite
    ▼
   (带备份 + 实体丢失检测的安全写入)
```

**关键实现细节**：

- `_distill_daily_to_memory`：从日志中提取持久事实，忽略一次性操作
- `_generate_retain_from_log`：提取三级结构化事实 — **W (World)** 世界事实 / **O(c=N)** 用户观点+置信度 / **S (Summary)** 模式总结
- `rewrite_memory_md`：合并新旧内容，去重去矛盾，按 `## 创作决策 / ## 故事状态 / ## 重要变更` 分组
- **安全机制**：重写前备份 MEMORY.md（保留最近 5 个版本），重写后检测关键实体丢失率，超过 50% 则回退

### 2.2 上下文构建与缓存（Stable Prefix）

```python
build_memory_context() 的两级缓存：

1. Stable Prefix Cache
   - 基于 MEMORY.md 的 SHA-256 hash
   - 内容不变则前缀不变 → LLM KV cache 持续命中（对标 Hermes 冻结快照）
   - 对话模式 (dialogue) 下直接注入

2. Memory Context Cache
   - Key = (title, round_count, chat_rounds, session_id)
   - TTL 可配置
```

### 2.3 检索引擎（与小说记忆共享）

```
FTS5 Trigram 广域召回 + jieba 语义重排：

  查询输入
    │
    ├──→ jieba 分词 → 长词(≥3字) → FTS5 Trigram 精确匹配
    │                → 短词(1-2字) → SQLite LIKE 精确匹配
    │
    └──→ 合并去重 → 时间衰减 → jieba 分词重叠度重排 → 最终结果

  其他能力：
    - 重叠滑窗分块：400 token 目标 + 80 token 重叠 + 行边界保留
    - 去重：SHA-256 hash，重复内容 increment reinforcement_count
    - 时间衰减：fact_type 差异化衰减率
      identity(0.1) < decision(0.3) < preference(0.5) < temporal(1.0) < state(2.0)
    - 强化：reinforcement_count 的 log1p 加权
    - source → fact_type 自动映射：field→identity, memory→decision, daily→temporal
```

### 2.4 消息压缩（LLM 摘要压缩）

当 token 使用量超过窗口阈值时，使用 LLM 生成对话摘要替代历史消息。

```
触发条件：token 使用量 > 窗口 × 50%

压缩流程：
  1. 预压缩记忆刷新：LLM 提取关键事实 → 写入 MEMORY.md（防止压缩丢失持久信息）
  2. LLM 生成摘要：历史消息 → 简洁摘要，保留关键操作和结果
  3. 替换历史：first_user + [历史摘要] + 最近 keep_recent_messages 条消息

边界保护：assistant(tool_calls) + tool_response 作为一个整体处理

未来优化：压缩模型可切换为小模型（如 Qwen3-0.6B），进一步降低成本
```

### 2.5 归因追踪

```python
class MemoryProvenance:
    source_type: str    # "chat" | "daily" | "chapter" | "distill"
    source_id: str      # 对话ID / 日期 / 章节号
    original_text: str  # 原始文本片段
    distilled_at: str   # 提炼时间戳
    distiller: str      # "agent" | "system" | "rewrite"
```

每一条记忆都记录完整来源链路，支持人工回溯验证提炼质量。

## 3. 小说记忆系统

小说记忆系统负责管理小说创作的结构化知识，确保角色、设定、伏笔等与章节内容保持同步。

### 3.1 字段体系

6 个字段文件 + 1 个章节索引 + 1 个元信息文件，构成小说的结构化知识图谱：

| 文件 | 职责 | `read_ch_field` | 更新触发 |
|------|------|-----------------|---------|
| settings.md | 世界观/力量体系/核心冲突 | `settings_read_ch` | 章节演进 |
| characters.md | 角色档案（核心7项/活跃4项/退场1行） | `characters_read_ch` | 角色行为变化 |
| relationships.md | 关系图谱（现状+演变转折点） | `relationships_read_ch` | 关系发展 |
| foreshadowing.md | 伏笔清单（5状态机+7类型） | `foreshadowing_read_ch` | 伏笔埋设/回收 |
| outline_historical.md | 历史大纲（已完成章节摘要） | `outline_historical_read_ch` | 章节完成 |
| outline_future.md | 未来大纲（规划中章节） | None（级联更新） | 其他字段更新后自动级联 |
| outline_structure.json | 章节索引（编号+标题+字数） | — | 章节增删 |
| meta.json | 元信息（进度/轮次） | — | 所有操作 |
| sync_state.json | 同步缓存（chapter_hashes + field_sync_hashes） | — | 章节保存/字段更新 |

### 3.2 字段更新机制

#### 3.2.1 问题

章节内容变更后，字段文件不会自动更新。当前过期检测机制基于 `read_ch_field`（已读到第几章），只能检测**新增章节**，无法覆盖所有变更场景。

**导致字段过期的完整场景**：

| 场景 | 触发方式 | 检测方式 | 实际影响 |
|------|---------|---------|---------|
| 写新章节 | `continue_writing` | ✅ `field_read_ch < chapter_idx` | 新角色/关系/伏笔未同步 |
| 重写章节 | `regenerate_chapter` | ✅ `chapter_hashes` 变更 → 快照不一致 | 角色行为/关系可能已变 |
| 修改章节内容 | `update_chapter` API | ✅ `chapter_hashes` 变更 → 快照不一致 | 同上 |
| 用户直接编辑章节文件 | 磁盘操作 | ✅ `detect_disk_changes` → `chapter_hashes` 更新 → 快照不一致 | 同上 |
| 删除章节 | `delete_chapter` | ✅ `chapter_hashes` 移除条目 → 快照不一致 | 伏笔/角色可能需移除 |

**受章节更新影响的字段**（5 个直接 + 1 个级联）：

| 字段 | `read_ch_field` | 影响方式 |
|------|----------------|---------|
| outline_historical | `outline_historical_read_ch` | 直接：读章节内容生成大纲 |
| settings | `settings_read_ch` | 直接：世界观/力量体系随章节演进 |
| characters | `characters_read_ch` | 直接：角色状态/能力随章节变化 |
| relationships | `relationships_read_ch` | 直接：人物关系随章节发展 |
| foreshadowing | `foreshadowing_read_ch` | 直接：伏笔埋设/回收/状态变更 |
| outline_future | `None` | 级联：通过 `_CASCADE_RULES` 在其他字段更新后自动级联 |

#### 3.2.2 方案设计

核心原则：**从源数据推导过期状态，而非手动维护变更列表。更新由用户主动触发，系统负责检测过期、提示和提供便捷入口。**

**变更追踪：`chapter_hashes` + `field_sync_hashes`**

当前 `read_ch_field` 只记录"已读到第几章"，无法检测章节内容变更。采用 **MD5 内容哈希比对** 方案：每个字段记录上次同步时的章节 MD5 快照，通过比对当前 MD5 与快照来推导过期状态。同步缓存存储在独立的 `sync_state.json` 文件中，与 `meta.json` 的核心元信息分离。

**`chapter_hashes` 字典**（源数据，章节文件当前内容指纹）：

- 在 `sync_state.json` 中存储 `chapter_hashes: dict[str, str]`，键为章节号（字符串），值为文件内容的 MD5 哈希
- `save_chapter` 保存后自动更新对应条目
- `delete_chapter_file` 删除后自动移除对应条目
- `detect_disk_changes()` 在 `_agent_node` 入口调用，检测外部编辑并更新

**`field_sync_hashes` 字典**（派生数据，字段上次同步时的快照）：

- 在 `sync_state.json` 中存储 `field_sync_hashes: dict[str, dict[str, str]]`，键为字段名（如 `"characters"`），值为该字段上次同步时的 `chapter_hashes` 快照
- 字段更新完成后，`field_sync_hashes[field_name] = copy(chapter_hashes)`
- 不同字段可以有不同的同步快照（用户可能只更新了角色，没更新关系）

**过期检测逻辑**（~~`_find_stale_fields`~~ **已从代码库移除**，以下为 v2 设计备忘）：

| 条件 | 含义 | 更新方式 |
|------|------|---------|
| `total_chapters > field_read_ch` | 有新增章节 | 回退增量更新 |
| `chapter_hashes[ch] != field_sync_hashes[field].get(ch)` | 章节内容变更 | 回退增量更新 |
| 两者均不满足 | 字段最新 | 无需更新 |

**为什么用独立文件 `sync_state.json`**：

- `meta.json` 读写频繁（每轮对话都更新轮次），保持精简
- `sync_state.json` 是纯派生数据，损坏可直接删除重建
- 500 章 × 5 字段的哈希快照约 120KB，不适合放在 meta.json 中
- 文件结构：

```json
{
  "chapter_hashes": {"1": "a1b2c3...", "2": "d4e5f6..."},
  "field_sync_hashes": {
    "characters": {"1": "a1b2c3...", "2": "d4e5f6..."},
    "relationships": {"1": "a1b2c3...", "2": "d4e5f6..."}
  },
  "last_sync_date": "2026-06-03"
}
```

#### 3.2.3 触发机制

> **状态（2025）**：`_find_stale_fields`、`update_stale_fields` 及过期检测已从代码库移除；以下为实现备忘，将在记忆系统 v2 重设计。

三种触发方式（历史设计，已移除）：

| 触发方式 | 时机 | 行为 |
|---------|------|------|
| ~~写新章节~~ | ~~`handle_continue_writing`~~ | ~~已移除自动过期流程~~ |
| ~~用户主动~~ | ~~`update_stale_fields`~~ | ~~工具已删除~~ |

**变更追踪**：`save_chapter`、`delete_chapter_file`、`detect_disk_changes` 自动维护 `chapter_hashes`，过期状态从 `_find_stale_fields` 推导，无需手动维护。

**重写/修改/删除章节不弹窗**：这些操作通过 `chapter_hashes` 自动追踪，下次写新章节或下次打开软件时会检测到并统一询问。

**确认行为**：用户确认 → 逐个调用 `handle_generate_field`，更新完成后 `field_sync_hashes[field] = copy(chapter_hashes)` 并保存 `sync_state.json`；用户拒绝 → 跳过，下次触发时再次询问。

#### 3.2.4 回退增量更新

检测到章节变更（MD5 快照不一致）时，将最早变更章节之前的 `field_read_ch` 回退，统一走增量更新：

1. 计算 `chapter_hashes` 与 `field_sync_hashes[field]` 的差异，得到变更章节列表
2. 取最早变更章节索引 `min_changed`，将 `field_read_ch` 临时回退到 `min_changed - 1`
3. 走正常的增量更新流程：读取 `field_read_ch` 之后的所有章节（包括变更章和新增章）
4. 将"当前字段内容 + 从变更点起的章节内容"发送给 LLM，基于变更内容修正字段
5. 更新完成后，`save_field_content` 将 `field_read_ch` 设为最新章节索引，`field_sync_hashes[field] = copy(chapter_hashes)`

**`iterative_generate_stream` 的合并逻辑**（回退 `field_read_ch` 后只有一种情况）：

- 有未读章节：只读未读章节（已包含变更章节）
- 无未读章节且无变更章节：基于当前字段内容让 LLM 直接修改

**回退 `field_read_ch` 后 LLM 会读到变更章节之后的所有章节**。理由：

- 变更章节之后的内容可能受变更影响（角色关系变化后的互动、伏笔回收后的情节），只读变更章节本身信息不完整
- 统一走增量更新路径，消除"变更章节 + 新增章节"两条分路合并的复杂逻辑
- 章节修改通常是自觉行为，每次修改量小，回退读取的额外 token 成本可控
- 少数重大改写后可通过 `update_stale_fields(full_refresh=True)` 触发全量重读

#### 3.2.5 更新工具

**新增 `update_stale_fields` 工具**（用户主动调用，无需弹窗确认）：

- 调用 `_find_stale_fields()` 检测所有过期字段
- 回退每个过期字段的 `field_read_ch` 到最早变更点，逐字段走回退增量更新
- 支持 `full_refresh=True`：读取所有章节全文，适用于重大改写后的全量重读
- 流式输出更新进度：`📋 正在更新{label}...` → `✅ 已更新：角色、关系、伏笔`
- 更新完成后 `field_sync_hashes[field] = copy(chapter_hashes)` 并保存 `sync_state.json`

#### 3.2.6 实现要点

- **新增 `sync_state.json` 文件**：存储 `chapter_hashes`（章节 MD5）、`field_sync_hashes`（字段同步快照）、`last_sync_date`（上次同步日期），与 `meta.json` 分离
- **移除 meta 字段**：`modified_chapters: list[int]`（由 `field_sync_hashes` 替代）
- **新增 `detect_disk_changes` 函数**：在 `_agent_node` 入口调用，比对 `chapters/*.md` 的 MD5 与 `chapter_hashes`，更新 `chapter_hashes`
- **修改 `_find_stale_fields`**：从 MD5 快照比对推导过期状态，返回过期字段列表和最早变更章节索引
- **修改 `save_chapter`**：保存章节后自动更新 `chapter_hashes`
- **修改 `delete_chapter_file`**：删除章节后自动移除 `chapter_hashes` 中对应条目，`field_sync_hashes` 通过 `copy(chapter_hashes)` 跟随清理
- **修改 `save_field_content`**：保存字段后自动 `field_sync_hashes[field] = copy(chapter_hashes)` 并保存 `sync_state.json`
- **修改 `handle_continue_writing`**：入口检测过期字段，弹窗询问是否更新
- **新增 `update_stale_fields` 工具**：注册到 tool schema，支持 `full_refresh` 参数；更新完成后 `field_sync_hashes[field] = copy(chapter_hashes)` 并保存 `sync_state.json`
- **修改 `iterative_generate_stream`**：检测到内容变更时，回退 `field_read_ch` 到最早变更章节之前，统一走增量更新路径
- **新增 `auto_stale_check` 逻辑**：进入 novel 时检查距上次字段同步时间，超过 24 小时则弹窗询问

### 3.3 小说创作的记忆需求分析

#### 3.3.1 作家视角的核心需求

| # | 需求                   | 当前覆盖                                     |
| - | -------------------- | ---------------------------------------- |
| 1 | **故事梗概记忆**（各弧发生了什么）  | settings.md（卷级规划）+ outline\_historical（已完成章节大纲）+ outline\_structure（章节索引） |
| 2 | **角色关系记忆**（随时间变化）    | relationships.md（"关系现状"+"关系演变：关键转折点"） |
| 3 | **角色背景记忆**（经历如何塑造性格） | characters.md（核心角色 7 项：角色/身份/实力/性格/外貌/背景/动机） |
| 4 | **角色情感记忆**（关键事件的影响）  | 章节生成时注入最近 5 章全文，LLM 直接从正文感知 |
| 5 | **角色动机记忆**（各角色当前目标）  | characters.md 核心角色的"动机"字段 |
| 6 | **创作设想记忆**（防止前后矛盾）   | MEMORY.md（## 创作决策/## 故事状态/## 重要变更）+ rewrite\_memory\_md 去重去矛盾 |
| 7 | **设定记忆**（世界观规则一致性）   | settings.md（风格定位/核心冲突/世界观/力量体系/卷级规划） |
| 8 | **伏笔记忆**（埋了什么坑没填）    | foreshadowing.md（5 状态机 + 7 种类型 + 伏笔状态报告） |

8 条需求全部已被现有机制覆盖，当前三层记忆架构无需新增层级或优化。

#### 3.3.2 事实层级与衰减率

事实天然存在层级，当前系统已通过 fact_type 差异化衰减率实现：

```
前提级事实（推翻几乎不可能）— 世界观基本规则、主角核心身份
  → fact_type=identity, 衰减率 0.1（最慢衰减）
  └─ 弧级事实（推翻需重大事件）— 主角核心关系、当前情感基调
      → fact_type=decision, 衰减率 0.3
      └─ 章级事实（推翻需合理转折）— 本期目标、临时盟友
          → fact_type=temporal, 衰减率 1.0
          └─ 场景级事实（下一句话可改变）— 当前所在位置、持有物品
              → fact_type=state, 衰减率 2.0（最快衰减）
```

当前 source → fact_type 自动映射（field→identity, memory→decision, daily→temporal）已覆盖主要场景。

## 4. 主流 Agent 记忆架构对比

### 4.1 Claude Code

| 维度            | 设计                                                          | 可借鉴之处              |
| ------------- | ----------------------------------------------------------- | ------------------ |
| **架构**        | 4 层记忆栈（Global → Project → Local → Auto Memory）              | Layer 叠加不覆盖，优先级递增  |
| **MEMORY.md** | 纯索引文件（每行一条指针，上限 200 行/25KB），正文在专题文件中                        | InkFlow 也可改为索引路由模式 |
| **记忆分类**      | user / feedback / project / reference，强制 Why + How to apply | 创作决策应包含 Why + How  |
| **提取机制**      | Perfect Fork Agent：分叉代理共享 prompt 前缀，利用 KV cache             | 可借鉴到记忆提炼的 LLM 调用优化 |
| **整理机制**      | /dream 四阶段：定向→收集信号→合并→裁剪索引                                  | 记忆定期整理的工程化思路       |

### 4.2 OpenClaw

| 维度       | 设计                                    | 可借鉴之处               |
| -------- | ------------------------------------- | ------------------- |
| **架构**   | 文件优先 + 混合搜索（BM25 + Vector）            | Hybrid 检索对中文小说内容更友好 |
| **身份层**  | SOUL.md（身份+价值观）+ HEARTBEAT.md（自主任务清单） | 可引入"创作哲学"文件         |
| **心跳机制** | 定时主动检查待办任务                            | 对长周期自主记忆维护极有价值      |
| **记忆刷新** | 上下文压缩前触发内存持久化                         | 防止关键信息随窗口滑动丢失       |

### 4.3 Hermes Agent

| 维度          | 设计                                                     | 可借鉴之处                                |
