# Phase 1: 需求分析 → PRD

**目标**: 把模糊的需求变成清晰的 PRD 文档。先搞清楚"做什么"，再谈"怎么做"。

**定位**: PRD 是整个 Vibe Coding 流程的起点。后续所有 Phase 都消费它——环境初始化依赖它选技术栈，技术方案依赖它设计规格，编码依赖它验证交付。不知道做什么，就无法选择技术栈。

## 执行流程

```
需求输入 → 上下文探索 → 识别项目特征 → 需求澄清 → 方案讨论 → 生成 PRD → 用户确认
```

### Step 1: 上下文探索

- 读取项目相关代码、文档、最近的 git 提交，理解项目现状
- 如果是已有项目，读取 `AGENTS.md` 获取项目约定
- 识别与需求相关的已有模块和服务

### Step 2: 识别项目特征

项目特征由**三个维度**决定，影响后续追问策略和 PRD 模板选择：

#### 维度一：项目规模

| 规模 | 判断标准 | PRD 策略 |
|------|---------|---------|
| **单功能** | ≤5 个子任务，单一组件类型 | 一个 PRD 文件搞定 |
| **大型系统** | >5 个子任务，或包含多种组件类型 | 先拆子系统，每个子系统一个 PRD |

大型系统必须先输出**系统分解图**，让用户确认拆分方式后再逐个子系统写 PRD：

```
系统名称
├── 子系统 A [组件类型: X] → PRD-A.md
├── 子系统 B [组件类型: Y] → PRD-B.md
└── 子系统 C [组件类型: Z] → PRD-C.md
```

#### 维度二：组件类型

每个子系统属于以下一种组件类型，决定 PRD 第 2-3 章的模板选择：

| 类型 | 特征 | 核心交付物 | 关键风险 |
|------|------|-----------|---------|
| **Web/App 产品** | 有 UI、用户交互、页面流程 | 页面+交互 | 体验差、状态遗漏 |
| **AI 功能** | 调用 LLM/AI 模型、生成式输出 | AI 输出+兜底策略 | 幻觉、延迟、成本失控 |
| **框架/库** | 提供 API 给其他开发者使用 | API 接口+文档 | API 设计错、扩展性差 |
| **算法/模型** | 数学推导、数据处理、训练评估 | 实现+精度指标 | 精度不达标、不可复现 |
| **CLI/工具** | 命令行交互、自动化脚本 | 命令+输出+退出码 | 参数设计差、管道不兼容 |
| **API/后端服务** | 网络接口、数据处理管道 | 端点+鉴权+部署 | 接口不稳、故障不恢复 |
| **数据处理** | ETL、数据管道、批处理 | 数据质量+吞吐量 | 数据丢失、不幂等 |

#### 维度三：原创 vs 复现

| 模式 | 特征 | 需求分析侧重点 |
|------|------|--------------|
| **原创** | 从零设计，没有参考实现 | 重点在"做什么"——需求澄清、方案对比 |
| **复现** | 有明确的参考实现（论文、开源项目、文档） | 重点在"怎么对齐"——参考映射、差异识别 |

复现模式需要额外输出**参考映射表**：

| 本项目模块 | 参考实现对应 | 对齐策略 | 已知差异 |
|-----------|------------|---------|---------|
| [模块A] | [参考项目的X模块] | 逐行对齐 / 行为等价 / 接口兼容 | [差异说明] |

### Step 3: 需求澄清

**不要直接写 PRD，先对齐理解。** 主动确认 > 被动等待。

#### 3a. 复述理解 + 主动追问（同步进行）

按以下格式输出，让用户逐项检查，同时附上追问：

```
1. 项目规模：[单功能 / 大型系统（含子系统拆分）]
2. 组件类型：[Web产品 / AI功能 / 框架库 / 算法模型 / CLI工具 / API服务 / 数据处理]
3. 实现模式：[原创 / 复现（附参考来源）]
4. 目标用户：[你理解的目标用户]
5. 核心功能：[3-5 个核心功能]
6. 不做的事情：[明确不做的功能]
7. 追问（2-5 个）：
   - [不确定的问题1]
   - [不确定的问题2]
```

理解有偏差则纠正后重新确认。追问维度按组件类型选择：

**通用追问**（所有类型必问）：

| 维度 | 追问内容 |
|------|---------|
| 边界 | 哪些这次不做？哪些永远不做？ |
| 技术约束 | 有技术栈限制吗？性能要求？ |
| 默认值 | 数量上限？保存时长？超时阈值？ |

**Web/App 产品追加**：

| 维度 | 追问内容 |
|------|---------|
| 用户与场景 | 目标用户是谁？使用场景？解决什么问题？ |
| 核心功能 | 最核心的 3-5 个？完成任务的流程？有哪些状态变化？ |
| 数据 | 存什么数据？从哪来？结构是什么？ |

**AI 功能追加**：

| 维度 | 追问内容 |
|------|---------|
| AI 行为 | 用什么模型？输入输出格式？期望的响应质量？ |
| 幻觉兜底 | AI 输出错误/编造时怎么处理？需要人工审核环节吗？ |
| 延迟与成本 | 可接受的响应延迟？Token 用量上限？单次调用成本上限？ |
| 上下文 | 需要多少上下文？上下文从哪来？超出窗口怎么截断？ |

**框架/库追加**：

| 维度 | 追问内容 |
|------|---------|
| API 设计 | 核心公开 API 有哪些？命名风格？同步还是异步？ |
| 扩展性 | 用户需要自定义哪些行为？扩展点/钩子有哪些？ |
| 兼容性 | 支持哪些运行环境？版本策略（semver）？Breaking change 策略？ |
| 开发者体验 | 错误信息风格？是否需要 CLI？文档深度？ |

**算法/模型追加**：

| 维度 | 追问内容 |
|------|---------|
| 数学规格 | 核心算法的数学公式/伪代码？输入输出维度？ |
| 数据 | 训练数据从哪来？格式？规模？评估集？ |
| 精度目标 | 期望的精度指标？基线对比？最低可接受精度？ |
| 可复现性 | 需要确定性结果吗？随机种子？浮点精度要求？ |

**CLI/工具追加**：

| 维度 | 追问内容 |
|------|---------|
| 命令结构 | 主命令+子命令？参数风格（posix/gnu）？ |
| 输出格式 | 默认输出格式？支持 JSON/YAML 输出？ |
| 管道兼容 | 是否支持 stdin/stdout 管道？退出码规范？ |
| 分发方式 | 全局安装？npx？Docker？ |

**API/后端服务追加**：

| 维度 | 追问内容 |
|------|---------|
| API 风格 | RESTful / GraphQL / gRPC？认证方式？ |
| 限流与容错 | 限流策略？熔断？降级？重试？ |
| 部署 | 单机 / 容器 / K8s？持久化方案？ |
| 可观测性 | 日志格式？监控指标？告警规则？ |

**数据处理追加**：

| 维度 | 追问内容 |
|------|---------|
| 数据源 | 数据从哪来？格式？频率？增量还是全量？ |
| 数据质量 | 脏数据比例？去重策略？缺失值处理？ |
| 幂等性 | 重复执行是否安全？断点续传？ |
| 吞吐量 | 数据量级？处理延迟要求？ |

**复现模式追加**：

| 维度 | 追问内容 |
|------|---------|
| 参考来源 | 论文？开源仓库？文档？哪个版本？ |
| 对齐程度 | 逐行对齐 / 行为等价 / 接口兼容？ |
| 已知差异 | 哪些地方故意不同？为什么？ |
| 验证方式 | 如何验证复现正确性？对比测试？精度对齐？ |

**追问规则**：
- 一次最多 3-5 个问题
- 多轮问答直到：理解一致、细节明确、边缘情况已讨论、用户自己也感到清晰
- 不要跳过基础问题，"显而易见"的细节也要确认

#### 3b. 产品验证三问

**适用场景**：面向用户的产品（C端/B端产品）。开发者工具、内部系统、框架/算法/CLI 可跳过。

1. **用户是谁？** — "所有人" = 没有用户
2. **痛点在哪？** — "我觉得需要" = 不是真痛点
3. **为何用你？** — "我们有最好的技术" = 用户不在乎

### Step 4: 方案讨论

提出 2-3 种实现方案，让用户做选择题：

| 方案 | 匹配需求点 | 优点 | 缺点 | 适用场景 |
|------|-----------|------|------|----------|

必须给出推荐方案及理由。用户确认方案后进入 PRD 生成。

### Step 5: 生成 PRD

将所有讨论结果整合为 PRD 文档。

**大型系统**：先输出系统分解图（子系统+组件类型+PRD文件名），用户确认后逐个子系统生成 PRD。每个子系统独立走 Phase 2-5。

**单功能**：直接生成 `PRD.md`。

根据组件类型选择对应模板（见下方"PRD 模板"章节）。

**迭代节奏**（按子任务数 × 复杂度判断）：

| 子任务数 | 功能复杂度低 | 功能复杂度中 | 功能复杂度高 |
|---------|------------|------------|------------|
| ≤2 | 一步到定稿 | 一步到定稿 | 初稿→定稿 |
| 3-5 | 一步到定稿 | 初稿→定稿 | 初稿→中稿→定稿 |
| >5 | 初稿→定稿 | 初稿→中稿→定稿 | 初稿→中稿→定稿 |

**复杂度判断**：低 = 纯展示/无交互；中 = 有表单/状态变化/前后端交互/AI 调用；高 = 多角色/权限/支付/复杂状态机/训练流程

**初稿**：需求背景与目标 + 需求范围 + 需求列表 + 方案对比
**中稿**：+ 核心流程图 + 架构/信息架构
**定稿**：+ 细节方案 + 边缘情况 + 非功能需求 + 验收标准

## PRD 写作规范

### 核心原则

1. **问题定义 > 代码实现** — 想得越清楚，AI 写得越准
2. **PRD 是单一事实来源** — 后续所有 Phase 都消费它，不依赖聊天上下文
3. **主动确认 > 被动等待** — 不要等用户补充，AI 主动追问、复述理解、挖掘盲点
4. **具体 > 抽象** — "0.5秒响应"而非"快"，"最多50条"而非"一些"，"Top-3 准确率≥85%"而非"效果不错"
5. **方案先行** — 先输出方案让用户确认，再写 PRD 定稿
6. **迭代式写作** — 初稿想"为什么"，中稿想"是什么"，定稿想"怎么做"

### 为什么每条铁律都重要

> AI 严格按字面理解 PRD，不会"脑补"。人读文档会跳过重复、脑补模糊处；AI 把 PRD 分解成信息块逐个处理，每个字段都影响代码。

**PRD 字段 → 代码的映射**：

| PRD 描述 | AI 生成的代码 |
|----------|-------------|
| "任务有标题、完成状态" | `{ title: string, completed: boolean }` |
| "用户可以添加多个任务" | `tasks: Array<Task>` |
| "点击添加按钮，任务出现在列表" | `handleAddTask()` 函数 |
| "点击勾选，任务显示删除线" | `toggleTask()` + CSS 样式 |
| "快速点击防抖" | `debounce()` 或 `disabled` 状态 |
| "数据需要保存" | localStorage 或数据库 |
| "AI 输出可能包含幻觉" | `validateOutput()` + 降级逻辑 |
| "API 返回值不可变" | `readonly` 或 `deepFreeze()` |
| "矩阵乘法用 float32" | `torch.float32` dtype 指定 |
| "命令支持 --json 输出" | `--json` flag + JSON formatter |
| "管道兼容" | `process.stdin.pipe()` + `process.stdout.write()` |

没写的 = AI 猜测的。猜错 → 返工；猜对 → 运气，不可靠。

## 写作铁律

| 铁律 | 违反后果 | 正确做法 |
|------|---------|---------|
| 没写的 = AI 猜的 | 猜错方向，返工 | 明确写出，不留模糊 |
| 抽象词 = 歧义 | 每人理解不同 | 用具体数值替代 |
| 没 Out-of-Scope | AI 加一堆功能 | In/Out 同等重要 |
| 没边缘情况 | 不做错误处理 | 每个交互写完整状态链/每个函数写边界输入 |
| 没默认值 | AI 按常见模式猜 | 明确数量上限、保存时长、超时阈值、精度要求 |
| 没优先级 | 全部实现，主次不分 | P0/P1/P2 标注 |
| AI 功能没兜底 | 幻觉直接展示给用户 | 每个AI输出必须有验证+降级策略 |
| 框架没 API 契约 | 实现和用法脱节 | 先写 API 签名再写实现 |
| 算法没精度目标 | 实现了但不可用 | 明确精度下限+评估方法 |
| 复现没参考映射 | 实现偏离参考 | 每个模块标注参考来源+对齐程度 |

### PRD 模板

根据组件类型选择对应模板。模板文件位于 `references/prd-templates.md`，Step 5 生成 PRD 时按组件类型读取对应模板。

| 模板 | 组件类型 | 核心差异章节 |
|------|---------|------------|
| 通用 | 所有 | 第 0-1 章（文档信息+需求背景）、第 4-5 章（验收+子任务） |
| A | Web/App 产品 | 交互状态链 + 信息架构 + 数据需求 |
| B | AI 功能 | Prompt 规格 + 输出校验 + 幻觉兜底 + 延迟成本 |
| C | 框架/库 | API 契约 + 扩展点 + 兼容性 |
| D | 算法/模型 | 数学规格 + 输入输出维度 + 数值稳定性 + 精度目标 |
| E | CLI/工具 | 命令树 + 参数规格 + 输出格式 + 退出码 |
| F | API/后端服务 | 端点规格 + 请求响应 + 限流容错 + 部署 |
| G | 数据处理 | 数据管道 + 质量规则 + 幂等性 + 吞吐量 |

## 范式切换检查

| 检查项 | 切换到 |
|--------|--------|
| 子任务 > 5 且跨团队协作？ | → **Spec Coding** 为主 |
| 需求涉及支付/资金/权限？ | → Phase 4 以 **TDD** 为主 |
| 需求高度依赖 UX 体验？ | → **Vibe Coding** 原型验证 |
| AI 功能涉及关键决策？ | → Phase 4 每个子任务必须人工审核 AI 输出 |
| 框架/库项目？ | → Phase 3 必须先完成 API 契约设计 |
| 算法/模型项目？ | → Phase 4 以 **TDD** 为主，先写评估脚本 |
| 复现项目？ | → Phase 4 每个模块完成后与参考实现对比验证 |
