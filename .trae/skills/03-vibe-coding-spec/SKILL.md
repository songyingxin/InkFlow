# Phase 3: 技术方案 & 规格设计（SDD）

**目标**: 确定实现路径，输出可执行的技术规格。**先设计核心契约，再设计功能实现。**

**前置依赖**: Phase 1（01-vibe-coding-prd）已输出 `PRD.md`（或多个 `PRD-*.md`），Phase 2（02-vibe-coding-init）已完成环境初始化。

## 执行流程

```
读取前置产出 → 核心契约设计 → 技术栈确认 → 实现规格设计 → 影响面评估 → 依赖排序 → 实施计划 → 技术决策记录 → 用户确认
```

**核心契约优先**：契约错误时，建在上面的功能越多，修正代价越大。先定义契约，再定义功能。

## 设计原则

| 原则 | 违反后果 | 正确做法 |
|------|---------|---------|
| 契约先于功能 | 功能建在错误的契约上，大面积返工 | 先设计核心契约，再设计功能实现 |
| 规格可执行 | 规格模糊导致实现偏差 | 每个规格字段都有明确的输入输出和行为约束 |
| 依赖显式化 | 隐式依赖导致实现顺序混乱 | 子任务间依赖关系必须可视化 |
| 风险前置 | 高风险项后置导致项目延期 | 风险高的先设计、先实现 |
| 复用可追溯 | 复用来源不明导致规格与实际不符 | 每个子任务标注复用来源（来自 Phase 2） |

## Step 0: 读取前置产出，提取设计参数

### 0.1 从 PRD 提取三维度

```
1. 项目规模: 单功能 / 大型系统
2. 组件类型: Web产品 / AI功能 / 框架库 / 算法模型 / CLI工具 / API服务 / 数据处理
3. 实现模式: 原创 / 复现（附参考来源）
```

**大型系统**：读取所有 `PRD-*.md`，每个子系统可能组件类型不同，需要分别设计规格。

**复现模式**：额外读取 PRD 中的参考映射表，确认对齐策略。

**PRD 字段缺失兜底**：如果 PRD 缺少三维度字段，主动询问用户确认，不猜测。

### 0.2 从 AGENTS.md 读取项目约定

读取 Phase 2 输出的 `AGENTS.md`，确认：
- 技术栈（语言 + 框架 + 关键依赖）
- 开发命令（构建/测试/Lint/类型检查）
- 代码风格约定
- 🔴 红线（禁止修改的模块）
- 组件类型专项约定
- 复现参考信息（如有）

**规格设计必须遵守 AGENTS.md 中的红线和约定，不凭空假设。**

## Step 1: 核心契约设计（最先做）

> 核心契约错误时，已经建在上面的功能越多，修正代价越大。先定义契约，再定义功能——这是避免返工的最有效策略。

**不同组件类型的"核心契约"不同**：

| 组件类型 | 核心契约（最先设计） | 为什么 |
|---------|-------------------|--------|
| **Web/App 产品** | 数据模型 | 数据结构决定页面交互和状态管理 |
| **AI 功能** | Prompt 规格 + 输出 Schema | AI 行为由 Prompt 和输出格式定义 |
| **框架/库** | 公开 API 契约 | API 是框架的唯一交付物，先签合同再施工 |
| **算法/模型** | 数学规格 + 维度推导 | 数学正确是算法的生命线，代码只是数学的表达 |
| **CLI/工具** | 命令接口 + 参数规格 | 命令结构决定用户交互，先定义命令再实现逻辑 |
| **API/后端服务** | 端点契约 + 数据模型 | 端点和数据模型是服务间通信的基础 |
| **数据处理** | 数据 Schema + 管道拓扑 | 数据格式决定转换逻辑，管道拓扑决定执行顺序 |

核心契约的详细模板位于 `references/spec-contracts.md`，按组件类型读取对应模板。

## Step 3: 实现规格设计

针对 `PRD.md` 中每个子任务设计详细实现规格。

### 通用规格字段

每个子任务必须包含：

| 字段 | 说明 |
|------|------|
| 规格描述 | 该做什么，输入输出，行为约束 |
| 改动文件 | 涉及的文件列表 |
| 复用来源 | 直接复用 / 修改参数 / 全新开发（来自 Phase 2 资产扫描） |
| 风险 | 无/低/中/高 + 说明 |
| 测试策略 | 如何验证这个子任务 |

### 按组件类型的规格补充

**Web/App 产品**：
- 交互状态链: [初始→加载→成功→失败→空] 每个状态的 UI 表现
- 数据流: [用户操作 → 状态变更 → UI 更新] 的完整链路
- 页面组件树: [页面 → 组件 → 子组件] 的层级关系

**AI 功能**：
- Prompt 版本: [v1 初始版本，标注变量占位符]
- 上下文构建: [从哪取上下文 → 拼接策略 → Token 预算]
- 输出校验规则: [格式校验 + 内容校验 + 安全校验]
- 降级链路: [校验失败 → 重试1次 → 切换模型 → 默认回复 → 提示用户]
- 成本预算: [单次上限 / 日上限 / 降级触发阈值]

**框架/库**：
- 公开 API 签名: [每个方法的参数、返回值、异常]
- 内部模块依赖: [模块 A → 模块 B → 模块 C]
- 扩展点实现: [钩子触发时机 + 回调签名 + 执行顺序]
- 类型导出: [哪些类型是公开的，哪些是内部的]

**算法/模型**：
- 层实现规格: [输入维度 → 计算逻辑 → 输出维度]
- 前向传播: [数据从输入到输出的完整路径]
- 反向传播: [梯度从输出到输入的传播路径]
- 参数初始化: [权重初始化策略 + 偏置初始化策略]
- 梯度处理: [梯度裁剪 / 梯度累积 / 混合精度]

**CLI/工具**：
- 子命令分发: [主命令 → 子命令 → 处理函数]
- 参数解析逻辑: [必填校验 + 类型转换 + 默认值填充]
- 输出格式化: [表格 / JSON / 着色 / 进度条]
- 管道处理: [stdin 读取 → 处理 → stdout 写出]

**API/后端服务**：
- 端点实现: [请求解析 → 业务逻辑 → 响应构造]
- 中间件链: [认证 → 限流 → 日志 → 业务处理 → 错误捕获]
- 鉴权逻辑: [Token 验证 → 权限检查 → 失败响应]
- 错误处理: [业务错误 → HTTP 状态码 + 错误体]
- 数据库操作: [查询 → 映射 → 分页]

**数据处理**：
- 管道步骤实现: [输入 Schema → 转换逻辑 → 输出 Schema]
- 数据校验逻辑: [Schema 校验 + 业务规则校验 + 异常处理]
- 幂等性保证: [执行标识 + 去重策略 + 重复执行结果]
- 断点续传: [偏移量记录 + 恢复逻辑 + 部分重跑]

### 复现模式规格补充

复现项目的每个子任务额外包含：

| 字段 | 说明 |
|------|------|
| 参考对应 | 对应参考实现的哪个模块/文件 |
| 对齐策略 | 逐行对齐 / 行为等价 / 接口兼容 |
| 已知差异 | 与参考实现的差异及原因 |
| 验证方式 | 如何验证与参考实现对齐（diff / 精度对比 / 行为对比） |

## Step 5: 依赖排序

确定子任务间的依赖关系和执行顺序：

```
[子任务A] → [子任务B] → [子任务C]
                ↓
           [子任务D]（依赖 B 的输出）
```

**排序原则**：
- 核心契约 → 基础功能 → 高级功能
- 被依赖的先做，依赖别人的后做
- 风险高的先做，尽早暴露问题

**按组件类型的典型依赖顺序**：

| 组件类型 | 典型顺序 |
|---------|---------|
| Web/App 产品 | 数据模型 → API 层 → 页面骨架 → 交互逻辑 → 样式优化 |
| AI 功能 | Prompt 规格 → 输出校验 → AI 调用链 → 降级策略 → 成本控制 |
| 框架/库 | API 契约 → 核心逻辑 → 扩展点 → 类型定义 → 文档示例 |
| 算法/模型 | 数学规格 → 基础层 → 模型组装 → 训练循环 → 评估脚本 |
| CLI/工具 | 命令接口 → 参数解析 → 核心逻辑 → 输出格式 → 管道支持 |
| API/后端服务 | 数据模型 → 端点骨架 → 业务逻辑 → 鉴权中间件 → 错误处理 |
| 数据处理 | 数据 Schema → 管道骨架 → 转换逻辑 → 质量校验 → 幂等性 |

## Step 6: 实施计划

将规格转化为细粒度可执行任务清单：

```
1. [步骤] → 涉及文件: [path] → 测试命令: [cmd]
2. [步骤] → 涉及文件: [path] → 测试命令: [cmd]
```

**步骤粒度**：每个步骤应是一个可独立验证的最小单元。Web/API 项目通常 2-5 分钟，算法/模型项目可能需要更长时间（一个层实现、一次训练验证）。

**大型系统**：按子系统分别输出实施计划，标注子系统间的集成点。

## 范式切换检查

根据组件类型和实现模式，检查是否需要调整 Phase 4 的编码范式：

| 检查项 | 切换到 |
|--------|--------|
| 子任务 > 5 且跨团队协作？ | → **Spec Coding** 为主 |
| 需求涉及支付/资金/权限？ | → Phase 4 以 **TDD** 为主 |
| 需求高度依赖 UX 体验？ | → **Vibe Coding** 原型验证 |
| AI 功能涉及关键决策？ | → Phase 4 每个子任务必须人工审核 AI 输出 |
| 框架/库项目？ | → Phase 4 必须先完成 API 契约实现 + 类型测试 |
| 算法/模型项目？ | → Phase 4 以 **TDD** 为主，先写评估脚本再写实现 |
| 复现项目？ | → Phase 4 每个模块完成后与参考实现对比验证 |
| 数据处理项目？ | → Phase 4 重点关注幂等性测试和数据质量校验 |
