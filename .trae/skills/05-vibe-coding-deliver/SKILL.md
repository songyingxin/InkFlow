# Phase 5: 验证交付

**目标**: 确保功能可用，交付完整，质量达标。**不是"再测一遍"，而是从交付视角做最终确认。**最终交付完成后生成 README.md 文件对项目整体做一个介绍。

**前置依赖**: Phase 4（04-vibe-coding-implement）已完成编码和三阶段审查。

**定位**: Phase 5 是 Vibe Coding 流程的终点。它消费所有前序产出——PRD 定义验收标准，task_spec 定义规格，progress.md 记录实现过程，AGENTS.md 定义项目约定。验证交付就是确认"我们做了该做的事，没做不该做的事"。

## 执行流程

```
读取前置产出 → 前置检查 → 全量测试 → 验收标准验证 → 非功能验证 → 复现对齐（如适用）→ 交付门禁 → 交付总结
```

**门禁机制**：每个验证步骤都有明确的通过/不通过判定。不通过时回到 Phase 4 修复，修复后从全量测试重新开始（回归验证），不跳步。

---

## 验证铁律

| 铁律 | 违反后果 | 正确做法 |
|------|---------|---------|
| "看起来没问题" ≠ 没问题 | 隐患上线后爆发 | 每个验收标准必须有可观测证据 |
| 测试通过 ≠ 可交付 | 功能对了但性能/安全不达标 | 测试 + 验收 + 非功能 + 安全四维验证 |
| 修了一个 bug ≠ 没引入新 bug | 修复引发回归 | 修复后必须回归验证，从全量测试重新跑 |
| Phase 4 审查过 ≠ Phase 5 不用查 | 审查时的代码 ≠ 最终代码 | Phase 5 验证的是最终交付物，不是审查记录 |
| 红线不可妥协 | 技术债越积越多 | AGENTS.md 红线违反必须修复，不接受"先上线再说" |
| 超范围 = 做多了 | 增加维护负担和风险 | 对照 PRD Out-of-Scope 检查，多了要删 |

## Step 1: 前置检查

**目的**: 在跑全量测试之前，先确认基本条件满足，避免浪费时间。

### 1.1 完成度检查

- [ ] progress.md 中所有子任务状态为 ✅ 完成？
- [ ] 无 ❌ 阻塞的子任务？
- [ ] 规格偏差（如有）已记录并经用户确认？

**存在未完成子任务时**：暂停交付，回到 Phase 4 完成后再进入 Phase 5。

### 1.2 构建验证

- [ ] 项目可正常构建（`npm run build` / `cargo build` / 等价命令）？
- [ ] 无编译错误、无构建警告？

**构建失败**：回到 Phase 4 修复，不进入测试。

### 1.3 超范围检查

对照 PRD 的 Out-of-Scope 清单，检查是否实现了不该做的功能：

| 检查项 | 结果 |
|--------|------|
| PRD Out-of-Scope 中的功能是否被实现？ | ✅ 无超范围 / ⚠️ 发现超范围功能 |
| 是否有 PRD 未提及的额外功能？ | ✅ 无 / ⚠️ [列出] |

**发现超范围功能**：与用户讨论是否保留或移除。超范围功能增加维护负担和风险，原则上应移除。

## Step 3: 验收标准逐项验证

对照 `PRD.md` 中定义的验收标准逐项验证。**这是交付的核心。**

### 3.1 验证方法

每个验收标准必须用**可观测的方式**验证，不接受"看起来没问题"：

| 验证方式 | 适用场景 | 示例 |
|---------|---------|------|
| 自动化测试 | 功能性标准 | 运行测试命令，查看通过/失败 |
| 手动操作 | 交互性标准 | 按操作步骤执行，截图/录屏记录 |
| 数值对比 | 性能/精度标准 | 运行基准测试，对比数值 |
| 日志/监控 | 可观测性标准 | 检查日志输出、监控指标 |
| 代码审查 | 代码质量标准 | Lint + 类型检查 + 人工审查 |

### 3.2 按组件类型的验收重点

**Web/App 产品**：
- 交互状态链完整性：每个状态（初始/加载/成功/失败/空）都有对应 UI
- 数据流闭环：用户操作 → 状态变更 → UI 更新 → 数据持久化
- 响应式布局：关键页面在不同屏幕尺寸下可用
- 可访问性：键盘导航、屏幕阅读器兼容

**AI 功能**：
- 输出质量：在测试集上的准确率/召回率是否达标
- 降级链路：校验失败 → 重试 → 切换模型 → 默认回复 → 提示用户，每一步都验证
- 成本控制：Token 用量在预算内，降级阈值正确触发
- 延迟：响应时间在可接受范围内

**框架/库**：
- API 契约完整：公开 API 签名与设计一致
- 类型导出正确：用户能正确导入和使用类型
- 扩展点可用：钩子/插件注册和执行正常
- 文档示例可运行：README/examples 中的代码能跑通

**算法/模型**：
- 精度达标：在评估集上的指标满足 PRD 定义的最低标准
- 数值稳定：边界输入不产生 NaN/Inf
- 可复现性：固定随机种子下结果一致
- 维度正确：所有层的输入输出维度符合数学规格

**CLI/工具**：
- 命令接口完整：所有子命令和参数按规格实现
- 退出码正确：成功=0，各种错误有对应退出码
- 管道兼容：stdin/stdout 管道正常工作
- 帮助信息：--help 输出完整准确

**API/后端服务**：
- 端点完整：所有 API 端点按规格实现
- 鉴权覆盖：所有需要鉴权的端点都有保护
- 错误处理统一：错误响应格式一致，状态码正确
- 限流生效：限流策略正确触发

**数据处理**：
- 管道完整：所有管道步骤按规格实现
- 幂等性保证：重复执行结果一致
- 数据质量：脏数据被正确处理，不丢失不损坏
- 断点续传：中断后可恢复，数据完整

| 1 | [标准描述] | [自动化测试/手动操作/数值对比/...] | ✅ 通过 | — |

## Step 4: 非功能需求验证

对照 PRD 中的非功能需求逐项验证。**功能对了但性能/安全不达标，同样不可交付。**

### 4.1 通用非功能检查

| 维度 | 检查项 |
|------|--------|
| 性能 | 关键操作响应时间是否满足 PRD 要求？ |
| 安全 | Phase 4 安全审查的遗留项是否已处理？ |
| 兼容性 | 目标环境/浏览器/运行时是否兼容？ |
| 可维护性 | 代码结构清晰，无死代码和遗留 TODO？ |

### 4.2 按组件类型的非功能重点

| 组件类型 | 非功能重点 |
|---------|-----------|
| **Web/App 产品** | 首屏加载时间、Lighthouse 评分、浏览器兼容性 |
| **AI 功能** | P99 延迟、Token 成本、降级切换时间 |
| **框架/库** | Bundle 体积、Tree-shaking 效果、冷启动时间 |
| **算法/模型** | 推理速度、GPU 显存占用、训练收敛时间 |
| **CLI/工具** | 启动时间、大文件处理内存占用、跨平台兼容 |
| **API/后端服务** | QPS 上限、P99 延迟、内存泄漏检测、长时间运行稳定性 |
| **数据处理** | 吞吐量、内存峰值、断点续传恢复时间 |

### 4.3 AGENTS.md 合规检查

验证交付物是否符合 AGENTS.md 中的项目约定：

- [ ] 代码风格与 AGENTS.md 一致？
- [ ] 🔴 红线未被违反（禁止修改的模块未被改动）？
- [ ] 组件类型专项约定已遵守？
- [ ] 构建命令正常工作？
- [ ] Lint 命令正常工作？

**🔴 红线违反 → 必须回到 Phase 4 修复，不可妥协。**

## Step 5: 复现模式验证（仅复现模式）

复现项目需要额外的参考对齐验证。**原创项目跳过此步骤。**

### 5.1 逐模块对齐验证

对照 PRD 中的参考映射表和 task_spec 中的参考对应，逐模块验证：

| 本项目模块 | 参考实现对应 | 对齐策略 | 对齐结果 | 差异说明 |
|-----------|------------|---------|---------|---------|
| [模块A] | [参考X] | 逐行对齐 | ✅ 对齐 / ⚠️ 差异 | [说明] |
| [模块B] | [参考Y] | 行为等价 | ✅ 对齐 / ⚠️ 差异 | [说明] |

### 5.2 按对齐策略的验证方法

| 对齐策略 | 验证方法 | 通过标准 |
|---------|---------|---------|
| 逐行对齐 | diff 对比输出 | 输出完全一致 |
| 行为等价 | 相同输入对比输出 | 功能行为一致 |
| 接口兼容 | API 签名/命令参数对比 | 接口可互换 |

### 5.3 算法/模型复现额外验证

| 检查项 | 验证方法 |
|--------|---------|

| 收敛曲线 | 对比训练 loss 曲线走势 |
| 推理结果 | 在相同测试集上对比输出分布 |

**对齐不通过**：记录差异，分析原因，与用户讨论是否接受或回到 Phase 4 调整。

## Step 6: 交付门禁

**所有验证步骤完成后，执行交付门禁判定。这是"能不能交付"的最终决策。**

### 6.1 门禁判定矩阵

| 门禁项 | ✅ 通过 | ⚠️ 有条件通过 | ❌ 不通过 |
|--------|--------|--------------|---------|
| 全量测试 | 全部通过 | — | 任一失败 |
| 验收标准 | 全部 ✅ | 无 ❌，有 ⚠️ 但用户已确认可接受 | 存在 ❌ |
| 非功能需求 | 全部达标 | 轻微偏差，用户已确认可接受 | 严重不达标 |
| AGENTS.md 合规 | 红线未违反，风格一致 | 风格轻微偏差 | 红线违反 |
| 复现对齐（如适用） | 全部对齐 | 有差异但用户已确认可接受 | 关键模块未对齐 |
| 超范围检查 | 无超范围功能 | 有超范围但用户确认保留 | — |

### 6.2 门禁结果

| 门禁结果 | 条件 | 动作 |
|---------|------|------|
| 🟢 **可以交付** | 所有门禁项 ✅ | 进入 Step 7 输出交付总结 |
| 🟡 **有条件交付** | 有 ⚠️ 但无 ❌，且用户已确认 | 进入 Step 7，交付总结中标注 ⚠️ 项 |
| 🔴 **不可交付** | 存在 ❌ 或红线违反 | 回到 Phase 4 修复，修复后回归验证 |

### 6.3 回归验证

修复后重新进入 Phase 5 时，**必须从 Step 2（全量测试）重新开始**，不跳步：

```
修复完成 → Step 2 全量测试 → Step 3 验收标准 → Step 4 非功能 → Step 5 复现（如适用）→ Step 6 门禁
```

**为什么不能只验证修复点？** 修复可能引入回归，只有全量验证才能确认交付物整体质量。

## 异常处理

| 异常 | 处理 |
|------|------|
| 前置检查未通过（未完成/构建失败） | 回到 Phase 4 完成/修复，不进入测试 |
| 测试回归 | 回到 Phase 4 定位并修复，修复后回归验证 |
| 验收标准 ❌ 未通过 | 回到 Phase 4 修复，修复后回归验证 |
| 验收标准 ⚠️ 部分通过 | 记录原因，与用户讨论是否可接受 |
| 非功能需求不达标 | 记录差距，与用户讨论是否可接受或回到 Phase 4 优化 |
| AGENTS.md 红线违反 | 🔴 **必须回到 Phase 4 修复**，红线不可妥协 |
| 发现超范围功能 | 与用户讨论保留或移除 |
| 复现对齐不通过 | 记录差异，分析原因，与用户讨论是否接受或回到 Phase 4 调整 |
| 精度不达标 | 检查数学公式、数值精度、参数初始化，与用户讨论调整策略 |
| AI 输出质量差 | 检查 Prompt、上下文、模型选择，调整降级策略 |
| 大型系统集成失败 | 定位故障子系统，回到 Phase 4 修复接口或逻辑 |
