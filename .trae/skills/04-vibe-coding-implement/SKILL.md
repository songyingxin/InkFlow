# Phase 4: 编码实现（TDD 驱动 + 安全审查）

**目标**: 逐个子任务实现代码，TDD 保证质量，安全审查兜底。

**前置依赖**: Phase 3（03-vibe-coding-spec）已输出 `task_spec.md`（或多个 `task_spec-*.md`）。

## 执行流程

```
读取前置产出 → 编码前准备 → [循环] 🔴写失败测试 → 🟢写最少实现 → 🔵重构 → ✅验证 → 安全长检 → 记录进度 → [全部完成] → 三阶段审查
```

**TDD 循环**：每个子任务严格遵循红→绿→重构，不多做不少做。三阶段审查是全部子任务完成后的最终质量关卡。

## Step 0: 读取前置产出

task_spec.md 已包含 PRD 三维度参数，无需再读 PRD：

4. 编码范式: 标准TDD / 契约优先 / 评估先行 / 校验先行 / 行为驱动 / Spec Coding

**大型系统**：读取所有 `task_spec-*.md`，按子系统顺序实现。

**复现模式**：确认每个子任务的参考对应和对齐策略。

**字段缺失兜底**：task_spec 缺少参数时从 AGENTS.md 推断，仍缺失则主动询问。

**编码必须遵守 AGENTS.md 中的红线和约定。**

## 编码范式

根据 task_spec.md 中的范式切换建议，确定本项目的编码范式：

| 范式 | 适用场景 | 核心节奏 |
|------|---------|---------|
| **标准 TDD** | 支付/权限/数据处理 | 🔴 写失败测试 → 🟢 最少实现 → 🔵 重构 |
| **契约优先 TDD** | 框架/库 | 先完成 API 契约 + 类型测试 → 再实现逻辑 |
| **评估先行 TDD** | 算法/模型 | 先写评估脚本 → 再写实现 → 对比精度 |
| **校验先行 TDD** | AI 功能 | 先写输出校验器 → 再写 AI 调用 → 验证降级 |
| **行为驱动 TDD** | CLI/工具 | 先写命令行为测试 → 再实现命令逻辑 |
| **Spec Coding** | 大型系统/跨团队 | 严格按规格实现，不多不少 |

## 核心节奏：红 → 绿 → 重构

每个子任务严格遵循 TDD 循环：

```
🔴 写失败测试 → 🟢 写最少实现 → 🔵 重构优化 → ✅ 验证通过（测试 + Lint + 类型检查）
```

**不同组件类型的"测试先行"含义不同**：

| 组件类型 | 🔴 先写什么测试 | 🟢 先实现什么 | ✅ 额外验证 |
|---------|---------------|-------------|-----------|
| **Web/App 产品** | 组件渲染 + 交互测试 | 数据模型 → 页面骨架 → 交互逻辑 | 视觉回归 |
| **AI 功能** | 输出校验器 + mock 测试 | 校验逻辑 → AI 调用链 → 降级策略 | 降级链路端到端 |
| **框架/库** | 类型测试 + API 契约测试 | API 签名 → 核心逻辑 → 扩展点 | 类型导出完整 |
| **算法/模型** | 评估脚本 + 维度测试 | 数学公式 → 基础层 → 模型组装 | 精度对比 |
| **CLI/工具** | 命令行为测试（subprocess） | 命令分发 → 参数解析 → 核心逻辑 | 管道兼容 |
| **API/后端服务** | 端点契约 + HTTP 测试 | 数据模型 → 端点骨架 → 业务逻辑 | 鉴权覆盖 |
| **数据处理** | Schema 测试 + 幂等性测试 | 数据 Schema → 管道骨架 → 转换逻辑 | 断点续传 |

---

## 编码原则

| 原则 | 违反后果 | 正确做法 |
|------|---------|---------|
| 一个子任务一个节奏 | 多任务并行导致遗漏 | 每次只聚焦一个子任务 |
| 先骨架后细节 | 细节做了一堆但主路径不通 | 先跑通主路径，再补充边界处理 |
| 规格合规 | 多做少做都算偏差 | 严格按 task_spec 实现，不多做不少做 |
| 验证先行 | 声称完成但实际有 bug | 完成前必须有测试 + Lint + 类型检查通过的证据 |
| 代码能跑 ≠ 代码安全 | 安全漏洞上线 | 安全审查不是可选项 |

## 子任务实现流程

### Step 1: 🔴 写失败测试

根据 task_spec 规格和组件类型，编写失败测试。

**按组件类型的测试编写重点**：

**Web/App 产品**：
- 组件渲染测试：给定 props，验证渲染输出
- 交互测试：用户操作 → 状态变更 → UI 更新
- 数据流测试：API 调用 → 数据转换 → 组件更新

**AI 功能**：
- 输出校验测试：给定 AI 输出，验证校验逻辑
- mock 响应测试：mock AI 返回，验证处理链路
- 降级测试：模拟校验失败，验证降级策略
- 成本控制测试：验证 Token 计数和阈值触发

**框架/库**：
- 类型测试：验证导出类型正确（tsd / 类型断言）
- API 契约测试：验证公开 API 签名符合设计
- 扩展点测试：验证钩子/插件注册和执行

**算法/模型**：
- 评估脚本：先写评估指标计算（精度/F1/loss）
- 维度测试：验证输入输出维度符合数学规格
- 数值对比测试：与已知结果对比（小规模数据）

**CLI/工具**：
- 命令行为测试：subprocess 调用，验证 stdout/stderr/退出码
- 参数解析测试：各种参数组合
- 管道测试：stdin 输入 → 处理 → stdout 输出

**API/后端服务**：
- 端点契约测试：HTTP 请求 → 验证状态码 + 响应体
- 鉴权测试：无 Token / 过期 Token / 无权限
- 错误处理测试：各种错误场景的响应格式

**数据处理**：
- Schema 测试：验证输入输出数据格式
- 幂等性测试：同一输入执行两次，结果一致
- 数据质量测试：脏数据 → 验证清洗结果
- 断点续传测试：中断后恢复，验证数据完整

运行测试，确认红灯。

### Step 2: 🟢 写最少实现

写最少代码使测试通过。运行测试，确认绿灯。

### Step 3: 🔵 重构优化

在测试保护下重构。重构后运行测试确认通过。

### Step 4: ✅ 全量验证

运行完整验证套件：

```
测试 → Lint → 类型检查 → 组件类型额外验证（见上方表格）
```

**任一环节不通过则修复后重新验证，不跳过。**

### Step 5: 安全长检

**通用安全检查**（所有类型）：

- [ ] 用户输入是否经过校验和转义？
- [ ] 错误信息是否泄露内部实现？

**按组件类型的安全检查**：

| 组件类型 | 安全检查项 |
|---------|-----------|
| **Web/App 产品** | XSS 防护 / CSRF Token / 敏感数据脱敏 |
| **AI 功能** | Prompt 注入隔离 / AI 输出过滤 / API Key 环境变量 |
| **框架/库** | 依赖漏洞 / 原型污染 |
| **算法/模型** | 数值溢出保护 / 随机种子可控 |
| **CLI/工具** | 路径遍历校验 / 命令注入防护 |
| **API/后端服务** | SQL 参数化 / 鉴权覆盖 / 限流 / 敏感数据加密 |
| **数据处理** | 日志脱敏 / 数据完整性校验 |

### Step 6: 复现模式验证（仅复现模式）

每个子任务完成后，额外执行参考对齐验证：

| 验证方式 | 适用场景 | 操作 |
|---------|---------|------|
| diff 对比 | 逐行对齐 | 对比输出与参考实现的输出 |
| 精度对比 | 算法/模型 | 在相同输入下对比数值差异（atol/rtol） |
| 行为对比 | 行为等价 | 对比功能行为是否一致 |
| 接口兼容 | 接口兼容 | 验证 API 签名/命令参数是否兼容 |

**对齐不通过**：记录差异，分析原因，决定是否调整实现。

### Step 7: 记录进度

将进度追加到 `progress.md`，格式：

```markdown
## 子任务 [N]: [名称] — ✅ 完成 / ⏳ 进行中 / ❌ 阻塞

- TDD: 🔴 [失败测试描述] → 🟢 [实现描述] → 🔵 [重构描述]
- 变更: `[文件路径]`: [做了什么]
- 测试: ✅ [测试名]
- 安全: ✅ 通过 / ⚠️ [说明]
- 复现: ✅ 对齐 / ⚠️ 差异: [说明]（仅复现模式）
```

## 规格偏差处理

实现中发现 task_spec 规格有问题时的处理流程：

| 偏差类型 | 示例 | 处理 |
|---------|------|------|
| **规格遗漏** | 发现未考虑的边界情况 | 补充测试 → 实现 → 记录到 task_spec 偏差日志 |
| **规格错误** | 规格中的接口设计与实际冲突 | 暂停 → 说明冲突 → 提出修正方案 → 用户确认后继续 |
| **规格过度** | 规格要求了不必要的功能 | 暂停 → 说明理由 → 用户确认后跳过 |
| **核心契约变更** | 数据模型/API 签名需要调整 | 🔴 **必须暂停** → 评估影响面 → 用户确认 → 更新 task_spec → 回退受影响的子任务 |

**铁律**：核心契约变更不得自行决定，必须经用户确认。

## 节奏控制

| 复杂度 | 汇报节奏 | 附加控制 |
|--------|---------|---------|
| **低** | 连续 2-3 个后汇报 | — |
| **中** | 每个完成后汇报 | — |
| **高** | 每个完成后汇报 + 展示关键代码 | — |
| **AI 关键决策** | 每个子任务 | 必须人工审核 AI 输出后再继续 |
| **算法层/模块** | 每个层完成后 | 运行维度测试确认通过再继续 |

## 大型系统处理

大型系统包含多个子系统时：

1. 按 task_spec 中的依赖顺序逐个子系统实现
2. 每个子系统按其组件类型的 TDD 节奏实现
3. 子系统间集成点需要额外的集成测试
4. 跨子系统通信（REST / gRPC / 共享数据库 / 消息队列）需端到端验证

**子系统完成标记**：每个子系统全部子任务完成后，输出子系统级汇总。

## 三阶段审查

所有子任务完成后执行：

### 阶段 1 — 规格合规性审查

代码是否完整实现了 task_spec 要求？有没有多做或少做？

**通过标准**：task_spec 中每个子任务的规格描述都有对应实现和测试覆盖。

**按组件类型的合规性重点**：

| 组件类型 | 合规性重点 |
|---------|-----------|
| Web/App 产品 | 交互状态链完整？数据流闭环？ |
| AI 功能 | Prompt 按规格？降级链路覆盖？成本控制生效？ |
| 框架/库 | API 契约完整？类型导出正确？扩展点可用？ |
| 算法/模型 | 数学公式正确？维度匹配？精度达标？ |
| CLI/工具 | 命令接口完整？退出码正确？管道兼容？ |
| API/后端服务 | 端点完整？鉴权覆盖？错误处理统一？ |
| 数据处理 | 管道完整？幂等性保证？数据质量校验？ |

### 阶段 2 — 代码质量审查

**通过标准**：无 Lint 错误，无类型错误，无遗留 TODO/临时代码。

检查项：
- 命名是否清晰一致？
- 结构是否合理（职责单一、层次清晰）？
- 异常处理是否完整？
- 是否有死代码或遗留 TODO？

### 阶段 3 — 安全审查

**通过标准**：按组件类型安全检查清单逐项通过，无已知安全风险。

**复现模式追加**：与参考实现的整体对齐度是否满足要求？

审查不通过则修复后重新审查，循环直到通过。

## 异常处理

| 异常 | 处理 |
|------|------|
| 编码遇阻 | 暂停，说明问题，提出替代方案让用户选择 |
| 测试持续失败 | 系统化调试：先假设 → 后验证，拒绝"猜修复" |
| 发现安全漏洞 | 立即暂停，修复安全漏洞优先级高于功能开发 |
| 规格偏差 | 按规格偏差处理流程执行 |
| 复现对齐失败 | 记录差异，分析原因，与用户讨论是否调整或接受 |
| 精度不达标 | 检查数学公式、数值精度、参数初始化，与用户讨论调整策略 |
| AI 输出质量差 | 检查 Prompt、上下文、模型选择，调整降级策略 |
