# Phase 2: 环境初始化 & 资产扫描

**目标**: 基于 PRD 为 Vibe Coding 建立干净的工程基础。

**前置依赖**: Phase 1（vibe-coding-prd）已输出 `PRD.md`（单功能）或多个 `PRD-*.md`（大型系统）。

自动检测项目状态，走不同分支：
- **空项目** → 脚手架模式（基于 PRD 选型技术栈，不可跳过）
- **已有项目** → 扫描模式（基于 PRD 评估资产复用，小项目可跳过）

## 执行流程

```
读取 PRD → 检测项目状态 → [空项目] 技术栈选型 → 项目初始化 → 输出
                          → [已有项目] 扫描资产 → 复用率评估 → 输出
```

**分支选择**：项目状态决定后续路径，空项目不可跳过初始化，已有项目小项目可跳过扫描。

## 初始化原则

| 原则 | 违反后果 | 正确做法 |
|------|---------|---------|
| 选型对齐 PRD | 技术栈与需求不匹配，后期返工 | 技术选型必须基于 PRD 的组件类型和约束 |
| 约定先于代码 | 没有约定导致风格混乱 | AGENTS.md 在写第一行代码前创建 |
| 能复用不重写 | 重复造轮子浪费时间 | 已有项目先扫描资产，评估复用率 |
| 冒烟测试保底 | 初始化有误但没发现 | 安装完依赖立即跑冒烟测试 |
| 复现模式参考先行 | 参考代码没准备好就开始实现 | 复现项目先配置参考代码环境 |

## Step 0: 读取 PRD，提取初始化参数

从 PRD 文档中提取三个关键维度，决定后续所有初始化策略：

```
1. 项目规模: 单功能 / 大型系统
2. 组件类型: Web产品 / AI功能 / 框架库 / 算法模型 / CLI工具 / API服务 / 数据处理
3. 实现模式: 原创 / 复现（附参考来源）
```

**大型系统**：读取所有 `PRD-*.md`，每个子系统可能组件类型不同，需要分别选型。

**复现模式**：额外记录参考来源（论文/仓库/文档），后续需配置参考代码访问。

**PRD 字段缺失兜底**：如果 PRD 缺少三维度字段，主动询问用户确认，不猜测。

## Step 1: 检测项目状态

扫描项目根目录，判断是否存在以下文件：
- `package.json` / `pom.xml` / `go.mod` / `Cargo.toml` / `requirements.txt` / `pyproject.toml`
- `src/` / `lib/` / `app/` 等源码目录
- `.git/` 目录

**存在任一** → 已有项目模式（跳到 Step 4）
**全部不存在** → 空项目模式（继续 Step 2）

## Step 2: 空项目 — 技术栈选型

基于 PRD 的组件类型推荐技术栈，给出 2-3 个选项及权衡：

### 按组件类型的推荐方向

| 组件类型 | 语言方向 | 框架/工具方向 | 测试框架方向 |
|---------|---------|-------------|------------|
| **Web/App 产品** | TypeScript / Python | React / Vue / Next.js / Nuxt | Vitest + Playwright |
| **AI 功能** | Python / TypeScript | LangChain / transformers / SDK | pytest + 评估脚本 |
| **框架/库** | TypeScript / Python / Rust | Rollup / tsup / Vite lib mode | Vitest + tsd（类型测试）|
| **算法/模型** | Python / C++ / CUDA | PyTorch / JAX / NumPy | pytest + 数值对比 |
| **CLI/工具** | TypeScript / Python / Go | Commander / Click / Cobra | pytest + subprocess / Vitest + execa |
| **API/后端服务** | TypeScript / Python / Go / Java | FastAPI / Express / Gin / Spring | pytest + httpx / Vitest + supertest |
| **数据处理** | Python / SQL | Airflow / dbt / Spark / Polars | pytest + 数据 fixture |

**选型原则**：
- 优先匹配 PRD 中的技术约束（如有）
- 优先选择项目成员熟悉的栈（如有）
- 给出推荐方案及理由，等用户确认后执行

**大型系统**：每个子系统可能需要不同技术栈，逐个子系统确认。如果子系统间语言不同，需额外确认跨语言通信方案。

### 复现模式额外步骤

复现项目需要额外配置参考代码环境：

| 操作 | 说明 |
|------|------|
| 克隆参考仓库 | `git clone` 到 `references/[name]`，加入 `.gitignore` |
| 记录参考版本 | 记录 commit hash / tag，确保可追溯 |
| 配置对比工具 | 如：diff 工具、精度对比脚本、行为对比测试 |
| 读取参考项目结构 | 识别参考项目的目录约定、构建方式、测试方式 |

**参考来源类型处理**：

| 参考类型 | 处理方式 |
|---------|---------|
| 开源仓库 | 克隆到 `references/`，记录版本 |
| 论文（无代码） | 下载 PDF 到 `references/papers/`，从论文提取伪代码/算法描述 |
| 商业产品（无源码） | 记录 API 文档/行为观察，以黑盒方式对齐 |
| 多来源混合 | 每个来源独立管理，在 AGENTS.md 中标注各模块的参考来源 |

**算法/模型复现额外检查**：

| 检查项 | 说明 |
|--------|------|
| GPU/CUDA 环境 | 检测 CUDA 版本、GPU 显存，与参考实现要求对比 |
| 依赖版本锁定 | PyTorch/transformers 等版本必须与参考一致，避免精度差异 |
| 随机种子 | 确认参考实现的随机种子设置方式 |
| 数值精度 | 确认参考使用 float32/float16/bfloat16 |

## Step 3: 空项目 — 项目初始化

### 3.1 执行初始化

- 执行项目初始化命令（npm init / cargo init / go mod init / poetry init 等）
- 安装核心依赖和开发依赖
- 创建目录结构骨架

### 按组件类型的目录结构

**Web/App 产品**：
```
src/
├── components/    # UI 组件
├── pages/         # 页面（或 routes/）
├── hooks/         # 自定义 hooks
├── services/      # API 调用
├── stores/        # 状态管理
└── utils/         # 工具函数
tests/
```

**AI 功能**：
```
src/
├── prompts/       # Prompt 模板
├── chains/        # AI 调用链
├── validators/    # 输出校验
├── fallbacks/     # 降级策略
└── utils/         # 工具函数
tests/
evals/             # AI 评估脚本
```

**框架/库**：
```
src/
├── core/          # 核心逻辑
├── plugins/       # 插件系统
├── types/         # 类型定义
└── utils/         # 内部工具
tests/
examples/          # 使用示例
```

**算法/模型**：
```
src/
├── models/        # 模型定义
├── layers/        # 网络层
├── data/          # 数据加载与预处理
├── losses/        # 损失函数
├── optim/         # 优化器
└── utils/         # 工具函数
tests/
configs/           # 训练/推理配置
scripts/           # 训练/评估脚本
references/        # 参考代码（复现模式，.gitignore）
```

**CLI/工具**：
```
src/
├── commands/      # 子命令
├── utils/         # 内部工具
└── index.ts       # 入口
tests/
```

**API/后端服务**：
```
src/
├── routes/        # 路由/端点
├── services/      # 业务逻辑
├── models/        # 数据模型
├── middleware/     # 中间件
├── config/        # 配置
└── utils/         # 工具函数
tests/
```

**数据处理**：
```
src/
├── pipelines/     # 数据管道
├── transforms/    # 数据转换
├── validators/    # 数据校验
├── sinks/         # 输出目标
└── utils/         # 工具函数
tests/
fixtures/          # 测试数据
```

### 3.2 规范文件创建

创建 `AGENTS.md`，按组件类型记录关键约定：

```markdown
# 项目约定

## 基本信息
- 组件类型: [Web产品 / AI功能 / 框架库 / 算法模型 / CLI工具 / API服务 / 数据处理]
- 实现模式: [原创 / 复现]
- 技术栈: [语言 + 框架 + 关键依赖]

## 开发命令
- 构建: [命令]
- 测试: [命令]
- Lint: [命令]
- 类型检查: [命令]（如适用）
- 格式化: [命令]

## 代码风格
- [约定，如：命名风格、文件组织、导入顺序]

## 🔴 红线
- [禁止修改的模块/文件]
- [必须遵守的约束]

## 组件类型专项约定
[按组件类型补充，见下方]
```

**组件类型专项约定**：

| 组件类型 | 专项约定 |
|---------|---------|
| Web/App 产品 | 路由约定、状态管理规范、组件命名规范 |
| AI 功能 | Prompt 版本管理、模型调用规范、输出校验规范、成本控制策略 |
| 框架/库 | API 设计规范（semver / breaking change）、导出规范、文档规范 |
| 算法/模型 | 数值精度约定、随机种子管理、检查点保存规范、GPU 使用规范 |
| CLI/工具 | 参数命名规范、退出码规范、输出格式规范 |
| API/后端服务 | 端点命名规范、错误码规范、鉴权规范、日志规范 |
| 数据处理 | 数据格式规范、幂等性约定、质量检查规范 |

**复现模式追加**：
```markdown
## 复现参考
- 参考来源: [仓库地址 / 论文 / 文档]
- 参考版本: [commit hash / tag]
- 对齐策略: [逐行对齐 / 行为等价 / 接口兼容]
- 参考代码位置: references/[name]/
```

### 3.3 测试框架配置

- 安装测试框架（按组件类型选型，见 Step 2）
- 创建测试目录
- 编写一个冒烟测试验证框架可用

**组件类型专项测试配置**：

| 组件类型 | 额外测试配置 |
|---------|------------|
| Web/App 产品 | 组件测试 + E2E 测试框架 |
| AI 功能 | AI 输出评估脚本 + mock AI 响应 |
| 框架/库 | 类型测试（tsd）+ API 兼容性测试 |
| 算法/模型 | 数值精度对比测试 + GPU 可用性检测 |
| CLI/工具 | 子进程测试 + stdout/stderr 捕获 |
| API/后端服务 | HTTP 请求测试 + 数据库 fixture |
| 数据处理 | 数据 fixture + 幂等性验证 |

### 3.4 Git 初始化

- `git init`
- 创建 `.gitignore`（含组件类型特定的忽略项）
- **复现模式**：将 `references/` 加入 `.gitignore`
- 初始提交

**按组件类型的 .gitignore 补充**：

| 组件类型 | 额外忽略项 |
|---------|-----------|
| Web/App 产品 | `node_modules/`, `dist/`, `.next/`, `.env.local` |
| AI 功能 | `.env`, `*.cache/`, `models/`（大模型文件） |
| 框架/库 | `dist/`, `node_modules/`, `*.tgz` |
| 算法/模型 | `*.pt`, `*.ckpt`, `*.safetensors`, `__pycache__/`, `.cuda_cache/`, `wandb/`, `references/` |
| CLI/工具 | `node_modules/`, `dist/` |
| API/后端服务 | `node_modules/`, `.env`, `*.log`, `__pycache__/` |
| 数据处理 | `*.log`, `__pycache__/`, `data/raw/`（原始数据） |

### 3.5 环境变量管理

涉及敏感信息或外部服务的项目需创建 `.env.example`：

| 组件类型 | 常见环境变量 |
|---------|------------|
| AI 功能 | `OPENAI_API_KEY`, `MODEL_NAME`, `MAX_TOKENS` |
| API/后端服务 | `DATABASE_URL`, `SECRET_KEY`, `PORT` |
| 数据处理 | `DATA_SOURCE_URL`, `OUTPUT_BUCKET` |

`.env.example` 提交到 Git，`.env` 加入 `.gitignore`。

## Step 4: 已有项目 — 扫描模式

> 小项目可跳过此模式，直接进入 Phase 3（vibe-coding-spec）。

### 4.1 项目上下文探索

扫描项目根目录结构，识别技术栈：
- 读取 `package.json` / `pom.xml` / `go.mod` / `Cargo.toml` / `requirements.txt` 等
- 识别框架、语言、关键依赖版本
- 查看目录结构约定
- 读取已有 `AGENTS.md`（如有）

### 4.2 代码规范文件

检查/更新 `AGENTS.md`，确保包含 Step 3.2 中的所有字段。已有项目需补充组件类型专项约定。

### 4.3 基于 PRD 的资产扫描

读取 `PRD.md` 中的子任务拆解，逐个评估已有代码的复用情况：

| PRD 子任务 | 已有 Reference | 复用方式 | 改动量 |
|-----------|---------------|---------|--------|
| [子任务A] | [已有服务名] | 直接复用 | 无 |
| [子任务B] | [已有服务名] | 修改参数 | 低 |
| [子任务C] | 无 | 全新开发 | 高 |

复用方式定义：
- **直接复用**: 无需修改，直接调用
- **修改参数**: 改配置/参数即可适配
- **全新开发**: 无可复用资产

**大型系统**：按子系统分别扫描，每个 `PRD-*.md` 独立评估。

**复现模式**：额外对比参考实现与已有代码的差异：

| 本项目模块 | 参考实现 | 已有代码 | 差异 | 对齐工作量 |
|-----------|---------|---------|------|-----------|
| [模块A] | [参考X] | [已有Y] | [差异] | [高/中/低] |

### 4.4 测试基线验证

运行现有测试，确认基线是干净的。

## 复用率评估 & 范式切换（仅已有项目模式）

计算复用率 = (直接复用 + 修改参数) / 总子任务数

| 复用率 | 建议 |
|--------|------|
| **> 60%** | 切换到 **Glue Coding** 范式为主 — 重点在胶水代码和集成测试 |
| **30%-60%** | 混合模式 — 复用部分 Glue Coding，新开发部分 Vibe Coding |
| **< 30%** | 继续标准 **Vibe Coding** 流程 |
