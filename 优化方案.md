# InkFlow 优化方案

基于对 OpenClaw、Hermes Agent、Claude Code 三个标杆系统的架构分析，结合 InkFlow 现有代码的审查结果，经讨论筛选后仅保留 1 项必要优化。

---

## 一、架构总览对比

| 维度 | OpenClaw | Hermes Agent | Claude Code | InkFlow 现状 |
|------|----------|-------------|-------------|-------------|
| 核心循环 | Heartbeat 定时驱动 + Agentic Loop | 单线程 ReAct 循环 | 单线程 while-loop (nO) | LangGraph StateGraph + Supervisor |
| 记忆架构 | 5文件 (SOUL/HEARTBEAT/MEMORY/active-context/daily) | 3层 (MEMORY.md/Skills/Session FTS5) | 5层压缩管线 + Markdown 长期记忆 | 3层 (chat.db/short_memory/MEMORY.md) + FTS5 |
| 工具系统 | Skill 技能系统 + USE/DON'T USE 守卫 | 自注册 + toolset 分组 | 7种权限模式 + ML分类器 | 自注册 + toolset 分组 + dispatch 增强 |
| 子Agent | 无（单Agent + 多通道） | 子Agent委派 + cron调度 | 单层SubAgent + 并行执行 | Lead→Subagent Handoff + Plan-Execute |
| 安全机制 | Prompt注入扫描 + 文件锁定 | 注入扫描 + 原子写入 + 文件锁 | 7级权限 + ML分类器 | 错误分类 + 增强提示 |
| 上下文管理 | 语义快照 + 增量索引 | 冻结快照(prefix cache) + nudge | 5层压缩管线(92%触发) | 分级压缩 + LLM摘要 |

## 二、优化项详细方案

### generate_* 工具返回摘要而非全文

**现状问题：**
- `generate_*` 类工具（续写章节、生成设定等）将全文通过 ToolResult 返回
- 全文进入 messages，单次生成可能 3000-8000 字，占 4-8% 上下文窗口
- 多次生成后上下文迅速膨胀，触发 LLM 压缩（额外 token 消耗）
- 全文已写入磁盘文件，上下文中重复存储纯属浪费

**参考：** Claude Code 的 SubAgent 摘要返回机制 — SubAgent 完成后只返回压缩摘要，不返回完整中间过程。全文已在文件中，需要时用 `read_novel_content` 精确读取。

**优化方案：**
1. `generate_*` 工具 handler 中，全文写入磁盘后，ToolResult 只返回摘要：
   ```
   "已生成第3章「暗夜追踪」，共2800字。核心情节：主角发现密道入口，与守卫发生冲突，获得关键线索。"
   ```
2. 摘要由 COMPRESSION_MODEL 生成（成本低），或从生成结果中提取首尾 + 关键句
3. `read_novel_content` 保持不变 — Agent 主动要读时必须返回完整内容

**预期收益：** 单次生成节省 3000-8000 token 上下文占用，减少 LLM 压缩触发频率

## 三、已排除的优化项及理由

| 原优化项 | 排除理由 |
|----------|---------|
| 冻结快照模式 | `_STABLE_PREFIX_CACHE` + SHA-256 hash 缓存已实现；MEMORY.md session 内不变（只写 short_memory.md），缓存天然命中，无需额外冻结 |
| memory_append 写入守卫 | `rewrite_memory_md()` 已在做记忆清洗，每次 rewrite 时 LLM 会过滤噪声；正则守卫容易误杀，改 schema description 效果也有限 |
| Subagent 并行执行 | 小说写作天然串行（先读再写、先大纲再角色），`depends_on` 虽已定义但实际场景几乎无独立并行步骤；并行还引入写冲突和状态同步问题 |
| 5层压缩管线 | 现有 2 层压缩 + 生成返回摘要已覆盖绝大多数场景；5 层管线是为 24/7 连续运行的 coding agent 设计的，InkFlow 对话轮次远未到需要渐进式压缩的程度 |
| Prompt 注入防御 | InkFlow 是本地单用户工具，用户只能害自己，威胁模型不成立；这是多用户/托管场景才需要的安全措施 |
| 原子写入与文件锁 | InkFlow 是单用户本地应用，同一时间只有一个进程写文件，并发写入不会发生 |
| Skill 自我进化 | 当前只有 3 个手写 Skill，手动系统尚未成熟，过早引入自动生成是过度优化 |
| Heartbeat 主动巡检 | OpenClaw 需要 Heartbeat 是因为它监控邮件/消息/CI 等外部事件；InkFlow 是小说写作工具，用户打开才用，没有"后台需要持续关注"的场景 |
| 实时转向 | 需要改 API/SSE/前端三端，实现成本高；小说写作中"走偏"代价低，等几秒生成完再纠正即可 |
| 记忆健康 API | 没有前端消费这个数据，加个没人调的 API 端点是死代码；等有仪表盘需求时再加，3 行代码的事 |
