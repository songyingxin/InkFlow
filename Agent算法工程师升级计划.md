## 三、升级计划

### Phase 1：InkFlow评估体系（第1-2周，P0最高优先级）

**目标**：用数据证明InkFlow的记忆系统和评估器是有效的

**为什么这是P0**：面试官不关心你"做了什么"，关心你"证明了什么"。没有量化数据，InkFlow就是一个"个人项目"；有了量化数据，它就是一个"有验证的Agent系统设计"。

#### 1.1 设计评测指标

| 指标 | 定义 | 测量方法 | 目标值 |
|------|------|---------|--------|
| **任务完成率** | Agent是否成功执行了用户请求 | 工具调用成功 / 总请求 | ≥85% |
| **上下文一致性** | 10轮+对话中，Agent是否遵守早期设定 | 设定关键词出现率（角色名/地名/设定规则） | ≥90% |
| **记忆召回准确率** | FTS5检索结果是否包含回答所需信息 | 人工标注检索结果相关性 | ≥80% |
| **评估器准确率** | 两级评估器的判断是否与人工一致 | 评估器结果 vs 人工标注 | 规则引擎≥95%，LLM≥85% |
| **压缩信息保留率** | 压缩后关键信息是否保留 | 压缩前后的关键实体覆盖率 | ≥85% |
| **Token效率** | 每次交互的平均token消耗 | 统计input/output token | 记录基线，后续优化对比 |

#### 1.2 搭建评测Pipeline

```
评测流程：
  准备测试集 → 批量运行 → 收集SSE事件 → 自动评分 → 生成报告

测试集设计（20个场景，覆盖核心功能）：
  ├─ 初始化类（3个）：创建新书、生成设定、生成大纲
  ├─ 生成类（5个）：生成章节、续写章节、生成角色、生成伏笔、生成关系
  ├─ 修改类（4个）：修改设定、修改大纲、修改角色、修改伏笔
  ├─ 查询类（3个）：查询角色信息、查询伏笔状态、查询大纲进度
  ├─ 长对话类（3个）：10轮+连续创作、跨session记忆保持、设定冲突检测
  └─ 异常类（2个）：模糊指令处理、工具调用失败恢复

实现方式：
  - 基于现有 run_e2e.py 扩展，增加自动评分逻辑
  - 每个场景定义：输入消息 + 期望事件类型 + 期望工具调用 + 评分规则
  - 输出：评测报告（每个场景的PASS/FAIL + 指标数值）
```

#### 1.3 对比实验（核心交付物）

| 实验 | 控制组 | 实验组 | 预期结论 |
|------|--------|--------|---------|
| **记忆系统有效性** | 关闭FTS5检索+short_memory（只用MEMORY.md） | 完整记忆系统 | 上下文一致性提升15%+ |
| **多Agent vs 单Agent** | 单Agent直接处理所有任务 | Supervisor→Subagent | 复杂任务完成率提升20%+ |
| **评估器必要性** | 无评估器（固定迭代次数） | 两级评估器 | 无效迭代减少50%+ |
| **压缩策略** | 无压缩（直接截断） | Hermes分级压缩 | 信息保留率提升30%+ |

**产出**：一份评测报告，包含4组对比实验的数据和结论。这是面试中最有力的武器。

### Phase 3：补Agent核心知识（第2-3周，与Phase 1-2并行）

#### 3.1 多Agent通信与编排（P1）

**当前状态**：InkFlow是单层Supervisor→Subagent，Subagent之间不直接通信

**需要补的能力**：

| 模式 | 说明 | 适用场景 | 学习方式 |
|------|------|---------|---------|
| **Supervisor** | Lead Agent统一调度 | 当前InkFlow已实现 | 已掌握 |
| **层级委托** | Subagent可以再委托子任务 | 复杂任务的递归分解 | 理解原理，能讲清楚利弊 |
| **并行执行** | 多个Agent同时处理独立子任务 | 同时生成角色+伏笔 | 在InkFlow中实现 |
| **协商/辩论** | 多Agent对同一问题给出不同方案，投票/LLM Judge裁决 | 质量敏感的决策 | 理解原理，能讲清楚 |
| **共享黑板** | Agent通过共享状态文件通信 | 需要实时同步的场景 | 理解原理 |

**面试必答**：
- "为什么选Supervisor而不是扁平架构？" → Supervisor模式让Lead Agent掌握全局上下文，避免Subagent间的信息孤岛。扁平架构在简单场景更灵活，但在长文本场景下会导致上下文碎片化。
- "为什么不让Subagent之间直接通信？" → 单层层级设计是有意为之。Subagent的上下文窗口已经很大（小说记忆+对话历史），如果再引入Agent间通信，上下文管理复杂度指数级增长。Lead Agent作为唯一的信息枢纽，保证了一致性。

**必答题（每个准备3分钟版本和1分钟版本）**：

**Q1: "你为什么从训练转向Agent？"**

1分钟版：不是转，是升级。基座越强，越需要能把强基座变成可靠Agent系统的人。我的训练经验让我比纯应用层工程师更理解模型的能力边界——比如在金山做GRPO时，我发现RL在客观题上几乎没提升（79.6→79.9），但在主观题上效果显著（48.94→62.19），这说明模型的能力边界决定了哪些问题适合用Agent解决，哪些需要从模型侧优化。

3分钟版：展开讲InkFlow中"模型理解如何指导Agent设计"的具体例子——比如我设计两级评估器，是因为我知道模型在判断"任务是否完成"时经常出错（SFT数据中这类信号很弱），所以用规则引擎兜底；比如我设计分层记忆，是因为我知道128K上下文窗口在10轮+对话中一定会溢出，所以必须把信息分层管理。

**Q2: "你的Agent系统和别人搭的有什么区别？"**

1分钟版：两个区别。第一，我有双子系统记忆设计——对话记忆管理交互历史，小说记忆管理结构化知识，通过FTS5检索按需召回，这不是框架能开箱即用的功能。第二，我有量化评估来证明它的有效性——记忆系统使上下文一致性提升XX%，评估器减少了XX%的无效迭代。

3分钟版：展开讲记忆系统的设计取舍——为什么用FTS5而不是向量数据库（小说场景的检索是关键词精确匹配为主，语义检索反而引入噪声），为什么MEMORY.md冻结而short_memory.md可写（长期记忆的稳定性vs短期记忆的灵活性），为什么6个字段文件分开存储而不是合并（不同字段的更新频率和检索需求不同）。

**Q3: "Agent系统最大的挑战是什么？你怎么解决？"**

1分钟版：可靠性，三个层面。模型输出不稳定——用结构化输出+重试+降级；上下文丢失——用分层记忆+FTS5检索；成本失控——用模型路由+缓存+压缩。每个方案我都有量化数据支撑。

3分钟版：展开讲"上下文丢失"这个最难的挑战——在长篇小说场景下，10轮对话后模型会"忘记"早期的设定（角色名写错、设定矛盾），这不是prompt能解决的，必须从记忆架构层面设计。我的方案是：MEMORY.md作为跨session的稳定锚点（冻结快照），short_memory.md作为session内的动态缓冲，FTS5检索按需召回相关记忆。评测数据显示这个方案使上下文一致性从XX%提升到XX%。

**Q4: "你怎么评估Agent系统的效果？"**

1分钟版：三层评估。任务完成率——规则引擎快速判断；行为质量——LLM-as-Judge评估输出合理性；系统效率——trace数据统计token消耗和延迟。我在InkFlow中实现了前两层，规则引擎覆盖95%场景，LLM评估器处理模糊情况。

3分钟版：展开讲评估器的设计——quick_evaluate的5条规则（写入工具成功→完成、写入工具失败→未完成、无工具有回复→完成、无工具无回复→未完成、task_complete无写入→模糊交LLM），以及为什么这样设计（覆盖率高、延迟低、成本低）。再讲LLM评估器的prompt设计——不是简单问"完成了吗"，而是给评估器提供任务描述、工具调用结果、Agent回复，让它判断是否满足用户需求。

**Q5: "你对Agent的未来怎么看？"**

1分钟版：短期——从demo到生产的工程化是核心挑战，评估和可观测性是关键。中期——Agent能力训练（Tool Use/规划微调）会成为新需求，让模型原生具备更好的Agent行为。长期——Agent会基础设施化，值钱的是在特定领域把Agent做到极致的人。

3分钟版：结合自己的经验展开——我在金山做GRPO时发现，RL在主观题上效果显著但在客观题上几乎没提升，这说明模型的能力边界决定了Agent系统的设计空间。未来基座模型的Agent能力会越来越强（更好的Tool Use、更长的推理链），但"如何让Agent在特定场景下可靠运行"这个问题不会消失，反而会因为应用场景的复杂化而变得更难。所以我的判断是：**基座越强，越需要能把强基座变成可靠Agent系统的人。**

**Q6: "你在Agent方面有什么不足？打算怎么补？"**

诚实回答：我的Agent工程化经验还不够——InkFlow是个人项目，没有经历过大规模用户的生产环境验证。我正在通过三个方向补：1）搭建评估体系和可观测性，让系统的行为可量化；2）学习Agent评测框架（AgentBench/SWE-bench），理解标准化评测方法；3）做Tool Use微调实验，从训练角度优化Agent行为。
