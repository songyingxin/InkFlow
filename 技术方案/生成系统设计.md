## 1. 字段生成流程

**源码**：`agent/tools/generate.py`

```
handle_generate_field(field, label, user_request, reread_all)
  → 确定现有内容 existing
  → 判断 reread_all（None → 自动判断，可能 interrupt 询问用户）
  → 根据字段类型选择流:
    ├── outline_historical → _build_historical_stream()
    │   └── iterative_generate_stream(read_ch_field="outline_historical_read_ch")
    ├── outline_future → _build_future_stream()
    │   └── future_outline_stream()  # 不走 iterative
    └── 其他字段 → generate_field_stream()
        └── iterative_generate_stream(read_ch_field=对应字段)
  → 流式收集 full_content
  → compute_diff_highlights(existing, full_content) → 计算变更区间
  → 发送 field_content 事件（含 content + highlights）
  → MemoryManager.save_field_content() → 保存
  → state.field_values[field] = full_content → 同步前端
```

**reread_all 判断逻辑**：
- `reread_all=True`：重新加载所有章节索引，全量重新生成
- `reread_all=False`：仅处理新增章节，增量生成
- `reread_all=None`：自动判断，当已有内容较长时可能 interrupt 询问用户

## 2. iterative_generate_stream 核心逻辑

**源码**：`agent/generation/base.py`

```
iterative_generate_stream(state, read_ch_field, existing, template_name, ...)
  ├── 计算 new_chapter_indices（未读章节）
  ├── reread_all=True → 重新加载所有章节索引
  ├── 无章节 + 无内容 → "从零生成" 模式
  ├── 无章节 + 有内容 → "暂无新增章节" + user_request
  └── 有未读章节 → 分批处理:
      ├── BATCH_SIZE=5
      ├── 每批: system_msg(template) + user_msg(章节内容+请求)
      ├── 多批间 yield _RESET 信号（通知调用方清空）
      └── current = batch_output（上一批输出作为下一批的"当前内容"）
```

**关键设计**：
- 模板通过 `load_template(template_name).format(**fmt_kwargs)` 渲染
- `current_content` 变量注入模板，让 LLM 看到已有内容
- `cross_deps` 自动加载：生成 characters 时自动注入 settings 内容
- `_RESET` 信号：多批次生成时通知调用方清空已有内容，前端需做平滑处理

## 3. 未来大纲特殊处理

**源码**：`agent/generation/base.py:future_outline_stream()`

不走 `iterative_generate_stream`（因为 `read_ch_field=None`，未来大纲不基于已读章节增量更新）。

一次性传入 5 个上下文：historical_outline + settings + characters + relationships + foreshadowing

`_build_chapter_context()` 构建章节索引（含已写/规划中状态）。

## 4. 章节生成

**源码**：`agent/tools/chapter.py`

### 4.1 续写（continue_writing）

```
handle_continue_writing(chapter_num, writing_instruction)
  → _resolve_chapter_target() → 确定章节号和标题
  → generate_chapter_title() → 生成标题（如大纲无标题）
  → _stream_chapter_content() → 流式生成正文
  → _self_review_chapter() → 自检（跑题/OOC）
  → MemoryManager.save_chapter() → 保存
  → _update_outline_after_write() → 更新 outline_structure.json
```

> 字段过期检测与 `update_stale_fields` 已移除，待记忆系统重设计。

### 4.2 重写（regenerate_chapter）

验证章节号 → 流式重新生成 → 自检 → 保存。

局部重写逻辑（关键词匹配 + MARK 分割）当前不可用，建议删除，局部修改走 update_field(patches)。

### 4.3 标题生成

`chapter_title_generate()` — 基于前 10 章标题风格 + 未来大纲。

### 4.4 正文生成

`chapter_content_stream()` — 注入设定/大纲/角色/伏笔/近5章正文。

### 4.5 自检机制（`_self_review_chapter()`）

- 使用 COMPRESSION_MODEL 检查两个维度：跑题（对比大纲规划）、OOC（对比角色设定）
- 检测到问题时自动重修，重修后不再二次自检

### 4.6 后处理（`_update_outline_after_write()`）

章节保存后更新 outline_structure.json 和 meta.total_chapters。

计算章节内容的 SHA-256 hash 并存入 ChapterOutline.content_hash，用于后续检测章节重写。

## 5. 增量更新流程

**源码**：`agent/tools/generate.py`

```
handle_update_outline_historical/future(state)
  → get_unread_chapter_indices("outline_historical_read_ch")
  → 加载未读章节摘要（每章截取前800字）
  → 构建 user_request（含章节概要）
  → update_field_stream(novel_state, field, existing, user_request)
    → load_template("update_field").format(label, memory_section, format_hint)
    → 注入 cross_deps 上下文
    → LLM 流式生成
  → apply_search_replace() 合并 → 返回 highlights
  → 合并失败 → compute_diff_highlights(existing, full_content) 计算 highlights
  → 发送 field_content 事件（含 content + highlights）
  → save_field_content() → 保存
```

**update_outline**：协调器，依次调用 historical + future。

**update_outline_future**：额外注入历史大纲内容作为上下文。

**未读章节判断**：`get_unread_chapter_indices(novel_state, "outline_historical_read_ch")` — historical 和 future 共用同一个 read_ch 字段。

## 6. 新书初始化

**源码**：`agent/tools/init.py`

```
handle_init_novel(title, genre, premise)
  → 检查已有文件 → interrupt 确认覆盖
  → MemoryManager.initialize_project_files() → 创建空模板文件
  → 依次生成 4 个字段:
    1. settings_md_content → generate_field_stream()
    2. characters_md_content → generate_field_stream()
    3. relationships_md_content → generate_field_stream()
    4. outline_future_md_content → future_outline_stream()  # 特殊处理
  → 保存 meta.title
```

**已知问题**：生成完未来大纲后不更新 `meta.total_chapters`，导致 `_build_chapter_context()` 返回错误数字。

## 8. 优化建议

> 与 [Agent整体设计 §8](Agent整体设计.md) 的汇总索引对应，此处为生成系统专项详细分析。

### P0 — 必须修复

1. **init_novel 不更新 total_chapters**：生成完未来大纲后 `meta.total_chapters` 仍为 0，导致 `_build_chapter_context()` 返回错误数字。建议：在 init_novel 末尾从 outline_future 解析章节数并更新 meta.total_chapters，或改为计算属性（详见 [Agent整体设计](Agent整体设计.md) §7.4）。
2. ~~**regenerate_chapter 不触发字段过期检测**~~：过期机制已整体移除，将在记忆系统 v2 重设计。

### P1 — 重要改进

3. **未来大纲上下文过大**：同时传 5 个文件（历史大纲+设定+角色+关系+伏笔）→ 可能超过 10 万 token。建议：按字段大小裁剪，超过阈值（如 3000 字）的字段截断为摘要。
4. **BATCH_SIZE 硬编码为 5**：当章节内容很长时，5 章可能超出上下文窗口。建议：根据 token 估算动态调整 batch_size。
5. **continue_writing 的 stale fields 询问打断流程**：用户说"写下一章"却被 interrupt 问"是否先更新设定"。建议：只在 stale 章节数 > 5 时才询问，少量过期不影响写作质量。

### P2 — 改善体验

6. **_RESET 信号前端闪烁**：多批次生成时，`_RESET` 信号通知调用方清空已有内容，前端可能看到内容闪烁。建议：前端对 generate_reset 事件做平滑处理（如淡入淡出）（详见 [前端交互设计](前端交互设计.md) §3）。
7. **章节生成上下文固定**：`_build_chapter_context()` 固定加载近 5 章正文，但长篇小说可能需要更远的上下文。建议：支持用户指定参考章节范围。
8. **update_field_stream 无格式约束**：LLM 生成 SEARCH/REPLACE 时可能偏离 format_hint。建议：在 update_field 模板中更强调输出格式，或增加后处理校验。
