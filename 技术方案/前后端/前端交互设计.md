## 1. SSE 事件流

| 事件类型 | 来源 | 前端处理 |
|---------|------|---------|
| token | Lead Agent 流式 | 追加到 streamingContent |

| subagent_token | Subagent 流式 | 追加到 streamingContent |
| subagent_tool_call | Subagent 工具调用 | 显示工具名 |
| subagent_tool_error | Subagent 工具失败 | 显示错误 |
| generate_start | 生成工具 | 打开对应编辑器 |

| chapter_title | 标题生成 | 设置 chapterTitle |

| handoff | 路由切换 | 显示"正在切换执行器" |

| session_end | 会话结束（用户关闭/超时） | 清空对话状态 |

### 1.1 field_content 事件与变更高亮

`field_content` 事件在字段生成或修改完成时发送，携带 `highlights` 字段标记变更区间：

```json
{
  "type": "field_content",
  "target": "settings_md_content",
  "content": "完整字段内容...",
  "highlights": [[0, 50], [200, 350]]
}
```

**highlights 格式**：`[number, number][]`，每个元素为 `[start, end]` 字符偏移区间，基于 `content` 字符串的索引。

**前端处理流程**：
1. `fieldValues[target] = content` — 更新字段内容
2. `fieldHighlights[target] = highlights` — 存储高亮区间（如无 highlights 则删除）
3. `editingField = target` — 切换到对应编辑器

**高亮渲染方案（Gutter 彩色条风格）**：

参考 Trae / VS Code 的 inline diff gutter 风格，编辑模式通过左侧彩色 gutter 条标记变更行，预览模式用淡色背景 + 左侧竖线标记变更区间。编辑区本身无背景色变化，保持干净的编辑体验。

| 模式 | 渲染方式 | 视觉效果 |
|------|---------|---------|
| 预览模式 | `renderedMarkdown` 计算属性在 Markdown 文本中插入 `<mark class="diff-new">` 标签，再经 `marked.parse()` 渲染 | 淡绿底色 + 2px 左侧竖线 |
| 编辑模式 | 左侧 4px gutter 区域，变更行显示绿色竖条 | 绿色 gutter 彩色条 |

**CSS 变量**（定义在 `global.css`）：
```css
--diff-new-bg: #edf3ec;           /* 淡绿底色（预览模式） */
--diff-new-bar: rgba(52, 101, 56, 0.55);  /* 绿色竖条 */
--diff-mod-bg: #e1f3fe;           /* 淡蓝底色（修改场景） */
--diff-mod-bar: rgba(31, 108, 159, 0.55);  /* 蓝色竖条 */
--diff-gutter-w: 4px;            /* gutter 宽度 */
```

**预览模式样式**：
```css
mark.diff-new {
  background: var(--diff-new-bg);
  border-left: 2px solid var(--diff-new-bar);
  padding: 1px 2px 1px 6px;
  border-radius: 0 2px 2px 0;
  animation: diff-reveal 500ms cubic-bezier(0.32, 0.72, 0, 1) both;
}
```

**编辑模式 Gutter 彩色条**：
- 左侧 4px 宽 gutter 区域，变更行显示绿色竖条
- 竖条使用 `var(--diff-new-bar)` 颜色，圆角 `0 1px 1px 0`
- textarea 保持正常背景，不做叠层处理

**Stagger 入场动画**：每个 highlight 区间带递增的 `animation-delay`（每段间隔 80ms，上限 800ms），高亮区域依次从下方渐入（`translateY(4px)` → `translateY(0)`），使用 spring 曲线。

**状态管理**：`editorStore.fieldHighlights` — `Record<string, [number, number][]>`，按字段名存储高亮区间。

## 2. 编辑器保存流程

```
用户点【保存】→ @save → saveFieldEdit()
  → editorStore.saveFieldEdit(field, content)
    → POST /api/fields/update
      → MemoryManager.save_field_content()
    → editorStore.fieldValues[field] = content
    → editorStore.resetDirty()
```

### 保存函数路由

| editingField | 调用的保存函数 |
|-------------|-------------|
| title | saveTitle() → saveFieldEdit('title', titleValue) |
| chapter_* | saveChapterEdit() → updateChapter(idx, title, content) |
| chapter_new | saveNewChapter() → addChapter(title, content) |
| 其他字段 | saveFieldEdit() → updateField(field, content) |

## 3. 设计规范与 Skill 体系

InkFlow 前端设计遵循 `.trae/skills/` 下的独立 skill 规范，确保设计决策有据可循、风格统一。

### 3.1 Skill 目录结构

```
.trae/skills/
├── frontend-design/          — 前端界面设计总纲
│   └── SKILL.md              — 有意为之的美学方向、避免 AI 通用审美、typography/color/animation
└── taste-skill/              — 高标准前端品味体系
    └── skills/
        ├── taste-skill/      — 基线配置 + 组件架构 + CSS 硬件加速
        ├── minimalist-skill/ — 极简编辑风格：暖调单色、排版对比、desaturated pastel
        ├── redesign-skill/   — 重新设计方法论：先审计再重构
        ├── brandkit/         — 品牌色彩系统
        ├── soft-skill/       — 柔和有机风格
        ├── brutalist-skill/  — 粗野主义风格
        └── ...               — 其他子 skill
```

### 3.2 当前项目的设计定位

InkFlow 采用 **minimalist-skill** 的极简编辑风格，结合 **taste-skill** 的基线配置：

| 维度 | 配置值 | 体现 |
|------|--------|------|
| DESIGN_VARIANCE | 8 | 墨韵体系非通用 SaaS 风格，有强烈的文化辨识度 |
| MOTION_INTENSITY | 6 | 变更高亮的 stagger 入场动画、编辑器 reveal-up 过渡 |
| VISUAL_DENSITY | 4 | 宽松行高（2.1）、大段间距、留白充裕 |

### 3.3 设计决策与 Skill 规则的对应

| 设计决策 | 遵循的 Skill 规则 | 具体体现 |
|---------|------------------|---------|
| 色彩体系：墨金 `#8b6914` + 宣纸 `#f7f4ee` | minimalist-skill：暖调单色系、desaturated pastel 强调色 | `--accent: #8b6914`、`--bg-root: #f7f4ee`、`--rose: #c45a5a` |
| 变更高亮：朱批风格 | minimalist-skill：色彩克制、用结构传达信息；frontend-design：有意为之的美学方向 | `rgba(196,90,90,0.18)` 底色 + `3px` 左侧竖线，非通用绿色底色块 |
| 字体：Noto Serif SC + JetBrains Mono | minimalist-skill：禁止 Inter/Roboto/Open Sans；frontend-design：distinctive font choices | 正文衬线体 + 代码等宽体，编辑器/预览统一 |
| 阴影：极淡 `--shadow-sm` | minimalist-skill：阴影 opacity < 0.05、禁止 `shadow-md/lg/xl` | 卡片 `box-shadow: var(--shadow-sm)` |
| 动画：`--spring-*` 缓动 | taste-skill：MOTION_INTENSITY=6、CSS 硬件加速 | `reveal-up 400ms var(--spring-slow)`、`diff-reveal 400ms var(--spring-medium)` |
| 禁止 emoji | taste-skill：ANTI-EMOJI POLICY | 工具栏用文字符号（H1/B/I/—）而非 emoji |
| 禁止渐变 | minimalist-skill：DO NOT use gradients | 高亮用纯色 `rgba()` 而非 `linear-gradient` |

### 3.4 变更高亮的 Skill 推导过程

高亮方案的设计推导路径：

```
需求：所有变更高亮，编辑/预览都可见
  │
  ├── frontend-design：选择有意为之的美学方向
  │   → 不能用通用绿色底色块（AI slop aesthetic）
  │   → 需要与墨韵体系统一的视觉语言
  │
  ├── minimalist-skill：色彩克制 + 结构传达
  │   → 不用大面积底色，用结构元素（竖线）标记
  │   → 强调色必须 desaturated、与主色系一致
  │   → 项目已有 --rose (#c45a5a)，朱砂色系
  │
  ├── taste-skill：MOTION_INTENSITY=6
  │   → stagger 入场动画（每段 60ms 递增）
  │   → diff-reveal: translateX(-4px) + opacity
  │
  └── 结论：朱批风格
      → rgba(196,90,90,0.18) 底色 + 3px 左侧竖线
      → 古文批注意象，与墨金体系经典对仗
```

## 4. 优化建议

> 与 [Agent整体设计 §8](Agent整体设计.md) 的汇总索引对应，此处为前端交互专项详细分析。

### P1 — 重要改进

1. **SSE interrupt 无超时处理**：用户确认弹窗无超时机制，如果用户长时间不操作，SSE 连接可能因超时断开，导致后端 interrupt 状态无法恢复。建议：增加超时自动取消逻辑（如 5 分钟无操作自动取消），或前端定期发送心跳保活。
2. **SSE 事件类型过多**：20+ 种事件类型增加前端复杂度，每种事件需要独立的处理逻辑。建议：归并为 3 大类（stream/token/subagent_token/reasoning → stream 类；generate_start/token/done/reset → generate 类；interrupt/handoff/plan_*/task_complete → control 类），前端按类别分发处理。

### P2 — 改善体验

3. **generate_reset 闪烁**：多批次生成时前端内容被清空再重填，用户体验差。建议：前端对 generate_reset 事件做平滑过渡（如淡入淡出、双缓冲），避免内容突然消失再出现。
4. **编辑器保存无冲突检测**：用户在前端编辑器修改字段内容时，如果 Agent 同时也在修改同一字段（如后台增量更新），保存时可能覆盖 Agent 的修改。建议：保存前检查字段内容是否被 Agent 修改过（可基于 content_hash 或时间戳），如有冲突提示用户选择保留哪个版本。
