# 高亮方案设计

> 面向小说作者，而非代码审查者。行级别，非字符级别。

---

## 1. 作者视角

作者使用 InkFlow 的工作流：

```
写提示词 → AI 生成/修改 → 扫一眼改了哪里 → 编辑调整 → 保存 → 继续写
```

高亮在这个流程中只解决一个问题：**"AI 改了哪些行？"**

作者不关心 diff 算法，不关心 insert 还是 replace，不关心被删掉的原文。他们只想快速定位变更位置，然后决定是保留、修改还是重写。

### 作者不需要什么

| 代码编辑器做的 | 为什么作者不需要 |
|---------------|----------------|
| 区分新增/修改（绿/蓝） | 作者只关心"AI 写了什么"，不关心操作类型 |
| 展示删除内容（红色+删除线） | 划掉的原文打断阅读节奏，作者只需要最终版本 |
| 逐词/逐段接受 | 小说是段落级思维，不是逐行审查 |
| 预览模式高亮 | 作者已经看着 streaming 写完了，预览时只想安静阅读 |

### 作者需要什么

1. **快速定位**：一眼扫到变更在哪个段落
2. **不干扰阅读**：高亮是路标，不是路障
3. **编辑不丢失**：编辑后高亮精确追踪到变更行的新位置，不残留、不漂移
4. **编辑无感**：高亮不影响打字、选词、删除

## 2. 交互设计

### 编辑模式：双层标记

```
┌──────┬─────────────────────────────────────┐
│      │  他站在窗前，望着远处的山峦。          │  ← 未修改
│  ██  │░░░░░░░░ 夕阳将天边染成一片金红，     │  ← AI 改动
│  ██  │░░░░░░░░ 云层像是被点燃了一般。        │  ← AI 改动
│      │  他叹了口气。                         │  ← 未修改
└──────┴─────────────────────────────────────┘
   ↑              ↑
 gutter竖条     行级淡色背景
 (精确定位)     (扫视感知)
```

**为什么是双层**：
- 单靠 gutter 竖条，视线需要左右跳转才能对齐"哪一行改了"
- 加上行级底色，变更区域形成视觉块，扫一眼就知道范围
- 这是 VS Code/Trae/Cursor 的标准做法，经过大量用户验证

**为什么只在编辑模式**：
- 编辑模式是"工作态"——作者需要知道改了哪里来决定下一步
- 预览模式是"阅读态"——作者想沉浸式阅读最终效果，高亮是干扰

### 预览模式：纯净阅读

无任何 diff 标记。作者已经通过 streaming 看着 AI 写完了内容，预览时不需要再被提醒"这是新的"。

### 高亮生命周期

高亮存储为**行号**，以 **pre-AI 原始内容**为基准。编辑后实时对比当前内容与原始内容，精确反映仍不同的行。

| 事件 | 行为 | 为什么 |
|------|------|--------|
| 新的 AI 生成/修改 | Store 捕获 pre-AI 原始内容 + 初始高亮 | SSE 携带字符偏移 → 转行号；保存原始内容供后续对比 |
| 用户编辑内容 | **逐行对比 pre-AI 与当前内容**，自动更新高亮 | 删除 AI 改动 → 内容恢复 → 高亮消失；保留的改动继续高亮 |
| 用户保存 | **不清除高亮** | 保存不等于"接受 AI 修改"，高亮持续导航 |
| Session 退出 | 清除所有高亮和原始内容 | 页面卸载时清空，不留残留 |

高亮消退的唯一路径：**新一轮 AI 生成覆盖**，或**内容恢复到与原始一致**。

## 3. 行业对比

| 特性 | VS Code | Trae | Cursor | **InkFlow** |
|------|---------|------|--------|-------------|
| Gutter 竖条 | 3px 多色 | 3px 多色 | popup 浮窗 | **4px 统一绿** |
| 行级背景色 | 有 | 有 | 有 | **已实现** |
| 预览模式高亮 | 有 | 有 | 有 | **不做** |
| 多色区分 | 绿/蓝/红 | 绿/蓝/红 | popup/ghost | **不做** |
| 删除内容展示 | 红色+删除线 | 有 | 有 | **不做** |
| 编辑时追踪 | 保存后消退 | 有 | 有 | **行级实时追踪** |

**核心取舍**：代码编辑器的多色体系服务于审查场景。小说作者只需要一个信号——"这里被 AI 动过"——单色绿色足够。

## 4. 技术实现

### 4.1 核心设计决策：行级别 + pre-AI 基准

两套 diff：
1. **初始高亮**：后端 `compute_diff_highlights(old, AI_new)` → 字符偏移 → Store 转行号
2. **编辑追踪**：前端 `computeLineDiff(pre_AI, current)` → 逐行对比 → 得到当前仍不同的行

```
AI 生成时:
  Store 存入 preEditContent = 原始内容（pre-AI）
  Store 存入 fieldHighlights = 逐行 diff(pre-AI, AI新内容) → [0, 3, 5]

用户编辑后:
  computeLineDiff(pre-AI, 当前内容) → [0, 5]  (第3行恢复，不再高亮)
  Store 更新 fieldHighlights = [0, 5]
```

**为什么用 pre-AI 基准**：pre-AI 是用户确认过的原始内容，用它对比是唯一精确的"哪些行和原来不同"。用 AI 生成内容做基准时，suffix 匹配会残留幽灵高亮。

### 4.2 编辑时追踪：逐行对比

不再追踪单次编辑的行偏移，改为每次编辑后全量对比 pre-AI 与当前内容。O(n) 复杂度，万字章节无压力。

```typescript
// MarkdownEditor.vue
function computeLineDiff(preAI: string, current: string): number[] {
  if (!preAI) return []
  const oldLines = preAI.split('\n')
  const newLines = current.split('\n')
  const result: number[] = []
  const maxLen = Math.max(oldLines.length, newLines.length)
  for (let i = 0; i < maxLen; i++) {
    const oldLine = i < oldLines.length ? oldLines[i] : undefined
    const newLine = i < newLines.length ? newLines[i] : undefined
    if (oldLine !== newLine) result.push(i)
  }
  return result
}

function applyContentChange(newContent: string) {
  emit('update:content', newContent)
  if (props.preEditContent) {
    emit('updateHighlights', computeLineDiff(props.preEditContent, newContent))
  }
}
```

**示例**：

```
pre-AI:    "写作类型：修仙\n主角：张三"
AI生成:    "你好写作类型：修仙\n主角：李四"  → 高亮 [0, 1]
用户删你好: "写作类型：修仙\n主角：李四"       → 高亮 [1] (第1行仍不同)
用户改回:  "写作类型：修仙\n主角：张三"       → 高亮 [] (完全恢复)
```

每次编辑后精确反映"哪些行和原来不同"，不会残留幽灵高亮。

### 4.3 渲染：行号直接使用

Store 存的是行号，组件直接用，无需任何字符遍历。

```typescript
// MarkdownEditor.vue
const diffGutterLines = computed(() => {
  if (!props.content || !props.highlights?.length) return []
  const lines = props.content.split('\n')
  const hl = new Set(props.highlights)
  return lines.map((_, i) => ({ idx: i, changed: hl.has(i) }))
})
```

对比旧版：从字符偏移计算高亮行需要 O(lines × highlights) 字符遍历，新版 O(lines) 的 Set 查找。

### 4.4 叠层结构

```
┌────────────────────────────────────────┐
│  .md-diff-line-bg  (z-index: 0)        │  ← 行级淡绿底色
│    pointer-events: none                │
├────────────────────────────────────────┤
│  .md-diff-gutter   (flex-shrink: 0)    │  ← 左侧 4px 竖条
│    overflow: hidden                    │
├────────────────────────────────────────┤
│  .md-textarea      (z-index: 1)        │  ← 文字输入区
│    background: transparent             │     高亮激活时背景变透明
│    padding: 28px 32px                  │
└────────────────────────────────────────┘
```

**滚动同步**：gutter 和 lineBg 通过 `@scroll` 事件同步 textarea 的 `scrollTop`。
**排版对齐**：三者的 `padding-top: 28px` 一致，保证高亮行与文字行精确对齐。

### 4.5 CSS 变量

```css
:root {
  --diff-bg: #edf3ec;                    /* 行级淡绿底色 */
  --diff-bar: rgba(52, 101, 56, 0.55);   /* gutter 绿色竖条 */
  --diff-gutter-w: 4px;                  /* gutter 宽度 */
}
```

单色体系。绿色低饱和度，不抢夺文字注意力。

### 4.6 完整数据流

```
Python (compute_diff_highlights)
  → SSE { type: "field_content", target: "...", content: "...", highlights: [[48,72], ...] }
  → EditorPage.handleSseEvent() → case "field_content"
  → editorStore.handleGenerateEvent()
    → preEditContent[target] = fieldValues[target]   ← 捕获 pre-AI 原始内容
    → fieldValues[target] = content
    → 字符偏移 → 行号转换 → fieldHighlights[target] = [2, 5]
    → editingField = target
  → MarkdownEditor 接收 :highlights="[2, 5]" :pre-edit-content="原始内容"
  → diffGutterLines computed → 行 0-1 无标记，行 2、5 有标记
  → gutter 竖条 + lineBg 行背景渲染

编辑时（输入、工具栏插入、加粗等统一走 applyContentChange）：
  onInput / insert / wrap → applyContentChange(newContent)
    → computeLineDiff(preEditContent, newContent) → [3, 5]
    → emit('updateHighlights', [3, 5])
  → EditorPage.onUpdateHighlights → fieldHighlights[target] = [3, 5]

新一轮生成开始（generate_start / generate_reset）：
  → delete fieldHighlights[target] + delete preEditContent[target]
```

**SSE 事件说明**：

| SSE event type | 携带 highlights？ | 高亮行为 |
|---------------|------------------|---------|
| `field_content` | **是**（字符偏移） | 转为行号，直接覆盖当前字段高亮 |
| `generate_start` | 否 | **清除**当前字段旧高亮 |
| `generate_reset` | 否 | **清除**当前字段旧高亮 |
| `generate_token` | 否 | 不触发高亮逻辑 |
| `generate_done` | 否 | 调用 `fetchState()` 全量刷新 |

章节续写走 `generate_token` 流式写入，不走 `field_content`，不产生高亮。高亮仅在 AI 批量修改已有内容时出现。

点击 gutter 变更行跳转到下一个变更位置，循环回到第一个。hover 时竖条变亮 + 显示 `▾` 箭头。

## 5. 设计约束

### 色彩

- **单色**：绿色标记所有变更，不引入第二色
- **低饱和度**：竖条 `rgba(52, 101, 56, 0.55)`，底色 `#edf3ec`，不刺眼
- **禁止外发光**：竖条是纯色块，不是 `box-shadow` 发光条
- **禁止紫蓝渐变**：AI slop 特征，明确排除

### 动画

- **无入场/退场动画**：高亮是持久标记，不是临时通知
- **即时切换**：新高亮替换旧高亮时直接切换

### 排版

- **不改变文字属性**：font-size、font-weight、line-height 不动
- **不侵入文本**：gutter 在文本区外，行背景是半透明叠层
- **预览模式零标记**：不插入 `<mark>` 标签

### 性能

- **万字章节流畅**：`computeLineDiff` O(n)，每次输入全量对比一次
- **pointer-events: none**：行背景层不拦截鼠标
- **z-index 最小化**：行背景 0，textarea 1
