## 1. 上下文压缩

> 详细设计见 [记忆系统设计 §2.4](记忆系统设计.md#24-上下文压缩参考-hermes-contextcompressor)，此处仅列出实现要点。

- **文件**：`agent/runtime/compression.py`
- 参考 Hermes 4-phase 压缩算法，摘要直接替换 messages 中间段，无需独立 `history_summary` 字段

**核心流程**：
```
compress(state)
  → Phase 1: 确定边界（tail 3 轮，不保护 head，_align_boundary_backward 对齐 tool 对）
  → Phase 2: 修剪（3 轮外 >200 字符的 tool_result 替换为占位符）
  → Phase 3: LLM 生成 7 段结构化摘要（传入 _previous_summary 增量更新）
  → Phase 4: 重组 messages = [summary] + tail + _sanitize_tool_pairs()
```

**触发条件**：`estimate_tokens(messages) > context_window × 0.5`

**参数来源**：`MessageCompressor(compact_threshold_ratio=0.5)`，不在 `token_config.json` 中，需修改代码调整。

**防抖**：连续 2 次压缩 token 节省 < 10% → 跳过 20 轮后重试

**预压缩记忆刷新**：压缩前先提取关键事实写入 short_memory.md（而非 MEMORY.md），保证 MEMORY.md session 内冻结。

**Prompt Assembly**：消息构建方式见 [记忆系统设计 §2.2](记忆系统设计.md#22-上下文构建与缓存prompt-assembly)，`state.messages` 含压缩摘要 + 最近 3 轮，作为 slot [6] 整体注入。

## 2. LLM 调用层

- **文件**：`agent/runtime/llm.py`
- 三种调用模式：

| 函数 | 返回类型 | 适用场景 | temperature |
|------|---------|---------|-------------|
| chat() | str | 摘要/标题/记忆提炼 | 0.7 |
| chat_stream() | AsyncGenerator[str] | 章节内容/字段流式生成 | 0.7 |
| chat_tools_stream() | AsyncGenerator[StreamChunk] | ReAct 工作流 | 0.3 |

**StreamChunk 结构**：`content` / `tool_calls` / `is_tool_call` / `reasoning_content`

**tool_calls 缓冲**：流式分片的 tool_call 片段通过 `tool_call_buffer` 拼装为完整调用，最终以单个 StreamChunk 返回。

**重试机制**：`chat()` 对 429/500/502/503 错误和超时自动退避重试（最多 3 次，带 jitter），其他错误直接抛出。`ContextOverflowError` 单独识别并向上传播。

**多模型支持**：
- `TOOL_CALL_MODEL`：ReAct 工作流（低 temperature，高精度）
- `COMPRESSION_MODEL`：摘要/评估/记忆提炼（低成本）
- `DEFAULT_MODEL`：默认生成任务

## 3. 任务评估

### 3.1 两级评估

```
quick_evaluate(called_tools, tool_results, agent_response) → bool|None
  ├── 无工具调用:
  │   ├── 有回复 → True（反问/已回答）
  │   └── 无回复 → False
  ├── 有写入工具:
  │   ├── 无失败 → True
  │   └── 有失败 → False
  ├── task_complete + 无写入 → None → LLM 评估器
  └── 有回复 + 无写入 → None → LLM 评估器
```

### 3.2 LLM 评估器

- **模板**：`templates/system/evaluator.md`
- 只在 `quick_evaluate()` 返回 None 时调用
- 典型场景：Reader 回答问题后调用 task_complete（无写入工具）
- 三种判断：COMPLETED / NOT_COMPLETED / NEEDS_INPUT
- 安全规则：无法确定 → COMPLETED（避免死循环）

### 3.3 评估在 AgentLoop 中的位置

```
_handle_subagent_result()
  ├── plan_status == "completed" → is_complete=True（不评估）
  ├── plan_status == "executing" → is_complete=False（继续执行）
  └── 无 Plan → _evaluate_and_decide()
      ├── completed=True → is_complete=True
      ├── completed=False + iteration < max → is_complete=False + reflexion
      └── completed=False + iteration ≥ max → is_complete=True（强制完成）
```

## 4. 模板系统

### 4.1 目录结构

```
templates/
├── fields/          — 字段生成与修改模板
│   ├── settings.md          — 写作设定
│   ├── characters.md        — 角色档案
│   ├── relationships.md     — 关系图谱
│   ├── foreshadowing.md     — 伏笔清单
│   ├── foreshadowing_scan.md— 伏笔扫描
│   ├── outline_historical.md— 历史大纲
│   ├── outline_future.md    — 未来大纲
│   ├── chapter_content.md   — 章节正文
│   ├── chapter_title.md     — 章节标题
│   └── update_field.md      — 局部修改
├── subagent/        — Subagent 系统提示词
│   ├── subagent_reader.md   — Reader
│   ├── subagent_creator.md  — Creator
│   └── subagent_editor.md   — Editor
└── system/          — 系统级模板
    ├── lead_harness.md      — Lead Agent 路由决策
    └── evaluator.md         — 任务评估器
```

### 4.2 加载机制

- **文件**：`agent/templates/loader.py`
- `_find_template(name)` — 在 templates/ 及子目录中递归查找
- 内存缓存 `_cache`，避免重复读磁盘
- 模板使用 Python `str.format()` 语法

## 5. 优化建议

> 与 [Agent整体设计 §8](Agent整体设计.md) 的汇总索引对应，此处为运行时专项详细分析。

### 消息压缩

1. ~~**压缩触发阈值过低**~~：已解决。`compact_threshold_ratio` 已从 0.5 提高到 0.7，详见 [记忆系统设计 §2.4](记忆系统设计.md#24-消息压缩)。
2. ~~**最近N轮对话重复**~~：已解决。去掉 `build_memory_context()` 中的"最近N轮对话"注入，`state.messages` 截断为有界，详见 [记忆系统设计 §2.2](记忆系统设计.md#22-上下文构建与缓存prompt-assembly)。
3. **预压缩记忆刷新成本高**：每次压缩都调用 LLM 提取关键事实写入 short_memory.md，增加一次 API 调用。建议：仅在距上次刷新超过 N 轮时才执行。
4. **摘要质量无校验**：LLM 生成的摘要直接替换历史消息，无长度或质量校验。建议：校验摘要长度不超过 old_text 的 30%，超长则截断。

### LLM 调用层

4. **chat_stream / chat_tools_stream 无重试**：`chat()` 对 500 错误自动退避重试（最多 3 次），但 `chat_stream()` 和 `chat_tools_stream()` 没有重试机制，API 错误直接传播。流式调用一旦中断，前端 SSE 连接断开，用户必须重新发送消息。建议：增加流式重试（重新发起请求），最多重试 1 次，并在重试时通过 SSE 推送 `retry_notice` 事件通知前端。
5. **temperature 硬编码**：生成类任务 0.7、工具调用 0.3，无法动态调整。建议：通过配置文件或参数传入，支持不同场景使用不同 temperature。
6. **max_tokens 硬编码 100000**：所有调用统一 100K max_tokens，简单任务（如标题生成、记忆提炼）浪费配额。建议：按任务类型设置不同的 max_tokens（标题 200、摘要 2000、章节 100000）。

### 任务评估

7. **评估器对 Plan 模式无效**：Plan 执行中，步骤完成后直接 `is_complete=False`（继续下一步），不经过评估器。只有最后一步完成后才设 `is_complete=True`。这意味着中间步骤失败时，评估器无法介入。建议：在 Plan 的每个步骤完成后也做 quick_evaluate，失败时提前终止 Plan。
8. **LLM 评估器调用频率未知**：建议加埋点统计 quick_evaluate 返回 None 的比例和 LLM 评估器的实际调用次数，用于调优规则引擎覆盖率。
9. **reflexion 注入方式**：评估器判定未完成时，reflexion 消息注入到下一轮的 system 消息中，但 Lead Agent 的 `_build_harness_messages()` 将其放在对话历史之前，可能被 LLM 忽略。建议：将 reflexion 放在更突出的位置（如 system 消息开头）。

### 模板系统

10. **模板变量无校验**：`str.format()` 在变量缺失时抛 KeyError，但错误信息不明确。建议：增加模板变量校验，在加载时检查所需变量是否完整，给出清晰的缺失字段提示。
11. **缓存无失效机制**：`_cache` 一旦加载永不清除（除非手动 `clear_cache()`）。开发时修改模板不会生效。建议：增加基于文件修改时间的缓存失效。
