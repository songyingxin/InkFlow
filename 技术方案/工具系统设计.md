## 一、注册与调度

> 源码：`agent/tools/registry.py` + `schema.py` + `dispatch.py` + `common.py`

### 1.1 注册机制

`@ToolRegistry.register(name, schema, *, toolset)` 装饰器，注册 handler 到 `ToolRegistry._tools`。

**注册流程**：
```
schema.py 定义 SCHEMA 常量
  → handler 文件 @register_tool("name", schema=SCHEMA, toolset="creator")
  → ToolRegistry.discover() 自动扫描 tools/ 目录，触发所有 handler 模块加载
  → ToolRegistry._tools[name] = _ToolEntry(name, schema, handler, toolset)
```

**自动发现机制**：`ToolRegistry.discover()` 使用 `pkgutil.iter_modules` 扫描 `tools/` 目录下所有 Python 模块，触发 `importlib.import_module` 加载，各模块的 `@register_tool` 装饰器在 import 时自动执行注册。`_discovered` 标志防止重复扫描。

**ToolRegistry API**：
- `register(name, schema, *, toolset)` → 装饰器，绑定 schema + handler + toolset
- `discover()` → 自动扫描 tools/ 目录，触发所有 handler 模块加载注册
- `get_handler(name)` → 获取 handler 函数
- `get_schema(name)` → 获取 JSON Schema
- `get_all_schemas()` → 所有工具 schema（供 LLM function calling）
- `get_schemas_for(tool_names)` → 按名称列表筛选 schema
- `get_names_for_toolset(toolset)` → 按工具集分类名获取工具名列表

**工具集分类（toolset）**：每个工具注册时属于一个 toolset，`get_names_for_toolset()` 按 toolset 筛选。当前 toolset 分类：

| toolset | 用途 | 包含的工具 |
|---------|------|-----------|
| `reader` | 审阅者工具集 | read_novel_content, check_consistency, analyze_pacing, foreshadowing_status |
| `creator` | 创作者工具集 | continue_writing, regenerate_chapter, generate_*, init_novel |
| `editor` | 修改者工具集 | update_field, update_outline_*, scan_foreshadowing |
| `write` | 写入工具集（评估器用） | continue_writing, regenerate_chapter, generate_*, update_*, init_novel, scan_foreshadowing |
| `memory` | 记忆工具集 | search_memory, memory_append, memory_rewrite |
| `lead` | 编排工具集 | handoff_to_reader/creator/editor |
| `default` | 通用工具集 | task_complete, read_novel_content |

角色的 `allowed_tools` 在 `registry.py` 中显式声明，不再通过 toolset 隐式解析。详见 [Agent整体设计 §2 模板系统](Agent整体设计.md#2-skill-系统)。

### 1.2 调度与错误处理

`dispatch_tool()` 统一分发：查找 handler → 解析参数 → 调用 → 错误增强

**错误分类**：

| 类别 | 匹配模式 | 处理 |
|------|---------|------|
| retryable | "未找到匹配"/"匹配失败"/"超时"/"429"/"503" | 增强错误消息 |
| unrecoverable | "不支持的字段"/"缺少参数"/"未知工具" | 附加"无法重试"提示 |
| unknown | 其他 | 原样返回 |

**update_field 增强错误**：失败时自动加载当前字段内容，附加摘要到错误消息，帮助 LLM 下次构造更精确的 patches。

### 1.3 共享基础设施

**ToolResult**（`common.py`）：
```python
@dataclass
class ToolResult:
    success: bool = True
    content: str = ""
    error: str | None = None
```

**ask_user_confirmation**（`common.py`）：
- 通过 LangGraph `interrupt()` 实现用户确认交互
- 图执行暂停并持久化到 checkpointer
- 前端通过 `POST /api/chat/resume` 恢复执行

**get_writer**（`common.py`）：
- 获取 stream_writer：优先从 state 取，否则用 LangGraph 上下文

### 2.1 task_complete

**源码**：`agent/tools/control.py`

标记任务完成，是唯一表明任务结束的方式。不调用此工具，任务将不会结束。

参数：

### 2.2 search_memory

**源码**：`agent/tools/read.py`（与 read_novel_content 同文件）

搜索记忆系统中的相关内容。支持按关键词搜索长期记忆、设定文件和对话记录。

参数：
- `query`（必填）：搜索查询关键词或短语
- `top_k`：返回结果数量，默认 5
- `source_filter`：限定搜索来源（memory/field/chat）

实现：调用 `MemoryManager.search_memory()`，底层为 FTS5 Trigram + jieba 重排。

### 2.3 memory_append

**源码**：`agent/tools/memory.py`

将重要事实追加到 `short_memory.md`。写入后立即可见（slot [3]），session 结束时统一提升到 MEMORY.md。

参数：
- `fact`（必填）：要记录的持久事实

### 2.4 memory_rewrite

**源码**：`agent/tools/memory.py`

触发 MEMORY.md 的立即整合重写。当积累较多新记忆或发现记忆矛盾时使用。

无参数。调用 `rewrite_memory_md()` 合并新旧内容，去重去矛盾。MEMORY.md 超过 5000 字符硬上限时也会自动触发。

## 三、读取与分析类工具

> 源码：`agent/tools/read.py` + `agent/tools/analysis.py`

| 工具 | Subagent | 说明 |
|------|---------|------|
| read_novel_content | 全部 | 读取小说内容 |
| check_consistency | Reader | 检查设定一致性 |
| analyze_pacing | Reader | 分析叙事节奏 |
| foreshadowing_status | Reader | 伏笔状态报告 |

### 3.1 read_novel_content

**源码**：`agent/tools/read.py`

#### 读取策略

| content_type | 策略 | 说明 |
|-------------|------|------|
| chapter | `_read_chapter()` | 指定章节号读取 |
| recent_chapters | `_read_recent_chapters()` | 最近 N 章（默认 3） |
| search | `_search_chapters()` | 关键词搜索章节+字段 |
| settings/characters/... | `_read_field_content()` | 读取字段内容 |

支持 `content_type` 为列表，批量读取多种类型。

#### 截断策略

`_truncate_content()` — 超过 3000 字时保留 2/3 头部 + 1/3 尾部。

**注意**：设定类字段（settings/characters/relationships/foreshadowing）不截断，返回完整内容。只有章节类内容（chapter/recent_chapters）才走截断逻辑。

#### query 过滤

`_filter_by_query()` — 按关键词过滤行，保留匹配行前后各 1 行上下文。

#### 搜索模式

`_search_chapters()` 先搜章节正文，再搜字段文件（`_search_fields()`），合并返回。

### 3.2 check_consistency

**源码**：`agent/tools/analysis.py`

检查小说设定的一致性，检测跨章节的矛盾和不一致。

参数：
- `scope`：检查范围（all/characters/settings/recent）

实现：加载相关设定字段 + 章节正文，由 COMPRESSION_MODEL 对比分析，输出矛盾列表。

检查维度：
1. 角色属性矛盾（年龄、外貌、能力等前后不一致）
2. 世界观设定矛盾（力量体系、地理、时间线等冲突）
3. 情节逻辑矛盾（事件顺序、因果关系不合理）
4. 称呼/名字不一致

### 3.3 analyze_pacing

**源码**：`agent/tools/analysis.py`

分析最近若干章节的叙事节奏，统计动作/对话/描写/心理的比例分布。

参数：
- `chapter_count`：要分析的章节数，默认 5

实现：加载章节正文，由 COMPRESSION_MODEL 分析各章叙事元素比例，输出节奏评估和改进建议。

叙事元素分类：动作 / 对话 / 描写 / 心理 / 过渡

### 3.4 foreshadowing_status

**源码**：`agent/tools/analysis.py`

汇总伏笔清单中所有伏笔的当前状态，按状态分类统计。

参数：
- `filter_status`：筛选特定状态（active/unresolved/all）

实现：解析伏笔清单中的状态标记（🔵规划中/🟡活跃中/🟢已回收/🔴已废弃/⚪已偏移），按状态分组统计，输出回收建议。

### 4.1 update_field

**源码**：`agent/tools/update.py`

#### 执行路径

```
handle_update_field(field, user_request, patches)
  ├── patches 模式:
  │   ├── apply_patches(existing, patches)
  │   │   ├── 精确匹配 (str.find) → 成功 ✅ → 记录 highlight 区间
  │   │   ├── 模糊匹配 (_fuzzy_find)
  │   │   │   ├── 空白归一化匹配 → 成功 ✅ (带警告) → 记录 highlight 区间
  │   │   │   └── 锚点前缀匹配 (前80字符) → 成功 ✅ (带警告) → 记录 highlight 区间
  │   │   └── 全部失败 → 记录未匹配
  │   ├── 部分成功 → 保存已匹配的 + 警告未匹配的 + 返回 highlights
  │   └── 全部失败 → 降级到 LLM 模式
  │       └── _update_field_via_llm()
  │           └── LLM 生成 SEARCH/REPLACE → apply_search_replace() → 保存 + highlights
  └── user_request 模式:
      └── _update_field_via_llm()
          ├── LLM 流式生成 SEARCH/REPLACE 块
          ├── apply_search_replace() 合并 → 返回 highlights
          ├── 合并失败 → 回退到 raw_output → compute_diff_highlights() 计算 highlights
          └── ask_user_confirmation() → interrupt() 用户确认
              ├── "应用修改" → save_field_content → 保存 + 发送 field_content(highlights)
              └── "放弃修改" → 返回
```

#### apply_patches 核心逻辑

三层匹配策略：
1. **精确匹配**：`str.find(old_text)` — 零容错，最高优先级
2. **空白归一化匹配**：`_normalize_ws()` 去除首尾空白、统一换行/空格后匹配
3. **锚点前缀匹配**：取 old_text 前 80 字符在原文中定位

部分成功时保存已匹配的补丁 + 警告未匹配的；全部失败时自动降级到 LLM 模式。

#### compute_diff_highlights

基于 `difflib.SequenceMatcher` 对比新旧内容，计算变更区间列表，用于前端高亮渲染。

```python
def compute_diff_highlights(old_content, new_content) -> list[tuple[int, int]]:
    # 空旧内容 → 全部高亮
    # 内容相同 → 无高亮
    # 否则 → 提取 insert/replace 操作的区间
```

**调用场景**：
1. `apply_patches()` 精确/模糊匹配成功时，记录替换区间为 highlight
2. `_update_field_via_llm()` 中 SEARCH/REPLACE 合并成功时，由 `apply_search_replace()` 返回 highlights
3. `_update_field_via_llm()` 中 SEARCH/REPLACE 合并失败时，回退到 raw_output，调用 `compute_diff_highlights(existing, full_content)` 计算 diff highlights

#### apply_search_replace

解析 LLM 输出的 `<<<<<<< SEARCH / ======= / >>>>>>> REPLACE` 块，内部转换为 patches 列表，复用 `apply_patches()` 的匹配逻辑。

#### 级联提示

修改字段后，`FieldRegistry.cascade_fields()` 返回下游字段，提示用户同步更新。

### 4.2 update_outline / update_outline_historical / update_outline_future

**源码**：`agent/tools/generate.py`

#### 增量更新流程

```
_do_incremental_update(state, unread, short_field, label)
  → 加载未读章节内容（截断到 800 字/章）
  → 构造 user_request：新增章节内容 + 局部修改指令
  → update_field_stream() → LLM 增量修改
  → save_field_content() → 保存 + 更新 read_ch
```

**update_outline**：协调器，依次调用 historical + future。

**update_outline_future**：额外注入历史大纲内容作为上下文。

**未读章节判断**：`get_unread_chapter_indices(novel_state, "outline_historical_read_ch")` — historical 和 future 共用同一个 read_ch 字段。

### 4.3 scan_foreshadowing

**源码**：`agent/tools/scan.py`

#### 扫描流程

```
handle_scan_foreshadowing(chapter_num)
  → 加载章节正文
  → _scan_chapter_foreshadowing() → LLM 检测伏笔变化
    → 使用 foreshadowing_scan 模板
    → 输入：当前伏笔清单 + 章节正文
  → _has_changes() → 判断是否有变化
  → _extract_new_foreshadowing_patches() → 提取补丁
  → apply_patches() → 自动更新伏笔清单
```

#### 自动更新机制

扫描结果中提取"新伏笔"和"伏笔回收"两类变更，构造 patches 自动应用到伏笔清单。

### 5.1 字段生成流程

**源码**：`agent/tools/generate.py`

```
handle_generate_field(field, label, user_request, reread_all)
  → 确定现有内容 existing
  → 判断 reread_all（None → 自动判断，可能 interrupt 询问用户）
  → 根据字段类型选择流:
    ├── outline_historical → _build_historical_stream()

```

## 六、优化建议

> 与 [Agent整体设计 §8](Agent整体设计.md) 的汇总索引对应，此处为工具系统专项详细分析。

### P0 — 必须修复

1. **update_outline_future 使用错误的 read_ch**：与 historical 共用 `outline_historical_read_ch`，语义错误。未来大纲的增量更新应该基于"未来大纲自身读到第几章"，而非历史大纲的进度。建议：为未来大纲增加独立的 `outline_future_read_ch` 字段（需同步修改 MetaInfo 和 FieldRegistry）。

### P1 — 重要改进

2. **工具数量过多（22个）**：大纲相关有 6 个工具（generate_outline/historical/future + update_outline/historical/future），LLM 经常在 generate_outline vs generate_outline_historical 之间选错。建议：合并为 `generate_outline(parts=["historical","future"])` 和 `update_outline(parts=["historical","future"])`，parts 参数指定更新哪部分。
3. **search_memory 与 read_novel_content(search) 重叠**：两者都能搜字段内容，LLM 经常选错。read_novel_content 的 search 模式是纯 Python 字符串匹配，远不如 FTS5。建议：删除 read_novel_content 的 search 模式，统一用 search_memory。
4. **scan_foreshadowing 自动更新伏笔清单很脆弱**：用 `foreshadowing[pos-30:pos]` 做精确匹配来插入内容，格式稍有变化就失败，且只处理"新伏笔"和"回收"，不处理"偏移"。建议：scan_foreshadowing 只返回扫描结果，让 Agent 自行用 update_field 更新。
5. **regenerate_chapter 局部重写不可用**：关键词匹配（"局部重写/只改/后半段"）+ MARK 分割逻辑，LLM 不会输出 `<<<<<<< MARK`。建议：删除局部重写逻辑，regenerate_chapter 只做整章重写，局部修改走 update_field(patches)。

### P2 — 改善体验

6. **截断策略已优化**：设定类字段不再截断，返回完整内容。章节类内容仍使用 2/3 头部 + 1/3 尾部截断。可进一步优化：基于 query 关键词定位，保留匹配行附近的上下文。
7. **update_field patches 失败率无统计**：LLM 生成的 patches 经常匹配失败，但无失败率统计和自动优化。建议：记录 patches 匹配成功率，当低于阈值时自动降级到 LLM 模式。
8. **interrupt 恢复时无已有内容传递**：用户确认后恢复执行时，不传递 interrupt 前的已有内容，导致重复读取。建议：interrupt 恢复时将已有内容注入上下文。
