## 1. 记忆系统总体架构

InkFlow 的记忆系统由两个职责明确的子系统组成：

```
┌──────────────────────────────────────────────────────────────┐
│                    InkFlow 记忆系统                            │
├──────────────────────────────┬───────────────────────────────┤
│     对话记忆系统              │       小说记忆系统              │
│  （管理 Agent 与用户的交互）    │  （管理小说创作的结构化知识）     │
│  ConversationMemory 门面      │  NovelMemory 门面              │
├──────────────────────────────┼───────────────────────────────┤
│ · 对话记录 (chat.db)         │ · 字段文件 6 个                │
│ · 会话元数据 (chat.db)       │   settings / characters /     │
│ · 短期缓冲 (short_memory.md) │   relationships / foreshadowing│
│ · 长期记忆 (MEMORY.md)       │   outline_historical /         │
│ · Agent 自主记忆管理          │   outline_future               │
│ · 检索引擎 (FTS5+jieba)      │ · 章节全文 (chapters/NNN.md)  │
│ · 会话管理 (Session)          │ · 章节索引 (outline_structure) │
│                              │ · 元信息 (meta.json)           │
├──────────────────────────────┼───────────────────────────────┤
│         共享基础设施                                           │
│  · 检索引擎 (FTS5+jieba，对话记忆和小说记忆共用)                │
│  · 持久化层 (persistence.py，统一文件读写+备份)                 │
│  · 配置 (TruncationConfig，所有截断/阈值参数)                   │
└──────────────────────────────────────────────────────────────┘
```

> 上下文如何消费记忆系统的数据（slot 设计、stable/volatile 划分、压缩策略、缓存策略）见 [上下文设计.md](上下文设计.md)。

**两个子系统的关系**：

- 对话记忆系统是**通用 Agent 基础能力**，负责"记住用户说了什么"
- 小说记忆系统是**领域专用知识管理**，负责"记住小说写了什么"
- 对话记忆的蒸馏产物（MEMORY.md 中的创作决策）会**影响**小说记忆的字段内容
- 小说记忆的字段内容会**注入**对话记忆的上下文构建中

**文件体系**：

```
workspace/书名/
├── MEMORY.md              ← 长期记忆（对话记忆：session 间更新的持久事实，session 内冻结）
├── short_memory.md        ← 短期缓冲（对话记忆：Agent 手动写入的临时事实，提升后清空）
├── settings.md            ← 设定（小说记忆）
├── characters.md          ← 角色档案（小说记忆）
├── relationships.md       ← 关系图谱（小说记忆）
├── foreshadowing.md       ← 伏笔清单（小说记忆）
├── outline_historical.md  ← 历史大纲（小说记忆）
├── outline_future.md      ← 未来大纲（小说记忆）
├── outline_structure.json ← 章节索引（小说记忆）
├── meta.json              ← 元信息（小说记忆）
├── chat.db                ← 对话记录 + 会话元数据 + checkpoint（对话记忆）
├── chapters/
│   └── NNN.md             ← 章节全文（小说记忆）
└── backups/               ← 备份
```

## 2. 对话记忆系统

对话记忆系统负责管理 Agent 与用户的交互历史，确保 Agent 在多轮对话中保持上下文连贯。

### 2.1 记忆蒸馏管线

参考 Hermes 的 Agent 自主记忆管理设计，引入 **short_memory.md 短期缓冲层**。核心约束：**MEMORY.md 在 session 期间冻结，Agent 手动写入走 short_memory，session 结束时统一提升**——保证 stable prefix 的 KV cache 持续命中。

```
chat.db 原始对话（全量存档，FTS5 按需检索）
    │
    │  Agent 手动 → memory_append / memory_rewrite → short_memory.md
    ▼
short_memory.md 短期缓冲（session 内可变，注入 slot [5]）
    │  Session 结束 → flush_short_memory（统一提升到长期记忆）
    ▼
MEMORY.md 长期记忆（session 内冻结，仅 session 间更新）
    │  超过上限 → rewrite_memory_md（Hermes 式硬上限压缩）
    ▼
   (带备份 + 实体丢失检测的安全写入)

FTS5 索引刷新（收尾）：
    index_all_memory_files → 重建内存索引
```

**三层记忆的职责分工**：

| 层 | 文件 | 内容 | 生命周期 | 写入者 | Prompt 位置 |
|---|------|------|---------|--------|------------|
| L1 原始存档 | chat.db + MemoryIndex | 全量对话 + 字段文件 FTS5 索引 | 永久 | 系统自动 | FTS5 按需检索 |
| L2 短期缓冲 | short_memory.md | Agent 手动写入的事实 | 临时（session 结束提升后清空） | Agent | slot [5]（volatile） |
| L3 长期记忆 | MEMORY.md | 提升后的持久事实 | 永久（rewrite 管理） | session 间提升 | slot [1]（stable） |

**关键设计决策**：

- **MEMORY.md session 内冻结**：MEMORY.md 是 slot [1] 的 stable prefix，用于 KV cache。Session 内修改会导致缓存失效。因此 Agent 手动写入走 short_memory.md，session 结束时统一提升
- **Agent 自主管理记忆 + nudge 提醒（对齐 Hermes）**：和 Hermes 一样，只有 Agent 自己决定什么值得记。没有自动提取。但纯靠 system prompt 静态指令容易被忽略，因此引入 **nudge 机制**——每 N 轮（默认 5，可配置）在 messages 中注入一条 system message提醒 Agent 反思并保存重要信息。nudge 是运行时强提醒，比静态 prompt 更可靠
- **short_memory.md 是缓冲区，不是蒸馏中间层**：写入后立即可见（slot [5]），提升时直接追加到 MEMORY.md，无需 LLM 调用
- **提升时机只在 session 结束**：不在压缩时提升

**MEMORY.md 更新机制**（仅 session 间）：

| # | 更新路径 | 触发条件 | 写入方式 | 职责 |
|---|---------|---------|---------|------|
| 1 | `flush_short_memory` | Session 结束时 | 追加 | 短期缓冲统一提升 |
| 2 | `rewrite_memory_md` | MEMORY.md 超限（>5000 字符） | 重写 | 容量管理 |

**short_memory.md 更新机制**（session 内）：

| # | 更新路径 | 触发条件 | 写入方式 | 职责 |
|---|---------|---------|---------|------|
| 1 | `memory_append` / `memory_rewrite` | Agent 手动调用 | 追加/重写 | Agent 自主决定记什么 |
| 2 | `flush_short_memory` | Session 结束时 | 清空 | 提升到 MEMORY.md 后清空 |

**与 Hermes 对比**：

| | Hermes | InkFlow |
|---|--------|--------|
| Agent 手动写入 | 直接写 MEMORY.md | 写 short_memory，session 结束提升 |
| MEMORY.md 更新 | session 内可变 | session 内冻结 |
| KV cache | Agent 写入后失效 | session 内持续命中 |
| 记忆提醒 | nudge（每 N 轮注入 system message） | nudge（同 Hermes） |

#### 2.1.1 Nudge 机制（对齐 Hermes）

参考 Hermes 的后台 nudge 审查线程设计，InkFlow 将 nudge 作为**同步注入**而非后台线程——在每轮 LLM 调用前检查是否需要注入 nudge 提醒。

**核心思路**：Agent 拥有 `memory_append` 工具，但纯靠静态 system prompt 指令，Agent 在密集的任务执行中容易"忘记"调用。nudge 是周期性的运行时强提醒，迫使 Agent 停下来反思"是否有值得记录的事"。

**触发条件**：

```
should_nudge(state)
  → count = (state.turn_count - state.last_nudge_turn)  # 距离上次 nudge 的轮数
  → count >= memory_nudge_interval（默认 5，可配置）
  → 返回 True → 注入 nudge system message
  → 注：nudge 只对 Creator 和 Editor 注入，Reader（零变更）不注入
```

**注入方式**：在 `state.messages` 末尾（当前用户消息之前）追加一条 system message。

**nudge 间隔重排**：nudge 间隔仅在 nudge 轮次**触发后**重新计数，非 nudge 轮次不做变更。

注意——nudge 不强制 Agent 回应；Agent 可以判断"Nothing to save"然后继续处理用户请求。nudge 消息末尾包含提示："如果没什么值得记录的，请直接说'Nothing to save.'然后继续处理用户的请求，不要停在这里。"

**nudge 模板**：`novel_agent/agent/templates/system/memory_nudge.md`

**与 Hermes 的差异**：

| | Hermes | InkFlow |
|---|--------|--------|
| 注入方式 | 后台线程（非阻塞） | 同步注入（当前轮） |
| 目标 | 所有 toolset | 仅 Creator / Editor |
| 写入目标 | 直接写 MEMORY.md | 写 short_memory.md |
| nudge 后记忆可见 | 下一轮（system prompt 重建） | 下一轮（slot [5] 重读） |
| 默认间隔 | 10 轮 | 5 轮（创作对话节奏更快） |

InkFlow 选择同步注入而非后台线程，因为创作任务的性质不同：代码 Agent 的后台审查不会打断开发流程，但创作 Agent 的"停下来反思"本身就有价值——用户在等待产出的间隙，Agent 整理记忆是一个自然的暂停点。

### 2.2 检索引擎（双索引源，对齐 Hermes session search）

InkFlow 的检索引擎同时搜索两个 FTS5 索引源，合并结果后注入 slot [6]。

**核心原则**：已在 prompt 中的内容不需要检索。MEMORY.md（slot [1]）和 short_memory.md（slot [5]）已直接注入 prompt，无需重复检索。

```
双索引源检索架构：

  查询输入
    │
    ├──→ 索引源 1：MemoryIndex（memory_index.db）
    │     索引内容：6 个字段文件（设定/角色/关系/伏笔/历史大纲/未来大纲）
    │     不索引：MEMORY.md（已在 slot [1]）、short_memory.md（已在 slot [5]）
    │     特点：字段内容不在 prompt 中，需按需检索，identity 类型，慢衰减
    │
    ├──→ 索引源 2：chat.db FTS5（messages_fts + messages_fts_trigram）
    │     索引内容：全量对话记录
    │     特点：原始对话，temporal 类型，快衰减
    │     对齐 Hermes 的 session search 能力
    │
    └──→ 合并去重 → 时间衰减 → jieba 分词重叠度重排 → 最终结果
```

**MemoryIndex 检索流程**（索引源 1）：

```
FTS5 Trigram 广域召回 + jieba 语义重排：

  查询输入
    │
    ├──→ jieba 分词 → 长词(≥3字) → FTS5 Trigram 精确匹配
    │                → 短词(1-2字) → SQLite LIKE 精确匹配
    │
    └──→ 合并去重 → 时间衰减 → jieba 分词重叠度重排 → 最终结果

  其他能力：
    - 重叠滑窗分块：400 token 目标 + 80 token 重叠 + 行边界保留
    - 去重：SHA-256 hash，重复内容 increment reinforcement_count
    - 时间衰减：fact_type 差异化衰减率
      identity(0.1) < decision(0.3) < preference(0.5) < temporal(1.0) < state(2.0)
    - 强化：reinforcement_count 的 log1p 加权
    - source → fact_type 自动映射：field→identity, chat→temporal
```

**chat.db 检索流程**（索引源 2，对齐 Hermes session search）：

```
chat.db 自带 FTS5 双分词器索引：

  查询输入
    │
    ├──→ CJK 文本 → messages_fts_trigram（Trigram 分词）
    ├──→ 英文文本 → messages_fts（unicode61 分词）
    │
    └──→ BM25 排序 → 返回 top_k 对话片段
```

**与 Hermes 的对比**：

| | Hermes | InkFlow |
|---|--------|--------|
| 索引源 | state.db（全量对话） | MemoryIndex（字段文件） + chat.db（全量对话） |
| 对话检索 | FTS5 + Gemini Flash 摘要 | FTS5 + jieba 重排 |
| MEMORY.md | 直接注入 prompt，不检索 | 直接注入 slot [1]，不检索 |
| short_memory | 不存在 | 直接注入 slot [5]，不检索 |
| 字段文件检索 | 不需要（无领域字段） | FTS5 检索（角色/关系/伏笔等不在 prompt 中） |
| 检索结果 | 对话片段 + LLM 摘要 | 字段片段 + 对话片段，按来源标签分类 |

### 2.3 会话管理（参考 Hermes Agent）

#### 2.3.1 设计背景

InkFlow 的 Session 由 `novel_agent/agent/memory/session.py` 中的 `Session` 类管理，负责会话生命周期（start/advance_round/end）和 nudge 间隔判断。Session 结束时触发 `flush_short_memory()`，将短期缓冲提升到 MEMORY.md。

#### 2.3.2 Session 生命周期

```
前端打开新对话 → 生成 session_id → POST /api/chat (session_id)
  → Session.start()：记录 started_at，设置 is_active=True
  → 后续所有消息标记该 session_id

Session 进行中：
  → Session.advance_round()：轮次递增，nudge 间隔判断
  → should_compress() 触发 → 压缩 state.messages（见上下文设计.md §3）
  → Agent 自主 → memory_append → 写入 short_memory.md
  → 每 N 轮 → nudge（注入 system message 提醒 Agent 保存重要信息）

前端关闭对话 / 用户长时间无操作（>30min）：
  → Session.end(conversation_memory)
      → conversation_memory.flush_short_memory()：将 short_memory 提升到 MEMORY.md
      → 记录 ended_at，设置 is_active=False

下次打开新对话：
  → 加载新的 MEMORY.md 快照作为 slot [1] stable prefix
  → build_memory_context() 加载 short_memory + FTS5 检索结果
  → state.messages 保留（同一 session 内），新 session 由前端重新发起
```

#### 2.3.3 Session 结束与 MEMORY.md 的关系

Session 结束时通过 `Session.end(conversation_memory)` 触发 `flush_short_memory()`，将短期缓冲中的所有内容提升到 MEMORY.md：

- **这是 MEMORY.md 在 session 间的唯一更新时机**（除 rewrite 容量管理外）
- 短 session：short_memory 可能为空（Agent 未调用 `memory_append`），提升时跳过
- 长 session：short_memory 积累了 Agent 多次手动写入的事实

**MEMORY.md session 内冻结的意义**：slot [1] 的 MEMORY.md 快照在整个 session 内不变，KV cache 持续命中。所有新事实通过 short_memory（slot [5]）立即可见，session 结束时统一提升。

## 3. 小说记忆系统

小说记忆系统负责管理小说创作的结构化知识，确保角色、设定、伏笔等与章节内容保持同步。

### 3.1 字段体系

6 个字段文件 + 1 个章节索引 + 1 个元信息文件，构成小说的结构化知识图谱：

| 文件 | 职责 | `read_ch_field` | 更新触发 |
|------|------|-----------------|---------|
| settings.md | 世界观/力量体系/核心冲突 | `settings_read_ch` | 章节演进 |
| characters.md | 角色档案（核心7项/活跃4项/退场1行） | `characters_read_ch` | 角色行为变化 |
| relationships.md | 关系图谱（现状+演变转折点） | `relationships_read_ch` | 关系发展 |
| foreshadowing.md | 伏笔清单（5状态机+7类型） | `foreshadowing_read_ch` | 伏笔埋设/回收 |
| outline_historical.md | 历史大纲（已完成章节摘要） | `outline_historical_read_ch` | 章节完成 |
| outline_future.md | 未来大纲（规划中章节） | None（级联更新） | 其他字段更新后自动级联 |
| outline_structure.json | 章节索引（编号+标题+字数） | — | 章节增删 |
| meta.json | 元信息（进度/轮次） | — | 所有操作 |

### 3.2 字段更新机制

#### 3.2.1 问题

写新章节后，字段文件不会自动更新。检测机制基于 `read_ch_field`（已读到第几章），`total_chapters > field_read_ch` 时判定过期。

**受章节更新影响的字段**（5 个直接 + 1 个级联）：

| 字段 | `read_ch_field` | 影响方式 |
|------|----------------|---------|
| outline_historical | `outline_historical_read_ch` | 直接：读章节内容生成大纲 |
| settings | `settings_read_ch` | 直接：世界观/力量体系随章节演进 |
| characters | `characters_read_ch` | 直接：角色状态/能力随章节变化 |
| relationships | `relationships_read_ch` | 直接：人物关系随章节发展 |
| foreshadowing | `foreshadowing_read_ch` | 直接：伏笔埋设/回收/状态变更 |
| outline_future | `None` | 级联：通过 `_CASCADE_RULES` 在其他字段更新后自动级联 |

#### 3.2.2 触发机制

| 触发方式 | 时机 | 行为 |
|---------|------|------|
| 写新章节 | `handle_continue_writing` | 仅写章节正文 |
| 字段增量更新 | 用户主动调用 `update_outline_*` / `update_field` / `generate_*` | 按工具语义执行 |

> **已移除**：`_find_stale_fields` 过期检测与 `update_stale_fields` 一键同步（待记忆系统 v2）。

**确认行为**：用户确认 → 逐个调用 `handle_generate_field`，更新完成后 `field_read_ch` 设为最新章节索引；用户拒绝 → 跳过，下次触发时再次询问。

#### 3.2.3 增量更新流程

1. 读取 `field_read_ch` 之后的新增章节全文
2. 将"当前字段内容 + 新增章节内容"发送给 LLM，基于新增内容补充字段
3. 更新完成后，`save_field_content` 将 `field_read_ch` 设为最新章节索引

### 3.3 小说创作的记忆需求分析

#### 3.3.1 作家视角的核心需求

| # | 需求                   | 当前覆盖                                     |
| - | -------------------- | ---------------------------------------- |
| 1 | **故事梗概记忆**（各弧发生了什么）  | settings.md（卷级规划）+ outline_historical（已完成章节大纲）+ outline_structure（章节索引） |
| 2 | **角色关系记忆**（随时间变化）    | relationships.md（"关系现状"+"关系演变：关键转折点"） |
| 3 | **角色背景记忆**（经历如何塑造性格） | characters.md（核心角色 7 项：角色/身份/实力/性格/外貌/背景/动机） |
| 4 | **角色情感记忆**（关键事件的影响）  | 章节生成时注入最近 5 章全文，LLM 直接从正文感知 |
| 5 | **角色动机记忆**（各角色当前目标）  | characters.md 核心角色的"动机"字段 |
| 6 | **创作设想记忆**（防止前后矛盾）   | MEMORY.md（## 创作决策/## 故事状态/## 重要变更）+ rewrite_memory_md 去重去矛盾 |
| 7 | **设定记忆**（世界观规则一致性）   | settings.md（风格定位/核心冲突/世界观/力量体系/卷级规划） |
| 8 | **伏笔记忆**（埋了什么坑没填）    | foreshadowing.md（5 状态机 + 7 种类型 + 伏笔状态报告） |

8 条需求全部已被现有机制覆盖，当前三层记忆架构无需新增层级或优化。

#### 3.3.2 事实层级与衰减率

事实天然存在层级，当前系统已通过 fact_type 差异化衰减率实现：

```
前提级事实（推翻几乎不可能）— 世界观基本规则、主角核心身份
  → fact_type=identity, 衰减率 0.1（最慢衰减）
  └─ 弧级事实（推翻需重大事件）— 主角核心关系、当前情感基调
      → fact_type=decision, 衰减率 0.3
      └─ 章级事实（推翻需合理转折）— 本期目标、临时盟友
          → fact_type=temporal, 衰减率 1.0
          └─ 场景级事实（下一句话可改变）— 当前所在位置、持有物品
              → fact_type=state, 衰减率 2.0（最快衰减）
```

当前 source → fact_type 自动映射（field→identity, memory→decision, chat→temporal）已覆盖主要场景。

## 4. 主流 Agent 记忆架构对比

### 4.1 Claude Code

| 维度            | 设计                                                          | 可借鉴之处              |
| ------------- | ----------------------------------------------------------- | ------------------ |
| **架构**        | 4 层记忆栈（Global → Project → Local → Auto Memory）              | Layer 叠加不覆盖，优先级递增  |
| **MEMORY.md** | 纯索引文件（每行一条指针，上限 200 行/25KB），正文在专题文件中                        | InkFlow 也可改为索引路由模式 |
| **记忆分类**      | user / feedback / project / reference，强制 Why + How to apply | 创作决策应包含 Why + How  |
| **提取机制**      | Perfect Fork Agent：分叉代理共享 prompt 前缀，利用 KV cache             | 可借鉴到记忆提炼的 LLM 调用优化 |
| **整理机制**      | /dream 四阶段：定向→收集信号→合并→裁剪索引                                  | 记忆定期整理的工程化思路       |

### 4.2 OpenClaw

| 维度       | 设计                                    | 可借鉴之处               |
| -------- | ------------------------------------- | ------------------- |
| **架构**   | 文件优先 + 混合搜索（BM25 + Vector）            | Hybrid 检索对中文小说内容更友好 |
| **身份层**  | SOUL.md（身份+价值观）+ HEARTBEAT.md（自主任务清单） | 可引入"创作哲学"文件         |
| **心跳机制** | 定时主动检查待办任务                            | 对长周期自主记忆维护极有价值      |
| **记忆刷新** | 上下文压缩前触发内存持久化                         | 防止关键信息随窗口滑动丢失       |
| **日志策略** | daily log append-only 永久保留，不蒸馏不删除，通过 SQLite 向量索引按需搜索 | InkFlow 无 daily log，靠 chat.db 全量存档 + FTS5 按需检索替代 |

**关键差异**：OpenClaw 的 daily log 是永久保留的（append-only），因为没有 chat.db 级别的全量对话存档。InkFlow 有 chat.db 做全量归档 + FTS5 搜索，无需 daily log 中间层。

### 4.3 Hermes Agent

| 维度          | 设计                                                     | 可借鉴之处                                |
| ----------- | ------------------------------------------------------ | ------------------------------------ |
| **L1 冻结快照** | MEMORY.md (2200 字符) + USER.md (1375 字符)，硬上限保护 KV cache | InkFlow 已有 Stable Prefix，理念一致        |
| **L2 情景技能** | Skills 目录，按需加载                                         | 小说 Agent 的 system prompts 实现了类似分离    |
| **L3 会话归档** | SQLite 全量会话存档，`session_search` 冷召回，不删除不蒸馏               | InkFlow 的 chat.db 实现了同类机制            |
| **硬上限精妙性**  | 故意收窄容量迫使高质量筛选                                          | InkFlow 的 Stable Prefix hash 缓存是同类理念 |
| **nudge 机制** | `nudge_interval` 定期提醒 Agent 反思并保存重要内容                    | 已引入，每 N 轮注入 system message 提醒 Agent 保存                       |

**关键差异**：Hermes 的 MEMORY.md 在 session 内可变（Agent 直接写入），InkFlow 的 MEMORY.md 在 session 内冻结（所有写入走 short_memory，session 结束时统一提升）。这保证了 InkFlow 的 KV cache 在整个 session 内持续命中，而 Hermes 依赖 nudge 机制提醒 Agent 主动保存。

| 能力维度           | InkFlow 当前                   | Claude Code    | OpenClaw    | Hermes      |
| -------------- | ---------------------------- | -------------- | ----------- | ----------- |
