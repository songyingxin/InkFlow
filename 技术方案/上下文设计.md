## 1. 上下文组成

### 1.1 数据源总览

上下文由以下数据源组装而成，各数据源的详细管理机制见 [记忆系统设计.md](记忆系统设计.md)：

| 数据源 | 内容 | 生命周期 | 上下文注入方式 | 管理模块 |
|--------|------|---------|--------------|---------|
| agents.md | 创作共识（跨书不变的创作哲学） | 极低（手动更新） | stable 层全量注入 | 模板加载 |
| MEMORY.md | 提升后的持久事实 | 永久（session 内冻结） | stable 层冻结快照 | ConversationMemory |
| lead-router.md | Harness 路由规则 + Plan 生成规则 | 极低（手动更新） | stable 层全量注入 | 模板加载 |
| 小说元状态 | 书名/进度/字段状态 | 中等（随创作推进） | stable 层全量注入 | PromptBuilder |
| nudge 提醒 | 记忆维护提醒 | 每轮变化 | volatile 层条件注入 | Session |
| short_memory.md | Agent 手动写入的事实 | 临时（session 结束提升后清空） | volatile 层全量注入 | ConversationMemory |
| FTS5 检索结果 | 相关记忆片段 | 每轮变化 | volatile 层按需检索 | MemorySearch |
| 反思/Plan | 执行状态 | 每轮变化 | volatile 层条件注入 | PromptBuilder |
| state.messages | 对话历史 | 每轮增长 | volatile 层直接承载 | AgentLoop |
| Subagent prompt | 专业化 system prompt | 极低（手动更新） | stable 层全量注入 | 模板加载 |
| memory_guide | 记忆操作指南 | 极低（手动更新） | volatile 层条件注入 | 模板加载 |

### 1.2 Stable / Volatile 划分

InkFlow 的 Prompt 构建采用 stable/volatile 分离设计，对齐 prompt caching 的 prefix matching 机制：

| 层 | 内容 | 变化频率 | Cache 命中率 |
|---|------|---------|-------------|
| **stable** | agents.md + MEMORY.md 冻结快照 + 路由模板/专业化 prompt + 小说元状态 | 极低（跨 session / 同类任务共享） | 最高 |
| **volatile** | nudge / short_memory / FTS5 检索 / 反思/Plan / 对话历史 / 用户消息 | 每轮变化 | 最低 |

消息排列顺序（最大化 prefix cache 命中）：

```
[stable]   agents.md + MEMORY.md + 路由模板/专业化 prompt + 小说元状态  ← 跨调用共享
[volatile] nudge + short_memory + FTS5 检索 + 反思/Plan              ← 每轮变化
[volatile] 对话历史 + 用户消息                                         ← 每次不同
```

### 1.3 记忆子系统接口

上下文组成通过以下接口消费记忆系统的数据（实现细节见 [记忆系统设计.md](记忆系统设计.md)）：

| 接口 | 返回内容 | 对应 slot | 代码位置 |
|------|---------|----------|---------|
| `build_stable_prefix(state)` | MEMORY.md 冻结快照 | [1] | ConversationMemory |
| `build_memory_context(state, current_query)` | short_memory 全量 + FTS5 双源检索结果 | [5] [6] | ConversationMemory |

> 核心约束：MEMORY.md 和 short_memory.md 已直接注入 prompt，FTS5 不检索它们（避免重复）。

### 1.4 核心代码映射

| 模块 | 文件 | 职责 |
|------|------|------|
| Prompt 组装 | `agent/prompt_builder.py` | stable/volatile 消息构建，统一入口 |
| 对话记忆 | `agent/memory/conversation/conversation.py` | L1/L2/L3 读写 + 上下文构建接口 |
| 小说记忆 | `agent/memory/novel/novel.py` | 6 字段文件 + 章节 + 元信息 |
| FTS5 索引 | `agent/memory/search.py` | Trigram 召回 + jieba 重排 + 时间衰减 |
| 上下文压缩 | `agent/runtime/compression.py` | LLM 摘要压缩 + 记忆预提取 |
| Session 管理 | `agent/memory/conversation/session.py` | 生命周期 + nudge + flush |
| 记忆更新 | `agent/memory/update.py` | MEMORY.md 重写 + 字段整合 |

## 2. 上下文组装

### 2.1 Lead Agent（Harness 模式）

```
build_lead_messages(state)
├── [0] system: agents.md（创作共识）                              ← stable
├── [1] system: MEMORY.md 冻结快照                                ← stable
├── [2] system: lead-router.md（含路由规则）                        ← stable
├── [3] system: 小说元状态（书名/进度/字段状态）                      ← stable
├── [4] system: nudge 提醒（每 5 轮注入）                           ← volatile
├── [5] system: 短期缓冲 short_memory.md                           ← volatile
├── [6] system: FTS5 双源检索结果（MemoryIndex + chat.db）          ← volatile
├── [7] system: 反思 / Plan 状态                                   ← volatile
├── [8]        : state.messages（first_user + 压缩摘要 + 最近 20 条）← volatile
└── [9]        : 当前用户消息                                       ← volatile
```

**FTS5 双源检索**（slot [6]）：
- 索引源 1：MemoryIndex（memory_index.db）— 6 个字段文件（设定/角色/关系/伏笔/大纲）
- 索引源 2：chat.db FTS5 — 全量对话记录
- 不检索：MEMORY.md（已在 slot [1]）、short_memory.md（已在 slot [5]）
- 查询词：`current_query`（从最近一条 user 消息提取）

**小说元状态**（slot [3]）格式：

```
书名：{title}
总章数：{total_chapters}
设定：{有/无} | 大纲：{有/无} | 角色：{有/无} | 伏笔：{有/无}
```

### 2.2 Subagent（Creator / Editor / Reader）

```
build_subagent_messages(task, config)
├── [0] system: agents.md（创作共识）                              ← stable
├── [1] system: MEMORY.md 冻结快照                                ← stable
├── [2] system: Subagent 专业化 prompt（creator.md / editor.md / reader.md）← stable
├── [3] system: 短期缓冲 short_memory.md                           ← volatile
├── [4] system: FTS5 双源检索结果（query=task）                     ← volatile
└── [5] user : task 描述                                          ← volatile
```

与 Lead Agent 的差异：
- 无小说元状态（slot [3]）— 任务描述中已包含必要信息，Subagent 不做路由决策
- 无 nudge（slot [4]）— Subagent 是单次执行，无需周期性记忆提醒
- 无反思/Plan（slot [7]）— Subagent 无内部执行状态
- 无对话历史（slot [8]）— Subagent 每次调用独立
- FTS5 检索 query 基于 task 而非 current_query
- 有条件注入 memory_guide（仅 Creator/Editor 有记忆工具时）

### 2.3 字段/章节生成

```
build_generation_messages(state, system_msg, user_msg)
├── [stable]   agents.md
├── [stable]   MEMORY.md 冻结快照
├── [stable]   任务模板 system prompt（fields/*.md / chapter_*.md）
├── [volatile] 短期缓冲 short_memory.md
├── [volatile] FTS5 双源检索结果
└── [volatile] user 消息（章节内容 + 请求）
```

字段/章节生成属于 Subagent 调用，不注入小说元状态。

## 3. 上下文压缩

InkFlow 的上下文压缩分为两条独立路径：**Lead Agent 对话压缩** 和 **Subagent 结果压缩**。两者的目标、触发条件、参数和流程完全不同。

### 3.1 Lead Agent 对话压缩

当 Lead Agent 的对话 token 超过阈值时，`MessageCompressor`（`agent/runtime/compression.py`）压缩对话历史。

**目标**：释放上下文窗口空间，让 Lead Agent 能继续对话。

```
compact_messages(messages, novel_state)
│
├── 1. Token 估算
│   └── estimate_tokens: CJK 字符 × 1.5 + ASCII 字符 ÷ 4
│
├── 2. 触发判断
│   ├── 有 context_window: estimated > window × 0.5 → 压缩
│   └── 无 context_window: messages > 40 条 → 压缩
│
└── 3. compact_by_summary
    ├── a. 分割点计算
    │   ├── split_point = len(messages) - 20（保留最近 20 条）
    │   ├── 不切断 tool_call ↔ tool_result 对（find_message_pair_boundary）
    │   └── 保证首条 user 消息不被归入旧消息
    │
    ├── b. 旧消息文本化
    │   ├── 每条消息截取前 compression_msg_chars（150）字符
    │   └── tool_calls 仅保留函数名列表
    │
    ├── c. 预压缩记忆刷新（第 1 次 LLM 调用）
    │   ├── 使用 COMPRESSION_MODEL（deepseek-v4-flash）
    │   ├── 输入截取前 compression_input_chars（2000）字符
    │   ├── 从旧消息中提取关键事实
    │   ├── 结果追加到 short_memory.md（非 MEMORY.md，session 结束时才提升）
    │   └── 无事实可提取时跳过
    │
    ├── d. 生成摘要（第 2 次 LLM 调用）
    │   ├── 使用 COMPRESSION_MODEL
    │   ├── 输入截取前 summary_input_max_chars（3000）字符
    │   └── 输出替换旧消息为 [历史摘要] system 消息
    │
    └── e. 消息重组
        ├── 首条 user 消息（保留）
        ├── [历史摘要] system 消息
        └── 最近 20 条消息（过滤孤立 tool_result）
```

> **与记忆系统设计.md 的差异**：记忆系统设计.md §2.4 描述了 4-Phase 压缩算法（边界→修剪→7 段结构化摘要→重组），当前代码实现是 3 步（文本化→记忆提取→简单摘要），尚未实现 Phase 2（旧 tool_result 修剪为"[旧工具输出已清除]"）和 7 段结构化摘要格式。详见 [记忆系统设计.md](记忆系统设计.md) §2.4。

**关键参数**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `context_window` | 由模型决定 | 上下文窗口大小 |
| `compact_threshold_ratio` | 0.5 | token 使用量达到窗口 50% 时触发 |
| `keep_recent_messages` | 20 | 保留最近 N 条消息不被压缩 |
| `max_messages_before_compact` | 40 | 无 context_window 时的消息数阈值 |
| `compression_msg_chars` | 150 | 每条消息在文本化时的截取长度 |
| `compression_input_chars` | 2000 | 记忆提取输入的最大字符数 |
| `summary_input_max_chars` | 3000 | 摘要生成输入的最大字符数 |
| `COMPRESSION_MODEL` | deepseek-v4-flash | 压缩使用的模型 |

### 3.2 Subagent 结果压缩

Subagent 完成任务后，将完整的执行记录压缩为简短结果摘要，返回给 Lead Agent。

**目标**：控制注入 Lead Agent 上下文的 Subagent 结果体积，避免单次 Subagent 调用占用过多 token。

```
_compress_result(task, messages, called_tools, tool_results)
│
├── 1. 无工具调用 → 取最后一条 assistant 回复（截取前 subagent_summary_chars 500 字符）
│
├── 2. 有工具调用 + 结果总长 < 800 → 直接拼接
│   ├── 格式：调用了{工具名}。{各结果截取前 subagent_tool_result_chars 200 字符}
│   └── 无 LLM 调用
│
└── 3. 有工具调用 + 结果总长 ≥ 800 → LLM 压缩摘要
    ├── 执行记录文本化
    │   ├── 仅保留 assistant + tool 消息
    │   └── 每条消息截取前 subagent_compress_msg_chars（300）字符
    ├── LLM 生成结果摘要（1 次 API 调用）
    │   ├── 使用 COMPRESSION_MODEL
    │   ├── 输入：任务描述 + 执行记录（截取前 subagent_compress_total_chars 3000 字符）
    │   └── 输出：结构化结果摘要（截取前 subagent_compress_result_chars 500 字符）
    └── 异常兜底：返回"调用了{工具名}完成任务"
```

**关键参数**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `subagent_summary_chars` | 500 | 无工具调用时，assistant 回复的最大截取长度 |
| `subagent_tool_result_chars` | 200 | 短结果拼接时，每条工具结果的截取长度 |
| `subagent_compress_msg_chars` | 300 | LLM 压缩时，每条消息的截取长度 |
| `subagent_compress_total_chars` | 3000 | LLM 压缩输入的最大字符数 |
| `subagent_compress_result_chars` | 500 | LLM 压缩结果的最大字符数 |
| `COMPRESSION_MODEL` | deepseek-v4-flash | 压缩使用的模型 |

### 3.3 两种压缩的对比

| 维度 | Lead Agent 对话压缩 | Subagent 结果压缩 |
|------|-------------------|-----------------|
| **触发时机** | 对话 token 超阈值时 | Subagent 每次完成任务后 |
| **压缩对象** | Lead Agent 的对话历史 | Subagent 的完整执行记录 |
| **压缩目标** | 释放上下文窗口空间 | 控制注入 Lead Agent 的结果体积 |
| **LLM 调用次数** | 2 次（记忆提取 + 摘要生成） | 0-1 次（短结果直接拼接，长结果 LLM 压缩） |
| **记忆提取** | ✅ 提取关键事实写入 short_memory.md | ❌ 无（结果摘要直接返回 Lead Agent） |
| **保留策略** | 保留最近 20 条 + 首条 user 消息 | 短结果保留原文截取，长结果压缩为摘要 |
| **结果格式** | [历史摘要] system 消息替换旧消息 | 纯文本摘要作为 tool_result 返回 |
| **对 Lead Agent 的影响** | 直接修改 state.messages | 通过 tool_result 注入对话 |

### 3.4 三方压缩方案对比

#### 3.4.1 Claude Code：五层渐进式压缩管线

核心思想：**最便宜的优先，最重的最后**。

**Lead Agent 压缩**：

| 层级 | 策略 | 成本 | 机制 |
|------|------|------|------|
| Level 1 | Tool Result Budget | 零 | 超过 50K 字符的工具结果持久化到磁盘，仅保留 2KB 预览 |
| Level 2 | History Snip | 零 | 特性门控的 token 释放，裁剪旧消息 |
| Level 3 | Microcompact | 零 API 调用 | 基于时间的裁剪 OR cache_editing（删除服务端缓存中的旧工具结果） |
| Level 4 | Context Collapse | 零（非破坏性） | 投影式折叠，约 90% 压缩率，可恢复 |
| Level 5 | Autocompact | 一次 API 调用 | Fork 子 agent 生成完整摘要，不可逆 |

**Subagent 压缩**：独立上下文窗口，完成后通过 handoff message 返回结构化摘要。

#### 3.4.2 OpenClaw：三级 KV Cache + MemCompressAgent

核心思想：**KV Cache 分级管理 + 语义去重**。

| 级别 | 存储 | 内容 | 容量 | 压缩方式 |
|------|------|------|------|---------|
| L1 热缓存 | GPU 显存 | 最近 8 轮完整对话 | 4096 token | 无损保留 |
| L2 摘要缓存 | 半常驻显存 | 结构化摘要 KV | 3×512 token | LLM 摘要 |
| L3 持久化缓存 | 内存/磁盘 | 向量嵌入 + 摘要 KV | 无限制 | 向量去重（98% 准确率） |

MemCompressAgent 后台运行，将旧对话转为结构化摘要 + 向量去重。CacheBlend 支持非前缀 KV Cache 复用。

#### 3.4.3 Hermes Agent：双层压缩 + 零上下文 Subagent

核心思想：**保护头尾、摘要中间**。

| 层 | 触发时机 | 阈值 | 机制 |
|---|---------|------|------|
| Gateway Session Hygiene | Agent 处理消息前 | 85% 上下文窗口 | 安全网 |
| Agent ContextCompressor | Agent 工具循环内 | 50% 上下文窗口 | 主压缩系统 |

Subagent 完全空白上下文（零对话历史、零记忆），仅通过 goal + context 接收任务信息。

### 3.5 压缩策略对比总表

| 维度 | Claude Code | OpenClaw | Hermes Agent | InkFlow |
|------|------------|----------|-------------|---------|
| **Lead 压缩层级** | 5 层渐进式 | 3 级 KV Cache | 双层（Gateway 85% + Agent 50%） | LLM 摘要（50% 阈值） |
| **Lead 零成本压缩** | 3 层（Budget/Snip/Microcompact） | L1 热缓存无损保留 | 无 | 无 |
| **Lead 有损压缩** | Autocompact（LLM 摘要） | L2 摘要 + L3 向量去重 | 摘要中间、保护头尾 | LLM 摘要 + 记忆预提取 |
| **Lead 压缩成本** | 0→0→0→0→1 API 调用 | 后台 agent 持续运行 | 1 API 调用 | 2 次 API 调用 |
| **Sub 上下文继承** | 独立上下文窗口 | 独立 session | 零上下文（仅 goal+context） | stable + volatile |
| **Sub 压缩方式** | 无需压缩（独立窗口） | 无需压缩（独立 session） | 无需压缩（零上下文） | 三级（直接取/拼接/LLM 压缩） |
| **Sub→Lead 结果传递** | handoff message | 摘要返回 | 结构化摘要 | tool_result 摘要 |
| **Sub→Lead 结果体积** | 可控（结构化摘要） | 可控 | 可控 | ≤500 字符 |
| **预压缩记忆** | Session Memory 后台维护 | MemCompressAgent | 无 | Lead: 提取事实到 short_memory / Sub: 无 |
| **可恢复性** | Level 4 可恢复，Level 5 不可逆 | L3 持久化可检索 | FTS5 冷检索可找回 | chat.db + FTS5 可检索 |

### 3.6 压缩优化建议

> 从长篇小说作者和长篇小说 Agent 的双重视角审视压缩设计。长篇小说的核心矛盾是：**上下文窗口有限，但叙事一致性要求无限**。

#### 3.6.1 ✅ 需要：零成本 Tool Result 裁剪

**现状**：压缩时对旧消息一刀切——要么保留原文（截取前 N 字符），要么完全替换为摘要。tool_result 可能包含大量章节内容（5K-10K 字符），在文本化时占用大量空间。

**建议**：在文本化阶段（步骤 b），对 tool_result 按类型差异化截取：

```python
TOOL_RESULT_BUDGET = {
    "read_chapter": 500,      # 已读章节保留摘要即可
    "search_memory": 800,     # 检索结果保留更多
    "update_field": 200,      # 更新确认极短
    "default": 1000,
}

for msg in old_messages:
    if msg.get("role") == "tool":
        tool_name = _extract_tool_name(msg, messages)
        budget = TOOL_RESULT_BUDGET.get(tool_name, TOOL_RESULT_BUDGET["default"])
        old_text += f"[tool:{tool_name}] {content[:budget]}\n"
```

**收益**：零 API 调用成本，减少文本化阶段的 token 浪费，让 summary_input_max_chars 的额度留给更有价值的对话内容。

**不影响语义保真**：tool_result 的详细内容已通过步骤 c（预压缩记忆刷新）提取为关键事实存入 short_memory.md，裁剪的是冗余原文而非信息。

#### 3.6.2 ⚠️ 不需要：压缩可恢复性——架构已支持，无需额外机制

**现状**：Lead Agent 压缩后旧消息被 [历史摘要] 替换，从 `state.messages` 中移除。但旧消息已通过 `save_chat_message` 存入 chat.db（见 [记忆系统设计.md](记忆系统设计.md) §2.1），FTS5 可检索。压缩前还提取了关键事实到 short_memory.md。

**曾考虑的方案**：在压缩摘要中嵌入"检索线索"，让 Agent 知道可以搜索什么。

**不需要的理由**：

1. **架构已提供可恢复性**：旧消息在 chat.db（L1），关键事实在 short_memory.md（L2），Agent 有 `search_memory` 工具。压缩是"从热记忆降级为冷记忆"，不是"永久丢失"。

2. **检索线索信噪比极低**：每次压缩追加 200-500 字线索，10 次压缩累积 2000-5000 字。绝大多数线索永远不会被用到，反而浪费上下文窗口。

3. **Agent 可从问题推导搜索词**：用户问"第 5 章那个角色说了什么"时，Agent 自然从问题中提取搜索词，不需要预计算的线索。

4. **"不知道该搜索"是 prompt 问题，不是架构问题**：应在 system prompt 中加引导，而非在摘要中塞入结构化线索。

**三方对比**：

| 系统 | 可恢复机制 | InkFlow 对应 |
|------|-----------|-------------|
| Claude Code Level 4 | Context Collapse 投影式折叠 | 不适用（API 模式无法控制 KV Cache） |
| OpenClaw L3 | 向量嵌入 + 摘要 KV 持久化 | 不适用（需要推理框架支持） |
| Hermes Agent | FTS5 + ChromaDB 冷检索 | **已有**：chat.db + MemoryIndex + search_memory |

#### 3.6.3 ⚠️ 不需要：压缩触发阈值分层——延迟压缩 ≠ 保留更多信息

**现状**：单一阈值 `compact_threshold_ratio = 0.5`，token 使用量达到窗口 50% 就触发压缩。

**曾考虑的方案**：按叙事阶段动态调整阈值——设定阶段 0.65、续写阶段 0.50、收束阶段 0.80。

**不需要的理由**：

1. **阈值控制"何时压缩"，不控制"压缩时保留什么"**。无论阈值 0.50 还是 0.80，压缩算法不变——同样提取事实、同样生成摘要。

2. **设定阶段提高阈值适得其反**。工具调用密集 = tool_result 体积大 = 恰恰是最需要压缩的场景。

3. **收束阶段提高阈值无法保护早期伏笔**。第 47 章回溯第 3 章的伏笔时，第 3 章的细节已历经 8-15 轮压缩。伏笔回溯的正确方案是 search_memory，不是更大的窗口。

4. **三方借鉴不成立**：Hermes 双层阈值是两个不同 Agent，Claude Code 多层触发是不同压缩方法，OpenClaw cache-ttl 基于时间衰减。

5. **记忆系统已是长程回溯的答案**。L1/L2/L3 + search_memory 构成了完整的长程回溯链路。

#### 3.6.4 ⚠️ 不需要：Subagent 上下文继承优化——当前继承已是按需的

**现状**：InkFlow 的 Subagent 继承了 stable + volatile 上下文（见 §2.2），包括 agents.md、MEMORY.md 冻结快照、专业化 prompt、short_memory + FTS5 检索结果。

**曾考虑的方案**：按角色差异化上下文继承——Reader 去掉 MEMORY.md/short_memory/FTS5，Editor 去掉 short_memory。

**不需要的理由**：

1. **Reader 必须有 MEMORY.md**。检查一致性必须知道已建立的设定。
2. **Reader 必须有 FTS5 检索**。有 `search_memory` 工具，需要检索上下文。
3. **Editor 必须有 short_memory**。有 `memory_append/memory_rewrite` 工具，需要理解当前记忆状态。
4. **"30-50% 窗口空间"严重高估**。实际仅 1300-1800 token / 128K ≈ 1-1.5%。
5. **FTS5 query=task 已在代码中实现**。
6. **三方对比不成立**。创意任务 vs 窄范围任务的本质差异。

#### 3.6.5 ⚠️ 不需要：渐进式压缩管线

Claude Code 的 5 层渐进式管线面向代码场景设计，零成本层在小说场景中风险过高——裁剪旧消息可能丢失角色动机线索，删除工具结果可能遗漏关键设定。InkFlow 的"单一但安全"策略（LLM 摘要 + 记忆预提取）是语义保真优先的正确选择。

#### 3.6.6 ⚠️ 不需要：KV Cache 分级管理

OpenClaw 的三级 KV Cache 需要推理框架层面支持，InkFlow 使用 API 调用模式，无法控制服务端 KV Cache 分级。且 InkFlow 不存在检索文档位置漂移问题，prefix matching 已足够。

## 4. 上下文缓存

### 4.1 Stable Prefix 冻结策略

**核心原则**：MEMORY.md 在 session 内冻结，保证 stable prefix 的 KV cache 持续命中。

```
Stable Prefix（build_stable_prefix()）
  ├── 内容：MEMORY.md 冻结快照
  ├── 位置：system prompt 最前面（slot [1]）
  ├── 缓存：基于 MEMORY.md 的 SHA-256 hash，内容不变则前缀不变
  └── 生命周期：整个 session 内不变

Dynamic Context（build_memory_context()）
  ├── 短期缓冲层：short_memory.md → slot [5]
  ├── 语义层：FTS5 检索结果 → slot [6]
  ├── 缓存 Key = (title, round_count, session_id, current_query[:50])
  └── TTL 可配置
```

**与 Hermes 的对齐**：Hermes 的 MEMORY.md 和 USER.md 在 session 开始时作为 frozen snapshot 注入 system prompt，session 内不修改。InkFlow 采用相同策略，但写入走 short_memory.md（session 结束时统一提升到 MEMORY.md），而非 Hermes 的直接写入 MEMORY.md。

### 4.2 Cache 边界标记

**现状**：未使用 `cache_control` 显式标记 cache 断点，依赖 API 自动 prefix matching。

**问题**：stable 层跨所有调用共享且内容稳定，是天然的 cache 候选，但缺少显式标记意味着 API 可能无法精确识别缓存边界。

**建议**：添加 `cache_control` 标记（P0 优先级，零成本改动）：

```python
# stable 层最后一条消息标记 cache_control
messages[-1]["cache_control"] = {"type": "ephemeral"}
```

### 4.3 Prefix 稳定性保障

| 策略 | 说明 |
|------|------|
| MEMORY.md session 内冻结 | 所有写入走 short_memory，session 结束时提升 |
| 工具集按角色固定 | Creator/Editor/Reader 各有独立工具列表，session 中途不变 |
| 动态信息放 volatile 层 | 小说状态、nudge、检索结果等在 volatile 层，不破坏 prefix |
| context 层位置靠后 | FTS5 检索结果放在 volatile 之后，stable + volatile 构成长 prefix 可被完整缓存 |

### 4.4 三方 Cache 策略对比

| 策略 | Claude Code | OpenClaw | Hermes Agent | InkFlow |
|------|------------|----------|-------------|---------|
| **Prefix 稳定性** | 极高（工具集不变、system prompt 不变） | 中（检索文档位置漂移） | 高（冻结快照） | 高（MEMORY.md 冻结） |
| **非前缀复用** | 无 | CacheBlend | 无 | 无 |
| **Cache 边界标记** | cache_control 显式标记 | cacheRetention 配置 | 无 | 无（待添加） |
| **中途变更处理** | system-reminder 注入而非修改 prefix | contextPruning.mode | Session 级隔离 | 无特殊处理 |
| **跨 Agent 复用** | Subagent 通过 handoff message | KVCOMM 跨 agent KV 通信 | 无 | 无 |

## 5. 第三方系统上下文设计分析

### 5.1 Claude Code

Claude Code 是 Anthropic 的 agentic coding tool，其上下文设计围绕 **prompt caching** 展开，将"cache 命中率"视为核心运维指标。

#### 5.1.1 Prompt 组装顺序

```
Layer 1: System Prompt + Tool Definitions    ← 全局缓存，跨 session
Layer 2: CLAUDE.md + Project Context         ← 项目级缓存
Layer 3: Session Context                     ← session 内缓存
Layer 4: Conversation Messages               ← 每轮增长
```

核心原则：**静态在前，动态在后**。

#### 5.1.2 五层压缩管线

| 层级 | 策略 | 成本 | 机制 |
|------|------|------|------|
| Level 1 | Tool Result Budget | 零 | 超过 50K 字符的工具结果持久化到磁盘，仅保留 2KB 预览 |
| Level 2 | History Snip | 零 | 特性门控的 token 释放，裁剪旧消息 |
| Level 3 | Microcompact | 零 API 调用 | 基于时间的裁剪 OR cache_editing |
| Level 4 | Context Collapse | 零（非破坏性） | 投影式折叠，约 90% 压缩率 |
| Level 5 | Autocompact | 一次 API 调用 | Fork 子 agent 生成完整摘要 |

#### 5.1.3 Cache 安全设计

- **不中途切换模型**：每个模型有独立缓存，切换 = 重建整个 prefix
- **不中途增删工具**：工具定义是 prefix 的一部分，变更 = 缓存失效
- **Plan Mode 设计为工具调用**：避免破坏 prefix
- **ToolSearch 延迟加载**：MCP 工具使用轻量 stub，完整 schema 按需加载
- **system-reminder 注入更新**：文件变更等信息通过 `<system-reminder>` 标签注入到下一条 user 消息，而非修改 system prompt

#### 5.1.4 Session Memory

后台进程持续维护结构化 markdown 笔记。压缩时直接使用这些笔记作为摘要，无需额外 LLM 调用。

#### 5.1.5 MCP（Model Context Protocol）

标准化通信协议，连接模型与外部工具。MCP 工具通过 ToolSearch 延迟加载，保证 prefix 稳定。

---

### 5.2 OpenClaw

OpenClaw 是开源 agentic 框架，编排多步 AI 工作流。核心挑战是：**检索文档跨轮次位置漂移导致 prefix cache 失效**。

#### 5.2.1 Prompt 结构与 Cache 问题

```
System Instructions        ← 稳定，prefix cache 有效
Conversation History       ← 逐步增长，prefix cache 有效
Retrieved Documents        ← 位置漂移，prefix cache 失效！
Tool Outputs               ← 位置漂移
User Input                 ← 每轮变化
```

#### 5.2.2 CacheBlend：非前缀 KV Cache 复用

- 识别并复用任意位置的已计算 KV cache 块，不要求 prefix 匹配
- 选择性重计算少量 token 的 KV 值，部分更新复用的 cache 块
- 通过锚点（anchor）机制校正 Attention 依赖偏移

#### 5.2.3 cacheRetention 配置

| 配置 | 说明 |
|------|------|
| `cacheRetention: "session"` | session 内缓存（默认） |
| `cacheRetention: "project"` | 项目级缓存，跨 session |
| `contextPruning.mode: "cache-ttl"` | 基于 TTL 的上下文裁剪 |
| Heartbeat keep-warm | 心跳保活，防止 5 分钟 TTL 过期 |

#### 5.2.4 三级 KV Cache

| 级别 | 存储 | 内容 | 容量 | 淘汰策略 |
|------|------|------|------|---------|
| L1 热缓存 | GPU 显存 | 最近 8 轮完整对话 | 4096 token | FIFO |
| L2 摘要缓存 | 半常驻显存 | 结构化摘要 KV | 3×512 token | LRU |
| L3 持久化缓存 | 内存/磁盘 | 向量嵌入 + 摘要 KV | 无限制 | 相关性检索 |

---

### 5.3 Hermes Agent

Hermes Agent 是 Nous Research 的自进化 AI Agent 框架，核心特色是 **三层记忆 + 闭环学习循环**。

#### 5.3.1 三层记忆架构

| 层 | 类型 | 存储 | 容量限制 | 生命周期 |
|---|------|------|---------|---------|
| L1 | 热记忆（工作记忆） | MEMORY.md + USER.md | 2,200 + 1,375 字符 | 永久（session 内冻结快照） |
| L2 | 外部记忆 | Honcho 语义搜索 / 8+ provider 插件 | 无硬限制 | 永久 |
| L3 | 冷检索 | SQLite FTS5 + ChromaDB 向量存储 | 无限制 | 永久 |

#### 5.3.2 冻结快照模式

MEMORY.md 和 USER.md 在 session 开始时作为 frozen snapshot 注入 system prompt，session 内不修改。记忆更新发生在 session 结束时（flush）。

#### 5.3.3 Skill 系统（程序化记忆）

三级渐进式加载策略，优化 token 使用：

| 级别 | 加载内容 | Token 消耗 |
|------|---------|-----------|
| Level 1 | 技能名称 + 简要描述 | ~20 tokens |
| Level 2 | 详细描述 + 参数说明 | ~200 tokens |
| Level 3 | 完整执行步骤 + 工具调用序列 | ~1000+ tokens |

#### 5.3.4 闭环学习循环

```
观察（Observe）→ 执行（Execute）→ 反思（Reflect）→ 沉淀（Crystallize）→ 复用（Reuse）
```

#### 5.3.5 FTS5 + ChromaDB 双检索

- **FTS5**：全文本搜索，覆盖所有 session，按需用 Gemini Flash 摘要
- **ChromaDB**：向量语义搜索，索引每次任务执行的时序记录
- 双检索互补：FTS5 精确匹配 + ChromaDB 语义相似

#### 5.3.6 记忆卫生原则

- 好记忆是声明式的："项目通过 GitHub push 部署"
- 坏记忆是任务日志："今天修了 bug X"
- 定期清理：移除过期路径、将过程性笔记转为声明性事实、将过程移入 Skill

## 7. 设计权衡分析

以下条目看似不足，实为小说创作场景的刻意设计权衡：

### 7.1 Subagent 与 Lead 的 stable 层"重复"

每次调用都重建 stable prefix（agents.md + MEMORY.md），看似浪费，实则是 **API 层 cache 复用的前提**：各 Subagent 是独立 API 调用，stable prefix 完全一致即可命中 API provider 的 prefix matching。

### 7.2 context 层位置靠后

FTS5 检索结果放在 volatile 之后，看似 prefix 匹配效率低，实则是 **最大化 prefix 长度的正确选择**：context 每轮变化，若放在 volatile 之前，则 volatile 层也无法被缓存；放在之后，stable + volatile 构成的长 prefix 可被完整缓存。

### 7.3 字段内容全量注入

章节生成时 settings/characters/relationships 等全量拼入 system prompt，看似冗余，实则是 **小说创作一致性保障的必要代价**：小说角色、设定、关系是强关联的整体，选择性检索可能遗漏关键细节导致人设崩塌或情节矛盾。全量注入的字段文件总量通常在 3K-8K token，在可接受范围内。

### 7.4 MEMORY.md 硬字符上限

仅按字符数限制而非信息密度优化，看似粗糙，实则是 **可预测性优先**：硬限制让行为可预测、可调试。当前方案通过 LLM 重写（非简单截断）来保信息，已兼顾了信息保留。

### 7.5 压缩策略"单一"

当前仅有 LLM 摘要压缩，看似缺少渐进式管线，实则是 **语义保真优先**：小说创作对上下文连贯性极度敏感，一次不当裁剪就可能导致人设矛盾或情节断裂。

### 7.6 不需要的优化

| 建议 | 不需要的理由 |
|------|------------|
| **system-reminder 动态更新** | 动态信息已在 volatile/context 层，stable 层仅含 agents.md + MEMORY.md，不存在 prefix 被动态更新破坏的场景 |
| **工具集稳定性** | 工具集已按角色固定，不存在 session 中途增删工具的场景 |
| **Microcompact（零成本压缩）** | 裁剪旧消息可能丢失角色动机线索，截断工具结果可能遗漏关键设定 |
| **Session Memory 后台维护** | 当前压缩流程已在做记忆预提取，后台维护反增 LLM 调用成本 |
| **字段检索式注入** | 字段全量注入是小说一致性保障的必要代价，6 个核心字段总量 3K-8K token |
| **CacheBlend 非前缀复用** | InkFlow 不存在检索文档位置漂移问题，且 CacheBlend 需要推理框架层面支持 |
| **Skill 系统** | Lead Agent 路由规则已是显式流程定义，创作流程每本书都不同不可复用 |
| **MEMORY.md 信息密度优化** | 硬字符上限是可预测性优先的刻意选择，LLM 重写已兼顾信息保留 |

## 8. 优化建议与路线图

### 8.1 ✅ 需要

#### 8.1.1 显式 cache_control 标记

**现状**：未使用 `cache_control` 显式标记 cache 断点，依赖 API 自动 prefix matching。

**必要性**：stable 层跨所有调用共享且内容稳定，是天然的 cache 候选。添加 `cache_control` 是零成本改动，收益明确。

#### 8.1.2 事实类型细粒度标注

**现状**：已有五级事实类型衰减率（identity/decision/preference/temporal/state），但 `_SOURCE_FACT_TYPE_MAP` 将所有 field 来源统一标记为 identity。

**必要性**：字段文件内部内容差异很大——世界观规则确实应标记为 identity，但角色关系变化更适合 decision，当前场景状态更适合 state。更细粒度的事实类型标注能提升 FTS5 检索的时间衰减精度，让检索结果更贴合查询意图。

### 8.2 优化路线图

| 阶段 | 优化项 | 预期收益 | 依赖 | 对应章节 |
|------|--------|---------|------|---------|
| P0 | cache_control 标记 | 零成本提升 cache 命中率 | API 支持 cache_control | 4.2 |

## 9. 总结

InkFlow 的上下文设计已在 stable/volatile 分离、MEMORY.md 冻结快照、FTS5 + jieba 检索等方面建立了良好基础，与 Hermes Agent 的设计理念高度一致。经过对优化建议的深度审视，大部分看似"不足"的设计实为小说创作场景的刻意权衡：

- **stable 层重复** → API cache 命中的前提
- **context 靠后** → 最大化 prefix 长度的正确选择
- **字段全量注入** → 小说一致性保障的必要代价
- **压缩策略单一** → 语义保真优先于压缩成本
- **硬字符上限** → 可预测性优先

### 9.1 已解决的矛盾：压缩不可逆 vs 叙事回溯刚需

压缩后的摘要确实丢失了细节，但 InkFlow 的三层记忆架构已提供可恢复性：旧消息在 chat.db（L1，FTS5 可检索），关键事实在 short_memory.md（L2），Agent 有 search_memory 工具。压缩是"从热记忆降级为冷记忆"，不是"永久丢失"。如果 Agent 不主动搜索，应通过 prompt 引导解决，而非在摘要中塞入检索线索（3.6.2）。

### 9.2 已解决的矛盾：单一阈值 vs 叙事密度波动

阈值控制"何时压缩"，不控制"压缩时保留什么"。设定阶段工具调用密集恰恰是最需要压缩的场景；收束阶段早期伏笔已历经多轮压缩，提高阈值无法保护。记忆系统（L1/L2/L3 + search_memory）已是长程回溯的答案，不需要通过阈值调优来模拟（3.6.3）。

### 9.3 已解决的矛盾：全量继承 vs 窗口稀缺

InkFlow 的 Subagent 继承了 stable + volatile 上下文，看似比三方系统多，但每个组件都有刚性需求：Reader 检查一致性必须知道 MEMORY.md，Editor 更新记忆必须理解 short_memory，FTS5 query=task 已在代码中实现。实际节省仅 1-1.5% 窗口空间（1300-1800 token / 128K），风险远大于收益。与 Hermes 的差异本质上是创意任务 vs 窄范围任务的区别（3.6.4）。

### 9.4 优化方向总结

| 优先级 | 方向 | 核心收益 | 借鉴来源 |
|--------|------|---------|---------|
| P0 | cache_control + Tool Result 裁剪 | 零成本效率提升 | Claude Code |
| P2 | 事实类型细粒度标注 | FTS5 检索精度提升 | 无（InkFlow 独有需求） |
